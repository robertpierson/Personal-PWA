Lesson 1

The forward pass as composed matrix operations

Say you're forecasting next-hour electricity demand for a mid-size community center. You've got a year of hourly readings — 8,760 rows, each with temperature, hour-of-day, and an occupancy estimate — and a network from Unit 1 that takes those three numbers in and produces a predicted kWh out. The naive way to get predictions for the whole year is a Python loop: grab row 1, run it through the network, store the output, grab row 2, repeat, 8,760 times. It works. It is also embarrassingly slow, because every one of those passes re-pays the same fixed overhead — function calls, memory allocation, cache misses — for a single row of arithmetic that a CPU or GPU could do thousands-wide in the same time. The fix is not a smarter loop. It's to stop thinking in rows at all.

Instead, stack rows into a matrix. Sixty-four hours of readings become a single array of shape (64, 3) — 64 rows, one per hour, 3 columns, one per feature (temperature in °C, hour-of-day as a number 0–23, occupancy as a headcount). This block of 64 hours is a batch, and running it through the network in one shot is batched forward propagation. The network doesn't know or care whether those 64 rows are consecutive hours or 64 hours sampled at random from across the year — matrix multiplication treats each row independently and identically. That independence is exactly what makes batching legal: the math applied to row 12 has zero interaction with the math applied to row 40.

The (batch, features) @ (features, units) contract

Recall from Unit 1 that a single dense layer computes z = w·x + b for one input vector x. Stack B input vectors as rows of a matrix X with shape (B, in_features), and the same layer becomes a matrix multiplication: Z = X @ W + b, where W has shape (in_features, units) and Z comes out shaped (B, units). Every row of X gets multiplied against every column of W and summed — that's what @ (matrix multiply) does — so row i of Z depends only on row i of X. Nothing leaks across rows. This is the contract you check every single time you wire up a layer: the trailing dimension of the input must equal the leading dimension of the weight matrix, and whatever's left over on each side survives into the output shape.

For our demand forecaster: input features are 3 (temperature, hour, occupancy), so the first hidden layer's weight matrix is shaped (3, 16) if you choose 16 hidden units. Feed in X of shape (64, 3), multiply by W1 of shape (3, 16), and you get Z1 of shape (64, 16) — 64 rows, each now described by 16 learned features instead of the original 3. An activation function (ReLU, say, from Unit 1) applies elementwise and doesn't change that shape at all. A second layer with W2 shaped (16, 1) takes you from (64, 16) down to (64, 1) — one predicted kWh value per hour in the batch.

Broadcasting the bias

There's a shape mismatch hiding in Z = X @ W + b that NumPy and every deep learning framework quietly paper over. X @ W produces shape (64, 16), but the bias vector b is only shaped (16,) — one bias per hidden unit, not one per row, because every row should get the same bias added to its corresponding unit. Broadcasting is the rule that lets NumPy add a (16,) vector to a (64, 16) matrix without you writing a loop: it treats the vector as if it were copied down all 64 rows, so b[j] gets added to every entry in column j. No copy actually happens in memory — the broadcast is computed on the fly — but conceptually it's "stretch the smaller shape until it matches, provided the trailing dimensions line up or one of them is 1." This is why bias shapes are always (units,), never (batch, units): the bias doesn't know or care how big the batch is, and it shouldn't have to.

Why row-major, batch-first

Every framework you'll touch — NumPy, PyTorch, TensorFlow — puts the batch dimension first: (batch, features), not (features, batch). Two reasons, one conceptual and one mechanical. Conceptually, it matches how you already think about the data: a CSV of hourly readings has one row per hour and one column per feature, and "row-major" storage (NumPy's default) lays memory out row by row, so row i is contiguous in memory — cheap to slice out, cheap to shuffle, cheap to drop into a mini-batch. Mechanically, keeping batch first means every layer's weight matrix shape is independent of batch size: a layer with a (3, 16) weight works whether your batch is 1 row or 8,760 rows, because batch size only ever shows up as the leading dimension of the activations, never baked into the weights. Change your batch size from 64 to 256 for a bigger training run and not one weight matrix has to change shape.

Memory footprint and caching activations

Every intermediate result — Z1, the activated A1, Z2, the final prediction — has to live in memory for at least as long as you need it. For a batch of 64 rows through a (3→16→1) network, that's cheap: A1 is 64 × 16 = 1,024 floats, about 4KB at 32-bit precision. But scale up — a full year of hourly data (8,760 rows) through a network with hidden layers of 256 units each — and a single activation tensor is 8,760 × 256 ≈ 2.24 million floats, roughly 9MB. Stack several such layers and the activation memory for one forward pass can dwarf the memory the weights themselves take up. This matters now, before we've written any backpropagation code, because Unit 3's gradient computation needs exactly these intermediate activations back — the chain rule multiplies local derivatives that depend on the values computed during the forward pass. So a real training loop doesn't discard Z1 and A1 after computing the final prediction; it caches them, trading memory for the ability to differentiate later. For now, just note that the forward pass isn't "compute an output and forget everything" — it's "compute an output and remember the trail."

import numpy as np

rng = np.random.default_rng(0)

# batch of 64 hourly readings: temperature (C), hour-of-day (0-23), occupancy (headcount)
batch_size = 64
X = np.column_stack([
    rng.normal(22, 6, batch_size),      # temperature
    rng.integers(0, 24, batch_size),    # hour of day
    rng.integers(0, 80, batch_size),    # occupancy
]).astype(np.float64)
print("X.shape:", X.shape)              # (64, 3)

# layer 1: 3 input features -> 16 hidden units
W1 = rng.normal(0, 0.1, (3, 16))
b1 = np.zeros(16)
Z1 = X @ W1 + b1                        # (64,3) @ (3,16) + (16,) -> (64,16)
A1 = np.maximum(Z1, 0)                  # ReLU, elementwise, shape unchanged
print("Z1.shape:", Z1.shape, " A1.shape:", A1.shape)

# layer 2: 16 hidden units -> 1 output (predicted next-hour kWh)
W2 = rng.normal(0, 0.1, (16, 1))
b2 = np.zeros(1)
Z2 = A1 @ W2 + b2                       # (64,16) @ (16,1) + (1,) -> (64,1)
y_pred = Z2                             # linear output head, regression target
print("Z2.shape:", Z2.shape)
print("first 3 predictions (kWh):", y_pred[:3].ravel())

# cache what Unit 3's backward pass will need
cache = {"X": X, "Z1": Z1, "A1": A1, "Z2": Z2}
print("cached activations:", list(cache.keys()))

Each line prints the shape immediately after the operation that produced it — trace how (64,3) becomes (64,16) becomes (64,1), and notice the cache dict at the bottom holding everything Unit 3 will need for gradients.

What changes, what doesn't, as batch size scales

Run this same code with batch_size = 1 and every weight matrix is untouched — only X, Z1, A1, and Z2 change their leading dimension. Run it with batch_size = 8760 — a full year in one shot — same story. This is the single fact that makes batching a free lunch on modern hardware: the FLOPs scale with batch size, but the code, the weight shapes, and the logic are identical no matter how many hourly rows you throw at it. The only real limits are memory (can you fit 8760 × 256 floats per layer, times however many layers, times however many tensors you're caching) and, eventually, diminishing returns on how parallel your hardware actually is. In practice you'll pick a batch size like 32, 64, or 256 that fits comfortably and use it consistently — a decision that resurfaces in Unit 3 when batch size and learning rate turn out to be coupled.

Key takeaways

  • Batching replaces a Python loop with one matrix multiplication — 64 hourly readings become a (64, features) matrix, and every row is processed identically and independently.
  • The shape contract is (batch, in_features) @ (in_features, units) → (batch, units) — the weight matrix's shape never depends on batch size, only on feature counts.
  • Broadcasting lets a (units,) bias vector add correctly to a (batch, units) pre-activation without a copy or a loop.
  • Batch-first, row-major layout keeps memory contiguous per-sample and keeps layer weight shapes independent of batch size.
  • Intermediate activations (Z1, A1, Z2) must be cached during the forward pass — Unit 3's backward pass reuses them, and they carry a real memory cost at scale.

Lesson 2

Loss functions: turning a prediction into one number

A loss function is a rule that takes a prediction and the true value it should have matched, and returns a single non-negative number that gets larger the more wrong the prediction was. That's the whole definition. Everything else — which specific formula you pick, how it behaves near zero, how it treats a wildly wrong prediction versus a slightly wrong one — is a design decision, and the right one depends on what kind of wrongness actually costs you something in your application. For our demand forecaster we have two prediction tasks running side by side: a regression task (predict next-hour kWh, a continuous number) and a binary classification task (predict whether this building will trip a peak-demand charge — a yes/no event that utilities often bill heavily for). Each needs its own family of loss functions.

Regression losses: MSE, MAE, Huber

Mean Squared Error (MSE) is the default regression loss for good reason: it's smooth everywhere, it has a clean derivative (which Unit 3 will use directly), and it penalizes big errors much more than small ones because the error gets squared.

MSE = (1/N) Σ (y_i − ŷ_i)²

If your model predicts 42 kWh for an hour where the true value was 45 kWh, that's an error of 3, contributing 9 to the sum. If instead it badly misses a demand spike and predicts 42 kWh when the true value was 90 kWh — the building just spun up every rooftop unit at once — that's an error of 48, contributing 2,304. The squaring means one bad spike-hour can dominate the entire loss computed over thousands of otherwise-decent hours. Sometimes that's exactly what you want — demand spikes are precisely the hours a utility cares about, because they're what triggers a peak-demand charge — and sometimes it's a liability, because a single sensor glitch (occupancy briefly logged as 9,999 due to a counter bug) can drag your whole training signal toward fitting garbage.

Mean Absolute Error (MAE) takes the absolute value instead of squaring:

MAE = (1/N) Σ |y_i − ŷ_i|

Now that same 48 kWh spike-miss contributes 48, not 2,304 — linear, not quadratic. MAE is far less sensitive to outliers, which makes it the safer choice when your training data has occasional bad sensor readings you can't fully clean out, or when you genuinely want "typical hour" accuracy rather than accuracy weighted toward rare extreme events. The trade-off: MAE's derivative is a constant magnitude (just the sign of the error) everywhere except exactly at zero error, where it's undefined — a property that will matter once Unit 3 discusses how gradient-based optimizers behave near a loss's minimum.

Huber loss is the compromise: quadratic (like MSE) for small errors, linear (like MAE) for large ones, switching at a threshold δ you choose.

Huber(e) = ½e² if |e| ≤ δ, else δ(|e| − ½δ) where e = y − ŷ

For our demand forecaster, a sensible δ might be 5 kWh — errors smaller than that (normal hour-to-hour prediction noise) get the smooth, well-behaved MSE treatment, while errors bigger than that (a missed spike, a sensor glitch) get damped to linear growth so they don't dominate training. Huber is the practical default in demand forecasting pipelines precisely because both failure modes are real: you want precision on ordinary hours and robustness against the occasional spike or bad reading.

Classification losses: cross-entropy

For "will this building trip a peak-demand charge this hour" — a binary yes/no — the output is a probability ŷ ∈ (0, 1), typically produced by a sigmoid activation on the final layer. Binary cross-entropy (BCE) measures how well that probability matched the true label y ∈ {0, 1}:

BCE = −(1/N) Σ [ y·log(ŷ) + (1−y)·log(1−ŷ) ]

Read it one term at a time. When the true label is 1 (the building did trip a peak charge), only the y·log(ŷ) term survives, and it rewards ŷ close to 1 with a loss near 0, while punishing ŷ close to 0 with a loss that blows up toward infinity. When the true label is 0, the (1−y)·log(1−ŷ) term takes over and does the mirror image. Unlike MSE, BCE's punishment for a confidently wrong prediction is unbounded — predict ŷ = 0.001 for an hour that does trip a peak charge and log(0.001) ≈ −6.9 makes that single sample dominate the loss.

If instead of two classes you had several — say, categorizing each hour into "normal," "elevated," or "peak-risk" demand tiers — you'd use categorical cross-entropy, the same idea generalized: a softmax layer turns raw output scores into a probability distribution over all classes, and the loss is the negative log-probability the model assigned to the true class:

CCE = −(1/N) Σ log( ŷ_i,c ) where c is the true class of sample i

The softmax + log-sum-exp stability trick

Softmax converts raw scores (logits) into probabilities via softmax(z)_j = exp(z_j) / Σ_k exp(z_k). Computed literally, this is a numerical landmine: if a logit is, say, 800 (easy to hit with poorly-scaled features or a few bad training steps), exp(800) overflows a 64-bit float completely — you get inf, then inf/inf is nan, and your whole training run silently corrupts. The fix, universally used in every real framework, is to subtract the maximum logit before exponentiating:

softmax(z)_j = exp(z_j − max(z)) / Σ_k exp(z_k − max(z))

This is mathematically identical — the max cancels out of numerator and denominator — but now the largest exponent computed is exp(0) = 1, and everything else is a fraction, so nothing overflows. When you then take the log of a softmax output for cross-entropy, framework code fuses the two steps into a single log_softmax that never actually materializes the raw probabilities, computing log Σ exp(z_k) (the log-sum-exp) in a similarly shifted, overflow-safe way. You rarely write this by hand, but you should recognize the pattern logits − max(logits) whenever you see it — it's this trick, not a stylistic choice.

import numpy as np

# five hourly predictions vs. true kWh (regression) and true peak-trip label (classification)
y_true_kwh   = np.array([45.0, 52.0, 38.0, 90.0, 47.0])   # actual demand
y_pred_kwh   = np.array([42.0, 55.0, 39.0, 42.0, 46.5])   # model missed the spike at index 3

y_true_peak  = np.array([0, 0, 0, 1, 0])                  # did this hour trip a peak charge?
y_pred_prob  = np.array([0.04, 0.10, 0.02, 0.55, 0.08])   # predicted probability of tripping

def mse(y, yhat):
    return np.mean((y - yhat) ** 2)

def mae(y, yhat):
    return np.mean(np.abs(y - yhat))

def huber(y, yhat, delta=5.0):
    e = y - yhat
    quad = 0.5 * e**2
    lin = delta * (np.abs(e) - 0.5 * delta)
    return np.mean(np.where(np.abs(e) <= delta, quad, lin))

def stable_bce(y, yhat, eps=1e-12):
    yhat = np.clip(yhat, eps, 1 - eps)     # guard log(0)
    return -np.mean(y * np.log(yhat) + (1 - y) * np.log(1 - yhat))

print(f"MSE:   {mse(y_true_kwh, y_pred_kwh):.3f}")
print(f"MAE:   {mae(y_true_kwh, y_pred_kwh):.3f}")
print(f"Huber: {huber(y_true_kwh, y_pred_kwh):.3f}")
print(f"BCE:   {stable_bce(y_true_peak, y_pred_prob):.3f}")

Same predictions, four different loss numbers — MSE (18.8) is dragged up hard by the single missed spike at index 3, MAE (3.5) barely notices it, Huber (10.75) sits between the two, and BCE reports the classification loss on an entirely different scale because it's measuring probability calibration, not kWh error.

Key takeaways

  • A loss function is just "prediction + truth → one number," but which number depends entirely on which errors you want to punish harder.
  • MSE squares errors and lets large misses (like a missed demand spike) dominate; MAE treats all error sizes proportionally and is robust to sensor-glitch outliers; Huber blends both, quadratic near zero and linear past a threshold δ.
  • Binary cross-entropy punishes confidently wrong probability predictions with unbounded loss — a critical property for a rare, costly event like a peak-demand trip.
  • Categorical cross-entropy generalizes BCE to multiple classes via softmax, and always needs the max-subtraction stability trick to avoid overflow in exp().
  • The same set of predictions produces meaningfully different loss values depending which function you apply — the choice is a modeling decision, not a formality.

Lesson 3

From per-sample loss to a training objective

Think of a per-sample loss like a single household's electric bill: it tells you exactly how much that one household owes, but it doesn't tell you anything about the utility's overall revenue until you decide how to combine every household's bill into one number. Do you sum all the bills, or average them? Does a home that used ten times the power count ten times as much, or the same as everyone else? Those combination choices — decisions you make after you already have a per-sample loss for every row in your batch — are what turn "how wrong was this one prediction" into "what number does the optimizer actually try to shrink." That combined number is the training objective, and how you build it has consequences that don't show up until training is already underway.

Reduction: mean vs. sum, and its hidden coupling to learning rate

Given per-sample losses for a batch of 64 hourly readings, you have to reduce them to one scalar before anything downstream can use them. The two obvious choices are mean (divide by batch size) and sum (don't). They look interchangeable — surely the optimizer just cares about the direction it needs to move? — but they aren't, because the size of the loss value directly scales the size of the gradient Unit 3 will compute from it. Sum over a batch of 64 produces a loss, and therefore a gradient, roughly 64 times larger than mean over the same batch. If you switch your batch size from 64 to 256 while using sum reduction, your gradients suddenly jump roughly 4x in magnitude with no change to the model or the learning rate — which, in practice, blows up training or forces you to hand-retune the learning rate every time you touch batch size. mean reduction keeps the loss (and its gradient) at a roughly constant scale regardless of batch size, which is why it's the near-universal default. The rule worth internalizing now, ahead of Unit 3: reduction and learning rate are coupled, and changing one without the other is a common, quiet source of training instability.

Class weighting and pos_weight for imbalanced peak events

Peak-demand trips are rare — maybe 3% of hours in a year actually trigger one. Train on unweighted BCE and the model can get 97% of hours "right" simply by predicting "no trip" every single time, because the loss barely notices the 3% of hours it's getting catastrophically wrong. Class weighting fixes this by multiplying the loss contribution of the minority class before averaging, so the rare-but-important peak-trip hours count for more:

BCE_weighted = −(1/N) Σ [ w_pos·y·log(ŷ) + w_neg·(1−y)·log(1−ŷ) ]

A common convention, often called pos_weight in framework APIs, sets the positive-class weight to the ratio of negatives to positives — if 3% of hours trip a peak charge, pos_weight ≈ 97/3 ≈ 32, meaning each true positive's error counts roughly 32 times as much as each true negative's. This doesn't change what the loss function fundamentally measures; it changes which mistakes the training objective treats as expensive, nudging the model toward catching more true peak events at the cost of a few more false alarms — a trade-off that's usually worth it, since missing a peak-demand trip costs the client real money and a false alarm costs nothing but a wasted glance.

Sample weighting

Class weighting assigns one weight per class. Sample weighting is more general: every individual row gets its own weight, chosen for whatever reason matters to your problem. You might down-weight hours from a known-faulty temperature sensor so a bad batch of readings doesn't corrupt training, or up-weight readings from summer afternoons — precisely the hours when demand spikes and peak charges are most likely — so the model spends its limited capacity getting those hours right rather than spreading effort evenly across a year that's mostly unremarkable overnight readings. Mechanically it's just one more multiplication before the reduction: multiply each per-sample loss by its weight, then average.

What a loss curve tells you — and what it doesn't

A loss curve — training loss plotted against training step or epoch — tells you whether the optimization process (which Unit 3 covers in full) is making progress: is the number the optimizer is trying to shrink actually getting smaller? That's it. It does not tell you the model is good, and it does not tell you the model generalizes to hours it hasn't seen. A loss curve dropping steadily to near zero on training data is equally consistent with "the model learned real structure in temperature, hour, and occupancy" and with "the model memorized this exact year's 8,760 rows and will fall apart on next year's data" — a failure mode called overfitting that later units address directly. A falling loss curve is necessary evidence of learning; it is not sufficient evidence of a good model.

The loss you optimize vs. the metric you report

This is the distinction that trips up almost everyone the first time: the loss function and the reported metric don't have to be the same thing, and often shouldn't be. You might train the regression head with MSE — because its smooth gradient makes optimization well-behaved — but report performance to a client in MAE, because "on average, we're off by 3.2 kWh" is a number a facilities manager can actually reason about, while a client rarely wants to hear "our mean squared error is 14.7." Similarly, you'd typically train the peak-trip classifier with BCE, but report performance with AUC (area under the ROC curve) or precision/recall at a chosen threshold, because AUC measures ranking quality across every possible decision threshold at once, which is what a client actually cares about when deciding how aggressive to set an alert. The loss is the thing gradient descent climbs down; the metric is the thing a human uses to judge whether the destination was worth reaching. They're related — a lower loss usually, not always, means a better metric — but conflating them is a category error you want to catch now, before Unit 3 hands you the machinery to actually minimize the loss.

import numpy as np

def compute_loss(y_true, y_pred, sample_weights=None, reduction="mean", pos_weight=None):
    """Per-sample MSE with optional sample weighting, class weighting (pos_weight),
    and a choice of reduction. Illustrates composition, not a specific loss family."""
    per_sample = (y_true - y_pred) ** 2

    if pos_weight is not None:
        # treat y_true as a 0/1 label; scale the positive-class rows up
        class_w = np.where(y_true == 1, pos_weight, 1.0)
        per_sample = per_sample * class_w

    if sample_weights is not None:
        per_sample = per_sample * sample_weights

    if reduction == "mean":
        return np.mean(per_sample)
    elif reduction == "sum":
        return np.sum(per_sample)
    else:
        raise ValueError(f"unknown reduction: {reduction}")

# fake 5-epoch trace: loss should trend down as a (toy, hand-picked) model improves
rng = np.random.default_rng(1)
y_true = np.array([45.0, 52.0, 90.0, 47.0, 38.0])   # note: index 2 is a peak-demand hour
sample_w = np.array([1.0, 1.0, 3.0, 1.0, 1.0])       # up-weight the peak hour

print(f"{'epoch':>5}  {'mean loss':>10}  {'sum loss':>10}")
for epoch in range(1, 6):
    noise_scale = 8.0 / epoch                        # predictions get closer to truth each epoch
    y_pred = y_true + rng.normal(0, noise_scale, size=5)
    mean_l = compute_loss(y_true, y_pred, sample_weights=sample_w, reduction="mean")
    sum_l  = compute_loss(y_true, y_pred, sample_weights=sample_w, reduction="sum")
    print(f"{epoch:>5}  {mean_l:>10.3f}  {sum_l:>10.3f}")

Watch two things: the mean-loss column shrinks steadily as simulated training "improves," while the sum-loss column is consistently ~5x larger for the same predictions — same model, same errors, different scale, purely because of the reduction choice.

Key takeaways

  • Reduction (mean vs. sum) rescales the gradient the optimizer sees; mean keeps loss magnitude stable across batch sizes, sum doesn't — this is why mean is the default and why the choice is coupled to learning rate.
  • Class weighting / pos_weight compensates for rare-but-costly events (peak-demand trips) that an unweighted average loss would let the model ignore.
  • Sample weighting generalizes class weighting to per-row control — down-weight bad sensor data, up-weight the hours you most need right.
  • A falling loss curve proves the optimizer is making progress on its objective; it does not prove the model generalizes.
  • The loss you optimize (MSE, BCE — chosen for gradient behavior) and the metric you report (MAE in kWh, AUC — chosen for human interpretability) are different tools for different jobs, and treating them as interchangeable is a common mistake.

Question 1 Multiple choice

A dense layer has weight matrix W of shape (3, 16) and you feed it a batch of 64 hourly readings, each with 3 features. What is the shape of the pre-activation output Z?

  1. (3, 16)
  2. (64, 3)
  3. (64, 16)
  4. (16, 64)
Show answer

Answer: C. X is (64, 3), W is (3, 16); matrix multiplication cancels the shared inner dimension (3) and keeps the outer dimensions, giving (64, 16) — 64 rows, 16 learned features per row.

(B) is a tempting distractor because it's just the input shape unchanged — but the whole point of the layer is to transform the feature dimension from 3 to 16, so the output can't have the same shape as the input.

Question 2 True / false

A bias vector for a layer with 16 hidden units should have shape (64, 16) so that it can be added directly to a batch of 64 pre-activations without broadcasting.

  1. True
  2. False
Show answer

Answer: False. The bias should be shape (16,) — one value per hidden unit, independent of batch size. Broadcasting is exactly the mechanism that lets a (16,) vector add correctly to a (64, 16) matrix without ever materializing a (64, 16) copy of the bias.

Giving the bias a batch dimension would mean it has to be re-shaped every time the batch size changes, defeating the point of the batch-independent weight/bias design and wasting memory for no benefit.

Question 3 Fill in the blank

A forward pass must cache its intermediate activations (like Z1 and A1) because Unit 3's   computation reuses these exact values via the chain rule.

  1. loss reduction
  2. backward pass / gradient
  3. softmax normalization
  4. batch shuffling
Show answer

Answer: B, backward pass / gradient. Backpropagation applies the chain rule layer by layer, and each layer's local derivative depends on the actual activation values computed during the forward pass — so those values must be kept in memory, not discarded.

Loss reduction (A) happens after the forward pass produces predictions and doesn't need intermediate layer activations, only the final output and the true labels.

Question 4 Multiple choice

Your demand forecaster's training data has a handful of hours where a faulty occupancy sensor logged a wildly wrong headcount, producing occasional huge prediction errors that don't reflect real model failure. Which regression loss is most robust to those occasional large outliers?

  1. MSE, because squaring rewards large errors with more learning signal
  2. MAE, because it penalizes all error magnitudes proportionally rather than quadratically
  3. Categorical cross-entropy, because it handles multiple classes
  4. Sum-reduced MSE, because summing dilutes any one outlier's effect
Show answer

Answer: B. MAE grows linearly with error size, so one sensor-glitch hour with a huge error contributes proportionally to the total loss rather than dominating it the way a squared error would.

(A) is backwards — MSE is exactly the loss that is most sensitive to outliers, since squaring a large error inflates it disproportionately, which is the opposite of what you want with unreliable sensor data.

Question 5 Bug fix

This softmax implementation works fine on small logits but returns nan once logits get large (e.g. from a poorly scaled layer). What line is the bug, and what's the one-line fix?

def softmax(z):
    exp_z = np.exp(z)
    return exp_z / np.sum(exp_z)
  1. Nothing is wrong; large logits are a data problem, not a code problem
  2. Subtract np.max(z) from z before exponentiating
  3. Replace np.exp with np.log
  4. Divide by len(z) instead of np.sum(exp_z)
Show answer

Answer: B. exp_z = np.exp(z - np.max(z)) keeps the largest exponent at exp(0) = 1 instead of overflowing on large logits, while leaving the mathematical result identical because the constant cancels between numerator and denominator.

(D) would change what the function computes entirely (it's no longer a valid probability distribution) rather than fixing the numerical issue — the bug is about overflow in exp(), not about the normalization constant's identity.

Question 6 Multiple choice

Only 3% of hours in your dataset actually trip a peak-demand charge. Trained with plain unweighted BCE, your classifier reaches 97% accuracy by predicting "no trip" for every hour. What's the most direct fix, given what this lesson covered?

  1. Switch reduction from mean to sum so rare events contribute more
  2. Apply a pos_weight to the positive (trip) class so its errors count more in the loss
  3. Switch from BCE to MSE, since MSE handles imbalance natively
  4. Increase the batch size until peak-trip hours appear more often
Show answer

Answer: B. A pos_weight (often set to the negative:positive ratio, here roughly 32) scales up the loss contribution of true peak-trip hours, so the model can no longer get a low loss just by always predicting "no trip."

(A) is a distractor because switching mean to sum rescales the whole loss uniformly — it doesn't change the relative weight between classes at all, so the imbalance problem is untouched.

Question 7 True / false

If your peak-demand classifier's training loss (BCE) steadily decreases over 50 epochs, that alone confirms the model will correctly flag peak-demand events on next year's data.

  1. True
  2. False
Show answer

Answer: False. A falling loss curve only shows that the optimizer is successfully shrinking the training objective — it's equally consistent with genuine learning and with the model memorizing this specific year's data (overfitting), which would fail on new data.

The tempting trap here is treating "loss went down" as proof of a good model — the loss curve and generalization are related but distinct, and only the latter is confirmed by evaluating on held-out data the model never trained on.
