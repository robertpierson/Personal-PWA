Lesson 1

The chain rule on a computational graph

In 1986, David Rumelhart, Geoffrey Hinton, and Ronald Williams published "Learning representations by back-propagating errors" in Nature, and the technique finally had a name and an audience. The underlying calculus was not new — Seppo Linnainmaa had described reverse-mode automatic differentiation in his 1970 thesis, and Paul Werbos had proposed applying it to neural networks in 1974 — but those results sat in obscurity for over a decade. What changed in 1986 was demonstration: Rumelhart, Hinton, and Williams showed that a network could learn internal, distributed representations of a problem's structure just by repeatedly applying the chain rule and nudging weights downhill. Three years later, Yann LeCun and colleagues at Bell Labs took the idea somewhere concrete and commercially urgent: reading handwritten ZIP codes off scanned mail. Their 1989 paper, "Backpropagation Applied to Handwritten Zip Code Recognition," trained a convolutional network on 16×16 pixel digit crops drawn from actual U.S. mail, and it worked well enough to matter to the Postal Service. That paper is arguably the first time backprop left the whiteboard and touched a real production pipeline.

Why did the idea sit unused for so long before that? Partly it was a matter of reputation: Marvin Minsky and Seymour Papert's 1969 book Perceptrons proved hard limits on what a single-layer network could represent, and though multi-layer networks were not what the book ruled out, the pessimism it created cooled funding and attention for over a decade — a period often called the first AI winter. But the more concrete obstacle was practical: nobody had shown an efficient way to compute the gradient of a loss with respect to every weight in a multi-layer network. Computing each partial derivative independently, layer by layer, from scratch, is expensive enough that it discourages the whole approach. Reverse-mode automatic differentiation removes that obstacle, and once it was demonstrated on a problem people cared about — misrouted mail costs money — the technique had nowhere to go but into every serious model built since.

Two graphs, one computation

Recall the forward pass from Unit 1: a scanned digit crop, flattened into a vector of pixel intensities, moves through a sequence of linear layers and activations — a forward graph where each node is an operation (a matrix multiply, a bias add, a ReLU, a softmax) and each edge carries a tensor of values downstream. The loss function at the end of that graph, cross-entropy against the true digit label from Unit 2, collapses everything to a single scalar: how wrong was this prediction. Backpropagation builds a second graph over the same nodes, running in the opposite direction, in which every edge instead carries a gradient — how much a small change in that value would change the final loss. This is the backward graph, and it is not a separate computation invented after the fact; it is the mechanical consequence of applying the calculus chain rule to every node in the forward graph, one at a time, starting from the loss and working back toward the pixels.

The key idea that makes this tractable is that every node only ever needs to know its own local gradient — the derivative of its own output with respect to its own inputs, evaluated at the values it saw during the forward pass. A matrix multiply node does not need to know anything about softmax or cross-entropy three layers downstream; it only needs to know how its own output changes when its own inputs change. The multivariable chain rule is what stitches these local gradients into a global answer: if a value z1 feeds forward into several later computations, the gradient flowing back into z1 is the sum of the gradients arriving from every path that used it, each scaled by the local gradient along that path. For a two-layer classifier this fan-in is modest, but in a real ZIP-code recognizer with a convolutional front end, a single feature map might feed a dozen downstream computations, and the chain rule guarantees each one contributes its share correctly, without double-counting or omission.

Why you cache activations

Local gradients are evaluated at the specific values the forward pass produced — the derivative of ReLU at a given point depends on whether that point was positive or negative, and the derivative of a matrix multiply with respect to its weights is literally the input that was multiplied. This means the backward pass cannot run on its own; it needs the intermediate activations that the forward pass computed, which is why every serious implementation caches them — the input to each layer, the output of each activation, the pre-softmax scores — for later reuse. For a sorting-facility model processing tens of thousands of digit crops per training batch, this cache is a real memory cost, one deep networks eventually trade off against recomputation (a technique called gradient checkpointing, out of scope here), but for the two-layer network in this unit the cache is simply "keep the arrays around a little longer."

Reverse-mode versus forward-mode

Automatic differentiation comes in two flavors that compute the exact same mathematical answer by different bookkeeping. Forward-mode AD picks one input, perturbs it symbolically, and propagates that perturbation forward through the graph to see how every output responds; the cost of one forward-mode pass scales with the number of inputs you care about. Reverse-mode AD does the opposite: it picks one output, and propagates backward to see how every input contributed to it; its cost scales with the number of outputs. A ZIP-code classifier has a single scalar loss (one output) and easily tens of thousands of weights (the inputs, from the network's point of view during training). Forward-mode would require one full pass per weight to get the complete gradient — utterly impractical. Reverse-mode requires exactly one backward pass, regardless of how many weights the network has, because it computes the gradient with respect to every input simultaneously. This asymmetry is the entire reason backpropagation — reverse-mode AD applied to a computational graph with a scalar loss — is the default training algorithm for every architecture in this course, from the plain two-layer network here to the deepest transformer stack.

class Value:
    """A scalar that remembers how it was made, so it can backprop."""
    def __init__(self, data, parents=(), local_grads=()):
        self.data = data
        self.grad = 0.0
        self._parents = parents       # tuple of Value objects used to build this one
        self._local_grads = local_grads  # d(self)/d(parent), same order as _parents

    def __add__(self, other):
        out = Value(self.data + other.data, (self, other), (1.0, 1.0))
        return out

    def __mul__(self, other):
        # local gradient of a*b w.r.t. a is b, and w.r.t. b is a
        out = Value(self.data * other.data, (self, other), (other.data, self.data))
        return out

    def relu(self):
        out = Value(max(0.0, self.data), (self,), (1.0 if self.data > 0 else 0.0,))
        return out

    def backward(self):
        # build a topological order so every node fires only after
        # every node that depends on it has already propagated its gradient
        order, visited = [], set()
        def visit(v):
            if id(v) not in visited:
                visited.add(id(v))
                for p in v._parents:
                    visit(p)
                order.append(v)
        visit(self)
        self.grad = 1.0  # dL/dL = 1, the seed for reverse mode
        for v in reversed(order):
            for parent, local in zip(v._parents, v._local_grads):
                parent.grad += v.grad * local  # multivariable chain rule: sum over paths

A minimal reverse-mode autodiff. Each op stores its parents and local gradients at construction time; backward() walks the topological order once, seeding the output's gradient at 1.0 and accumulating parent.grad += v.grad * local — exactly the chain rule, one multiply-and-add per edge.

Key takeaways

  • Backprop is reverse-mode automatic differentiation applied to the forward computational graph — not a separate algorithm bolted onto the network, but the chain rule mechanized one node at a time.
  • Every node needs only its local gradient, evaluated at the forward-pass values it saw; the multivariable chain rule sums contributions when a value feeds multiple downstream paths.
  • Caching forward activations is mandatory because local gradients (ReLU's derivative, a matmul's derivative with respect to its weights) depend on the specific values computed on the way forward.
  • Reverse-mode wins when there are few outputs and many inputs — one scalar loss and thousands of weights is the canonical case, and it is exactly the shape of every network trained in this course.
  • LeCun's 1989 Bell Labs system is the historical bridge from Rumelhart, Hinton, and Williams's theoretical demonstration to a working, economically motivated ZIP-code reader.

Lesson 2

Backprop through a two-layer network, by hand

A sorting machine flags a scanned envelope crop, runs it through the two-layer network from Units 1–2 — linear, ReLU, linear, softmax — and gets a distribution over the ten digits for that ZIP-code position. Say the true digit is 7 and the network's highest probability landed on 1. The forward pass alone gives no instruction for fixing this: it produces a number, cross-entropy loss, that says how wrong the guess was, but nothing about which of the network's thousands of weights should move, or in which direction, or by how much. That is exactly the gap backpropagation closes, and for a network this small — small enough that every array fits on one page — it is worth deriving every gradient by hand once, so that the NumPy version in later units is never a black box.

Fix notation to match the architecture from Units 1–2. A mini-batch of N digit crops (Unit 3's third lesson will fix N; for now treat it as generic) arrives as x, shape (N, D), where D is 256 for a 16×16 crop or 784 for a 28×28 crop. The first linear layer has weights W1, shape (D, H), and bias b1, shape (H,), producing pre-activation z1 = x @ W1 + b1, shape (N, H). ReLU gives the hidden activation h = relu(z1), same shape. The second linear layer has weights W2, shape (H, C), and bias b2, shape (C,), with C = 10 digit classes, producing scores z2 = h @ W2 + b2, shape (N, C). Softmax turns those scores into probabilities p, same shape, and cross-entropy against the one-hot labels y gives the scalar loss L, averaged over the batch.

The softmax + cross-entropy gradient

The single most useful identity in this whole derivation is what happens when softmax feeds directly into cross-entropy. Written separately, the softmax Jacobian is a full C×C matrix per example, and the cross-entropy gradient with respect to p involves a division by p that looks fragile near zero. Multiplied together via the chain rule, nearly everything cancels, leaving one of the cleanest results in deep learning:

dL/dz2 = (p − y) / N

That is: the gradient flowing into the pre-softmax scores is just the predicted probability vector minus the one-hot true label, divided by the batch size to account for averaging. If the network assigns 0.7 probability to the correct digit, the gradient there is negative and small in magnitude — nudge that score up gently. If it assigns only 0.05 to the correct digit, the gradient is close to −1/N — nudge that score up hard. The gradient's magnitude is automatically proportional to how wrong the prediction was, with no extra weighting needed.

Propagating to W2 and b2

From dz2, shape (N, C), the weight gradient falls out of the matrix-calculus form of the chain rule: since z2 = h @ W2 + b2, the gradient with respect to W2 is h transposed, matrix-multiplied against dz2:

dW2 = hᵀ @ dz2   (H, N) @ (N, C) → (H, C), matching W2

db2 = Σ over batch of dz2   → (C,), matching b2

Notice the shapes are not a coincidence: the gradient of any parameter always comes out with exactly that parameter's shape, which is both a mathematical necessity and the cheapest sanity check available — if dW2 and W2 disagree in shape, the derivation has a bug. To keep propagating backward, the gradient also needs to flow through h itself, back toward the first layer:

dh = dz2 @ W2ᵀ   (N, C) @ (C, H) → (N, H), matching h

Through ReLU, to W1 and b1

ReLU's local gradient is the simplest nonlinearity to backprop through: its derivative is 1 wherever the forward pre-activation z1 was positive, and 0 wherever it was zero or negative — which is precisely why the forward pass must cache z1, not just h. The chain rule gives:

dz1 = dh ⊙ (z1 > 0)   elementwise mask, shape (N, H)

and then the same matrix-calculus pattern as before, one layer earlier:

dW1 = xᵀ @ dz1   (D, N) @ (N, H) → (D, H), matching W1

db1 = Σ over batch of dz1   → (H,), matching b1

Every gradient the network needs — dW1, db1, dW2, db2 — is now expressed in terms of quantities the forward pass already computed (x, z1, h, p) plus the label y. No further calculus is required to train this network; everything downstream is bookkeeping.

A worked example small enough to check by hand

To verify the mechanics without a computer, shrink the problem to a single example (N = 1), two input features, a two-unit hidden layer, and three output classes instead of ten — the mechanism is identical, just smaller than a real 256-input, 10-class ZIP-code layer. Let x = [1.0, −1.0], W1 = [[0.5, −0.2], [0.3, 0.8]], b1 = [0.1, −0.1]. Then z1 = x @ W1 + b1 = [0.3, −1.1], and ReLU gives h = [0.3, 0.0] — the second hidden unit is dead for this input. With W2 = [[1.0, 0.5, −0.5], [0.2, −0.3, 0.1]] and b2 = [0, 0, 0], the scores are z2 = h @ W2 + b2 = [0.3, 0.15, −0.15]. Softmax of that is approximately p = [0.400, 0.345, 0.255]. Suppose the true digit corresponds to class 0, so y = [1, 0, 0]; then dz2 = p − y ≈ [−0.600, 0.345, 0.255].

From there, dW2 = hᵀ @ dz2 has a nonzero first row (since h_0 = 0.3) of approximately [−0.180, 0.103, 0.077] and a second row of exactly [0, 0, 0], because a dead ReLU unit — zero on the forward pass — contributes nothing to any weight gradient this step. Continuing back, dh = dz2 @ W2ᵀ ≈ [−0.555, −0.198], and since z1 = [0.3, −1.1] the ReLU mask is [1, 0], giving dz1 ≈ [−0.555, 0]. Finally dW1 = xᵀ @ dz1 works out to approximately [[−0.555, 0], [0.555, 0]]. Every one of these numbers can be reproduced with a calculator in under five minutes, which is exactly the point: before trusting a NumPy implementation on a real batch of ZIP-code crops, trust it on a case you can check by hand.

import numpy as np

def backward(x, y_onehot, cache):
    """
    x:        (N, D)  input pixel batch
    y_onehot: (N, C)  true digit labels, one-hot
    cache:    dict with z1 (N,H), h (N,H), p (N,C) from the forward pass
    returns:  dict of gradients, one entry per parameter
    """
    N = x.shape[0]
    z1, h, p = cache["z1"], cache["h"], cache["p"]
    W2 = cache["W2"]

    dz2 = (p - y_onehot) / N                      # (N, C)
    dW2 = h.T @ dz2                               # (H, C)
    db2 = dz2.sum(axis=0)                         # (C,)

    dh = dz2 @ W2.T                               # (N, H)
    dz1 = dh * (z1 > 0)                           # (N, H), ReLU mask
    dW1 = x.T @ dz1                               # (D, H)
    db1 = dz1.sum(axis=0)                         # (H,)

    # every gradient must match its parameter's shape — cheap, load-bearing check
    assert dW2.shape == W2.shape
    assert dW1.shape == cache["W1"].shape
    assert db1.shape == cache["b1"].shape
    assert db2.shape == cache["b2"].shape

    return {"dW1": dW1, "db1": db1, "dW2": dW2, "db2": db2}

A full NumPy backward pass for the two-layer ZIP-code classifier. Every line traces directly back to one of the derivations above; the assertions are not decoration — a transposed matmul or a forgotten .T shows up immediately as a shape mismatch rather than a silently wrong gradient.

Key takeaways

  • Softmax plus cross-entropy collapses to p − y — one of the few places in deep learning where a seemingly complex derivative simplifies to something you can compute in your head.
  • Weight gradients are always the transposed input times the incoming gradient (dW2 = hᵀ @ dz2, dW1 = xᵀ @ dz1), and bias gradients are always a sum over the batch — the same pattern at every linear layer.
  • ReLU backprop is a binary mask from the cached pre-activation: a unit that was zero (or negative) on the forward pass gets exactly zero gradient, regardless of how large the downstream gradient is.
  • Gradient shapes must match parameter shapes — this is the single cheapest correctness check available and catches the majority of hand-derivation and implementation bugs.
  • A small, hand-checkable example (two inputs, two hidden units, three classes) exercises the exact same mechanism as the full 256-input, 10-class network, just at a size a calculator can verify.

Lesson 3

Gradient descent: batch, stochastic, and mini-batch

Gradient descent is the update rule that turns the gradients from Lesson 2 into an actual improvement in the network: θ ← θ − η · ∇L(θ), where θ stands for any parameter (a weight matrix, a bias vector), ∇L(θ) is the gradient of the loss with respect to that parameter computed by backpropagation, and η (eta) is the learning rate — a single scalar controlling how big a step to take. Every training run in this unit is built entirely from that one line, applied over and over, to every parameter, using the gradients derived in Lesson 2. This unit deliberately stops at plain stochastic gradient descent — no momentum, no Adam, no learning-rate schedules — because those refinements only make sense once the raw update rule and its failure modes are second nature; Unit 4 owns the optimizers.

How much data feeds one gradient

The update rule itself never changes, but there is a real design decision in how much data goes into computing ∇L(θ) before each update, and it has direct consequences at a sorting facility processing scanned mail around the clock. Batch gradient descent computes the gradient over the entire training set — every labeled ZIP-code crop available — before taking a single step. This gives the most accurate estimate of the true gradient and the smoothest possible descent, but if the training set has 60,000 crops, that is 60,000 forward and backward passes for one update, an enormous amount of compute and memory for very little progress, and it does not fit typical GPU or CPU memory once the dataset grows past a modest size.

Stochastic gradient descent (SGD, in the narrow historical sense of the term) takes the opposite extreme: compute the gradient from a single example and update immediately. This is cheap per step and lets the network start learning before it has even seen the whole dataset once, but the gradient from one digit crop is a noisy, high-variance estimate of the true gradient over the whole distribution — the loss trace bounces around visibly from step to step. That noise is not purely bad: it can help the optimizer hop out of shallow local minima or saddle points that a smoother, more accurate gradient would settle into. But taken to the extreme, one-example updates also under-use modern hardware, which is built to multiply large matrices in parallel, not to process single vectors one at a time.

Mini-batch gradient descent is the practical compromise nearly every real system uses, including the ZIP-code sorter in this course: compute the gradient over a modest batch, typically 32 to 256 examples, average it, and update. This keeps the gradient estimate far less noisy than single-example SGD while still taking many updates per pass through the data, and — critically for throughput — batch sizes in that range are exactly what makes efficient use of vectorized matrix multiplication on the hardware actually doing the sorting. In casual usage, and for the rest of this course, "SGD" refers to this mini-batch version unless stated otherwise.

Epochs, steps, and the learning rate

One step is a single call to the update rule, using the gradient from one mini-batch. One epoch is one full pass through the entire training set. With 60,000 labeled ZIP-code crops and a batch size of 128, one epoch takes 60000 / 128 ≈ 469 steps; training for 20 epochs means roughly 9,380 update steps total, and each of those steps is one opportunity for the learning rate to help or hurt.

Of every hyperparameter in this unit, the learning rate is the one most likely to make or break a training run, because it interacts multiplicatively with everything else: a network that would otherwise learn perfectly well can fail outright from a poorly chosen η. Set it too high and the updates overshoot the loss surface's curvature — each step lands further from the minimum than the last, and the loss trace does not merely fail to improve, it grows, often diverging to NaN within a few dozen steps as weights blow up. Set it too low and each step barely nudges the weights; the loss decreases, technically, but so slowly that a sorting facility retraining its classifier overnight would not see meaningful improvement by morning — this is a crawl, distinct from failure but wasteful of real wall-clock time. Between those extremes sits a third failure mode worth naming separately: oscillation, where the learning rate is not large enough to diverge outright but is large enough to repeatedly overshoot the minimum and bounce around it, so the loss trace zigzags without ever settling — often mistaken for a "plateau" until you watch it long enough to see it does not flatten so much as jitter.

A first look at vanishing and exploding gradients

The two-layer ReLU network in this unit is shallow enough to sidestep a problem that becomes serious the moment networks get deep: gradients that shrink or grow uncontrollably as they propagate backward through many layers. Consider a deep stack of sigmoid activations instead of ReLU — an architecture common before ReLU became standard. The sigmoid function's derivative is at most 0.25, reached only at its midpoint, and typically much smaller. Backpropagating through ten stacked sigmoid layers multiplies roughly ten such derivatives together via the chain rule, and even at the generous ceiling of 0.25 per layer that is 0.25¹⁰ ≈ 0.00000095 — the gradient reaching the earliest layers is vanishingly small, and those layers barely update at all no matter how many training steps run. This is the vanishing gradient problem, and it is one of the concrete historical reasons ReLU, whose derivative is a clean 1 for any positive input, displaced sigmoid as the default hidden activation.

The opposite failure, exploding gradients, happens when weights are scaled such that each layer's local gradient is greater than 1 rather than less — the product across layers grows rather than shrinks, and a single unlucky batch can produce gradients large enough to send an update rule into the same divergence a too-high learning rate causes. The standard first-line defense is gradient clipping: after computing the gradient for a mini-batch, measure its overall norm, and if that norm exceeds a chosen threshold, rescale the entire gradient down to that threshold before applying the update. Clipping is a guard against explosion specifically — it caps how large a single step can be — but it does nothing to fix vanishing gradients, which is a problem of the gradient being too small to begin with, not too large; no amount of clipping makes a near-zero gradient carry more signal. Deeper architectures need real remedies for vanishing gradients — better activations, normalization, skip connections — which are out of scope for this unit's plain two-layer network but worth naming now, because the diagnosis starts here.

import numpy as np

def train_one_epoch(X, Y, params, lr, batch_size=128, rng=None):
    """
    X: (N, D) all training crops this epoch, Y: (N, C) one-hot labels
    params: dict of W1, b1, W2, b2 (mutated in place)
    lr: learning rate for this run
    """
    rng = rng or np.random.default_rng()
    N = X.shape[0]
    order = rng.permutation(N)          # shuffle every epoch — no fixed ordering
    losses = []

    for start in range(0, N, batch_size):
        idx = order[start:start + batch_size]
        xb, yb = X[idx], Y[idx]

        cache = forward(xb, params)                 # from Units 1-2
        loss = cross_entropy(cache["p"], yb)
        grads = backward(xb, yb, cache)              # from Lesson 2

        for name in ("W1", "b1", "W2", "b2"):
            params[name] -= lr * grads["d" + name]    # plain SGD, no momentum

        losses.append(loss)
    return losses

Key takeaways

  • The update rule is always θ ← θ − η∇L(θ) — batch, stochastic, and mini-batch gradient descent differ only in how much data is averaged into ∇L(θ) before each step.
  • Mini-batches of 32–256 examples are the practical default because they cut gradient noise far below single-example SGD while still matching the vectorized width of real hardware — unlike full-batch, which is accurate but too slow to iterate.
  • An epoch is a full pass over the data; a step is one update — with N training crops and batch size B, one epoch takes N/B steps, and this ratio is what "overnight retraining" throughput actually depends on.
  • The learning rate is the single highest-leverage hyperparameter: too high diverges to NaN, too low crawls without meaningful progress, and a rate just below the divergence threshold often oscillates rather than converging cleanly.
  • Vanishing and exploding gradients are opposite failures of the same chain-rule product across layers — gradient clipping caps explosion by rescaling an over-large gradient norm, but it has no effect on a gradient that vanished from too many small local derivatives multiplied together.

Question 1 Multiple choice

A ZIP-code classifier has one scalar cross-entropy loss as output and roughly 40,000 weights as inputs to differentiate with respect to. Why does reverse-mode automatic differentiation (backpropagation) beat forward-mode here?

  1. Forward-mode cannot represent nonlinear operations like ReLU or softmax.
  2. Reverse-mode cost scales with the number of outputs (one loss), so a single backward pass yields the full gradient; forward-mode cost scales with the number of inputs (40,000 weights), requiring one pass per weight.
  3. Reverse-mode uses less memory because it never caches forward activations.
  4. Forward-mode only works for convolutional networks, not fully connected ones.
Show answer

Answer: B. Reverse-mode's cost is proportional to the number of outputs, and there is exactly one — the scalar loss — so one backward pass produces every weight's gradient at once. Forward-mode's cost is proportional to the number of inputs, so it would need one pass per weight, roughly 40,000 forward-mode passes to match a single backward pass.

A is wrong — both modes handle nonlinear operations fine, they just differ in bookkeeping direction. C is wrong — reverse-mode is exactly the technique that requires caching forward activations, which costs extra memory rather than saving it. D is wrong — the input/output asymmetry argument applies to any architecture with a scalar loss, convolutional or not.

Question 2 True / false

Because backpropagation only needs each node's local gradient, a correct implementation never needs to store any of the forward pass's intermediate activations.

  1. True
  2. False
Show answer

Answer: False. Local gradients are evaluated at the specific values the forward pass produced — ReLU's derivative depends on whether the pre-activation was positive or negative, and a matmul's gradient with respect to its weights is literally the cached input. Without caching those forward values, the backward pass has nothing to evaluate the local gradients at.

The tempting reasoning is that "local" means the gradient calculation is self-contained and needs no outside information — but self-contained still means it needs its own forward-pass inputs, which is precisely what gets cached.

Question 3 Fill in the blank

For a network ending in softmax followed by cross-entropy loss, with predicted probabilities p and one-hot true labels y, the gradient flowing back into the pre-softmax scores z2 simplifies to   (ignoring the batch-averaging factor).

Show answer

Answer: p − y. The softmax Jacobian and the cross-entropy gradient combine and cancel almost entirely, leaving the predicted probability vector minus the one-hot label — a result whose magnitude is automatically proportional to how wrong the prediction was.

A common wrong guess is −y · log(p) — that expression is the cross-entropy loss itself (summed over classes), not its gradient with respect to z2; the log and the softmax Jacobian have already been differentiated away by the time you reach dz2.

def train_one_epoch(X, Y, params, lr, batch_size=128):
    N = X.shape[0]
    losses = []
    for start in range(0, N, batch_size):
        xb = X[start:start + batch_size]
        yb = Y[start:start + batch_size]
        cache = forward(xb, params)
        loss = cross_entropy(cache["p"], yb)
        grads = backward(xb, yb, cache)
        for name in ("W1", "b1", "W2", "b2"):
            params[name] -= lr * grads["d" + name]
        losses.append(loss)
    return losses
  1. The learning rate is applied inside the loop instead of after it.
  2. There is no shuffling — every epoch slices the same fixed contiguous blocks of X and Y in the same order, so if the data is sorted or grouped by digit, batches are non-representative every single epoch.
  3. cross_entropy is called before backward, which is the wrong order.
  4. The gradients should be added to the parameters, not subtracted.
Show answer

Answer: B. The loop always takes contiguous slices of X and Y in the same fixed order every epoch. If the underlying ZIP-code data was collected or stored grouped by digit (all 0s, then all 1s, and so on — a realistic scenario for scanned batches from a single sorting run), every mini-batch is systematically unrepresentative, and the same skewed batches repeat epoch after epoch, biasing the gradient estimate. The fix is to shuffle an index array once per epoch and slice through the shuffled indices, as shown in Lesson 3's training loop.

A is wrong — applying the update per-batch inside the loop is correct mini-batch behavior. C is wrong — computing the loss for logging and computing gradients for the update can happen in either order since both read from the same cache without mutating it. D is wrong — subtracting the gradient scaled by lr is exactly the correct gradient descent direction; adding it would ascend the loss instead of descending it.

Question 6 Multiple choice

A sorting facility needs to retrain its digit classifier overnight on 500,000 labeled crops using a single GPU built for vectorized matrix operations. Which gradient descent variant fits best, and why?

  1. Batch gradient descent, because it produces the most accurate gradient every step.
  2. Single-example stochastic gradient descent, because it takes the most update steps per epoch.
  3. Mini-batch gradient descent with a batch size like 128 or 256, because it keeps gradient noise moderate while matching the GPU's vectorized throughput.
  4. It does not matter, since all three variants take the same total wall-clock time for a fixed number of epochs.
Show answer

Answer: C. Mini-batches sized to match the GPU's parallel width get far more updates per pass over the data than full-batch descent, while producing a much less noisy gradient estimate than single-example SGD, and they use the hardware the way it was built to be used.

A is wrong in practice at this scale — one update per 500,000 examples means almost no progress per unit of wall-clock time, even though each individual gradient is highly accurate. B is wrong — single-example updates fail to use the GPU's parallel matrix hardware, wasting most of its throughput even though it does take more steps. D is wrong — batch size directly changes how many matrix multiplies happen per step and how well those multiplies use the hardware, so wall-clock time varies substantially between the three variants.

Question 7 Multiple choice

Training the same two-layer ZIP-code classifier for five epochs at three different learning rates produces these loss traces (starting near ln(10) ≈ 2.30, the loss of guessing uniformly among 10 digits):
Run 1: [2.30, 1.44, 0.97, 0.71, 0.55]
Run 2: [2.31, 4.02, 19.87, nan, nan]
Run 3: [2.30, 2.29, 2.28, 2.27, 2.26]
Which run used a learning rate that is too high?

  1. Run 1
  2. Run 2
  3. Run 3
  4. None — all three are healthy, just at different speeds.
Show answer

Answer: B, Run 2. The loss grows rather than shrinks and reaches nan within a few epochs — the signature of a learning rate large enough that each update overshoots the loss surface's curvature and the weights diverge.

Run 1 is the healthy run — steady, substantial decrease each epoch. Run 3 is not "healthy but slow," it is diagnostically a crawl: the loss barely moves off its random-guessing starting point after five full epochs, meaning the learning rate is too low to make practical progress, not merely a slower version of Run 1's curve.

Question 8 True / false

Gradient clipping — rescaling the gradient when its norm exceeds a threshold — is an effective fix for vanishing gradients in a deep sigmoid network.

  1. True
  2. False
Show answer

Answer: False. Gradient clipping caps a gradient that has grown too large, which addresses exploding gradients. Vanishing gradients are the opposite problem — the gradient has shrunk toward zero from many small local derivatives (like sigmoid's, at most 0.25) multiplied together across layers — and no amount of rescaling a near-zero value makes it carry more signal.

The confusion is understandable because both problems come from the same source, a long product of per-layer local gradients via the chain rule, and both are diagnosed by watching gradient magnitudes during training — but one problem is "too big" and the other is "too small," and clipping only addresses the former.
