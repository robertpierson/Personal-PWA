Lesson 1

Sequence labelling and tagging schemes

Named entity recognition (NER) is the task of finding spans of text that refer to a category of real-world thing — a drug, a dose, a disease, a symptom — and labelling each span with its type. The standard way to make this trainable is to turn it into sequence labelling: a per-token classification problem, where every sub-word token in a note gets exactly one label from a fixed tag set, and a fine-tuned encoder (recall the token-classification head from Course 3 Unit 5) predicts that label independently for each position in the sequence.

Take a fragment of a real clinical note: "Patient started on metformin 500 mg twice daily for type 2 diabetes; reports new-onset lactic acidosis." A human annotator reading this sees four things worth tagging: a drug (metformin), a dosage (500 mg twice daily), a disease (type 2 diabetes), and — critically — an adverse event (lactic acidosis). The job of NER is to recover exactly those four spans, with the right boundaries and the right type, from nothing but the token sequence.

BIO / IOB2: the workhorse scheme

The dominant scheme for turning "these words form a span of type DRUG" into per-token labels is BIO (also called IOB2). Every token gets one of three tag families: B-TYPE marks the first token of an entity of that type, I-TYPE marks every subsequent token inside the same entity, and O marks a token outside any entity. The dash-prefixed type (DRUG, DOSE, DISEASE, ADE for adverse drug event) is what makes BIO carry both boundary information and category information in one tag.

The rule that makes BIO strict is this: an I-TYPE tag is only valid immediately following a B-TYPE or another I-TYPE of the same type. A bare I-DRUG opening a sentence, or an I-DISEASE following an I-DRUG, is an illegal transition — there is no such thing as "continuing" an entity that never started, or continuing one entity as if it were another. This constraint sounds pedantic until Lesson 2, where it becomes the entire reason CRFs exist.

BIOES / BILOU: sharper boundaries

BIO conflates two different situations under B-: "this is a single-token entity" and "this is the first token of a multi-token entity" look identical from the tag alone unless the tagger also inspects the following token. BIOES (or equivalently BILOU: Begin, Inside, Last/End, Outside, Unit) splits this apart with two extra tags: E-TYPE (or L-) explicitly marks the last token of a multi-token entity, and S-TYPE (or U-) marks a single-token entity in its own right. For "500 mg twice daily" that's B-DOSE I-DOSE I-DOSE E-DOSE; for the single token "metformin" it's S-DRUG rather than BIO's plain B-DRUG.

The empirical payoff, documented across multiple NER benchmarks including clinical corpora like i2b2 and n2c2, is a measurable F1 bump — typically one to two points — because the model gets an explicit, learnable signal for "this span ends here," which matters enormously for dosage strings and multi-word disease names where getting the last token right (mg vs mg/day, "diabetes" vs "diabetes mellitus") is the difference between a usable extraction and a truncated one. The cost is a larger tag set (roughly double, since B/I/E/S all need per-type variants) and correspondingly more parameters in the final classification layer — usually a good trade for clinical text where boundary errors are expensive.

Span-based tagging as an alternative

An entirely different family skips per-token tags and instead scores spans directly: enumerate candidate start/end index pairs up to some maximum length, run each candidate span's pooled representation through a classifier that outputs an entity type or "not an entity," and keep the high-scoring, non-overlapping ones. This span-based approach costs more compute (candidate spans grow quadratically with sentence length) but sidesteps BIO's biggest structural weakness, which brings us to the reason clinical text specifically forces this conversation.

Nested and overlapping entities

Consider the phrase "stage II breast cancer." A clinician reading this note wants two overlapping facts extracted: that "breast cancer" is a disease entity, and that "stage II breast cancer" (or "stage II" as a modifier of the disease span) is itself a clinically meaningful staging entity. Under flat BIO, every token gets exactly one tag — there is no way to say that "breast" and "cancer" belong simultaneously to a DISEASE span and to a larger STAGE_DISEASE span, because BIO's data structure is a single label sequence, not a set of possibly-overlapping intervals. You could tag B-DISEASE I-DISEASE over "breast cancer" and lose the staging relationship, or invent a composite tag like B-STAGED_DISEASE and lose the fact that "breast cancer" alone is also a valid, separately-useful disease mention elsewhere in the same note. Neither captures both truths at once.

This is exactly the case where flat BIO fails and where practical systems reach for one of two escapes: run two independent flat taggers (one for diseases, one for stage/qualifier spans) and merge their outputs downstream, or move to a genuinely nested-NER architecture — span-based models are the natural fit here, since nothing stops two overlapping spans from both scoring above threshold, or a stack of BIO layers, one per nesting depth. Nested entities are common enough in biomedical text — "stage II breast cancer," "family history of type 2 diabetes," "suspected drug-induced lactic acidosis" — that clinical NLP pipelines budget for this from the start rather than treating it as an edge case.

The label-imbalance reality

One more property of sequence labelling that shapes everything downstream: in a typical clinical note, the overwhelming majority of tokens are O. A discharge summary might run 400 tokens and contain six entities spanning 20 of them — a 95%-O label distribution is normal, not pathological. A model can hit high token-level accuracy by essentially always predicting O, which is precisely why accuracy is the wrong metric here (Lesson 3 develops this in full) and why training typically down-weights or reweights the O class, or evaluates strictly on entity-level F1 rather than raw per-token accuracy.

sentence = ["Patient", "started", "on", "metformin", "500", "mg",
            "twice", "daily", "for", "type", "2", "diabetes", "."]

entities = [
    (3, 3, "DRUG"),      # metformin
    (4, 7, "DOSE"),      # 500 mg twice daily
    (9, 11, "DISEASE"),  # type 2 diabetes
]

def to_bio(tokens, entities):
    tags = ["O"] * len(tokens)
    for start, end, etype in entities:
        tags[start] = f"B-{etype}"
        for i in range(start + 1, end + 1):
            tags[i] = f"I-{etype}"
    return tags

def bio_to_spans(tags):
    spans, cur = [], None
    for i, tag in enumerate(tags):
        if tag.startswith("B-"):
            if cur: spans.append(cur)
            cur = [i, i, tag[2:]]
        elif tag.startswith("I-") and cur and tag[2:] == cur[2]:
            cur[1] = i
        else:
            if cur: spans.append(cur); cur = None
    if cur: spans.append(cur)
    return spans

tags = to_bio(sentence, entities)
print(list(zip(sentence, tags)))
print(bio_to_spans(tags))          # round-trips back to (3,3,'DRUG') etc.

# What BIO CANNOT represent: overlapping spans on the same tokens.
# "stage II breast cancer" needs BOTH:
#   breast, cancer          -> DISEASE
#   stage, II, breast, cancer -> STAGED_CONDITION
# A single flat tag sequence can only assign one label per token,
# so one of these two true spans must be dropped or merged.

The round trip (tags → spans → tags) works cleanly for flat, non-overlapping entities; the commented-out case shows the exact structural limit BIO hits on nested clinical mentions.

Key takeaways

  • BIO/IOB2 encodes both boundary (B vs I vs O) and type in one tag per token, with the hard constraint that I-TYPE must follow B-TYPE or I-TYPE of the same type.
  • BIOES/BILOU adds explicit end (E/L) and single-token (S/U) tags, typically buying 1–2 F1 points by giving the model a direct signal for span boundaries — at the cost of a larger tag set.
  • Flat tagging schemes assign exactly one label per token, so they structurally cannot represent nested or overlapping entities like "stage II breast cancer" without dropping one of the two true spans.
  • Span-based tagging (score candidate start/end pairs directly) trades higher compute for the ability to keep overlapping spans, and is the standard escape hatch for nested biomedical entities.
  • Clinical notes are dominated by O tokens (often 90%+), which makes token-level accuracy a misleading training and evaluation signal from the very first lesson of this unit.

Lesson 2

Models for NER: from CRF to BERT

Before transformers, extracting drugs and diagnoses from clinical text was a decade-long arms race against one specific failure mode: models that scored each token independently would happily emit O I-DRUG O or B-DISEASE I-DRUG — tag sequences that are locally plausible token-by-token but globally nonsensical, because nothing enforced the BIO grammar from Lesson 1 across positions. The history of NER modelling is largely the history of fixing that one problem, with increasingly powerful feature extractors bolted on top.

HMMs and the independence problem

The earliest statistical approach to clinical NER used Hidden Markov Models (HMMs): treat the tag sequence as a Markov chain, where each tag depends on the previous tag, and each observed token depends only on its own tag. HMMs got the sequential dependency right in principle but were weak feature extractors — they modelled P(token | tag) generatively, which made it awkward to use the rich, overlapping features that actually help in clinical text: capitalization ("Metformin" at a sentence start vs mid-sentence), suffix patterns (drug names ending in -mab, -nib, -statin), and neighboring tokens like "mg" or "daily" that signal a dosage is nearby. That weakness pushed the field toward discriminative models.

Linear-chain CRFs: modelling transitions explicitly

The linear-chain Conditional Random Field (CRF), dominant from roughly 2005 through the mid-2010s, fixed the independence problem directly. Instead of predicting each tag from local features alone, a CRF scores an entire tag sequence y given the input x using two kinds of learned scores: emission scores (how well tag y_i fits token x_i, from hand-engineered or embedding features) and transition scores (how well tag y_i follows tag y_{i-1}, learned as a small matrix over the tag set). Critically, the transition matrix can learn that the score for "I-DRUG follows O" should be effectively -∞, ruling out illegal BIO sequences by construction rather than hoping the model learns to avoid them.

score(x, y) = Σᵢ emission(yᵢ, xᵢ) + Σᵢ transition(yᵢ₋₁, yᵢ)

Finding the highest-scoring legal tag sequence out of the exponentially many possible sequences is done with Viterbi decoding, a dynamic-programming algorithm that reuses partial best-paths instead of enumerating every sequence. This is the same core idea as Viterbi decoding in speech recognition and in the forward-backward algorithm for HMMs, adapted here to maximize a discriminative score rather than a generative likelihood.

BiLSTM-CRF: 2015–2018's dominant architecture

Feature-engineered CRFs required a human to hand-craft the suffix rules, capitalization patterns, and dictionary-lookup features that fed the emission scores. The BiLSTM-CRF architecture (Lample et al. 2016, Huang et al. 2015) removed that manual step: a bidirectional LSTM reads the token sequence (often over word embeddings plus a character-level CNN or LSTM to capture morphology — useful for drug suffixes and dosage-unit tokens like "mg" or "mcg"), and its hidden states become the emission scores fed into the same CRF layer with its learned transition matrix. This combination — neural feature extraction, symbolic-style transition constraints — dominated clinical and biomedical NER benchmarks (i2b2, n2c2, BC5CDR) from roughly 2015 to 2018 and is still a reasonable baseline to cite today.

BERT token classification, with or without a CRF

Course 3 Unit 5 covered fine-tuning BERT for token classification: a linear head over each sub-word's final hidden state, trained with per-token cross-entropy. That setup is exactly the "independent per-token softmax" problem this lesson opened with — nothing stops it from predicting I-DRUG at the start of a sentence. In practice, BERT's contextual representations are so much stronger than an LSTM's that the model rarely makes egregious illegal-transition errors — it has implicitly learned the BIO grammar from data — and plain BERT token classification now matches or beats BiLSTM-CRF on most clinical NER benchmarks without an explicit transition matrix at all.

Adding a CRF layer on top of BERT (BERT-CRF) still buys a small, consistent F1 improvement in adverse-event and drug-dosage extraction tasks specifically, because these entities are short, formulaic, and boundary-sensitive ("500 mg" vs "500 mg twice daily"), and a hard transition constraint squeezes out the last few illegal-sequence errors a purely local softmax occasionally still makes. Whether the added training complexity — the CRF layer needs its own forward-backward pass during training and Viterbi at inference, both slower than a plain softmax — is worth a half-point to one-point F1 gain is a real engineering trade-off, not an automatic yes.

Constrained decoding: the lightweight alternative

A cheaper middle ground is constrained decoding: keep the plain per-token softmax from BERT, but at inference time mask out illegal transitions directly in the decoding loop — if the previous tag was O, zero out the probability of every I-* tag before taking the argmax (or before a lightweight beam search). This gets the "no illegal sequences" guarantee of a CRF's transition matrix without learning a transition matrix at all — the constraint is a fixed rule derived from the tagging scheme's grammar (Lesson 1's B/I/O rules), not a learned parameter. It adds no training cost and only a few lines of post-processing, which is why many production clinical-NLP pipelines use constrained decoding over a plain BERT tagger instead of paying for a full CRF layer.

Choosing between the four in practice

For a clinical NLP team scoping a new extraction pipeline today, the decision usually comes down to how much labelled data exists and how boundary-sensitive the entities are. A BiLSTM-CRF is rarely the first choice from scratch anymore — it needs more labelled examples to reach the same F1 a fine-tuned BERT model hits with a fraction of the data, because BERT arrives pretrained on enormous general and (with models like BioBERT or ClinicalBERT) biomedical-domain text, while a BiLSTM starts from either random weights or generic word embeddings. Where a BiLSTM-CRF still earns its place is in latency-constrained settings — an LSTM forward pass is cheaper than a transformer's, which matters if a hospital system needs to tag thousands of notes per minute on modest hardware. For dosage and adverse-event extraction specifically, where getting the exact right-hand boundary of a span wrong turns "500 mg twice daily" into a silently truncated and clinically wrong fact, teams lean toward BERT-CRF or at minimum constrained decoding rather than trusting a plain softmax to have fully internalized the BIO grammar on every rare or unusual phrasing. The safest default for a new project without a strong latency constraint is plain fine-tuned BERT token classification with constrained decoding bolted on at inference — nearly all of a CRF's benefit, none of its training overhead.

import numpy as np

tags = ["O", "B-DRUG", "I-DRUG", "B-DOSE"]
n = len(tags)

# emissions[t][k] = model's raw score for tag k at position t
emissions = np.array([
    [3.0, 0.5, -1.0, 0.2],   # "Patient"
    [0.1, 2.8,  1.0, 0.3],   # "metformin"
    [-2.0, -1.0, 3.5, 0.0],  # continuation piece (sub-word)
])

# transitions[i][j] = learned score for moving from tag i to tag j
transitions = np.array([
    [ 1.0, 0.8, -5.0, 0.6],   # from O
    [-5.0, 0.2,  2.5, 0.4],   # from B-DRUG
    [-5.0, 0.1,  1.8, 0.3],   # from I-DRUG (I-DRUG->I-DRUG stays high)
    [ 0.5, 0.4, -5.0, 0.2],   # from B-DOSE
])
# note: transitions into I-DRUG from O/B-DOSE are -5.0 -- effectively illegal

def viterbi(emissions, transitions):
    T, K = emissions.shape
    dp = np.full((T, K), -np.inf)
    back = np.zeros((T, K), dtype=int)
    dp[0] = emissions[0]                       # no incoming transition at t=0
    for t in range(1, T):
        for k in range(K):
            scores = dp[t-1] + transitions[:, k] + emissions[t, k]
            back[t, k] = np.argmax(scores)
            dp[t, k] = scores[back[t, k]]
    path = [np.argmax(dp[-1])]
    for t in range(T-1, 0, -1):
        path.append(back[t, path[-1]])
    path.reverse()
    return [tags[i] for i in path], dp[-1].max()

best_path, score = viterbi(emissions, transitions)
print(best_path)   # ['O', 'B-DRUG', 'I-DRUG'] -- legal, not ['O','B-DRUG','B-DOSE']

The heavily-penalized transitions (-5.0) into I-DRUG from anything but B-DRUG/I-DRUG are exactly the CRF's learned analogue of the BIO grammar rule; Viterbi's dynamic program finds the best legal path without ever enumerating all 4³ sequences.

Key takeaways

  • Independent per-token softmax (plain HMM emissions or a naive classification head) can emit illegal BIO sequences like an I-DRUG with no preceding B-DRUG — nothing in the model architecture forbids it.
  • A linear-chain CRF fixes this by scoring whole tag sequences with emission + transition scores, and Viterbi decoding finds the best legal sequence in polynomial time via dynamic programming.
  • BiLSTM-CRF (2015–2018) paired neural feature extraction with a CRF's transition constraints and was the dominant clinical NER architecture before BERT.
  • Plain fine-tuned BERT token classification now matches or beats BiLSTM-CRF on most benchmarks because its contextual representations implicitly learn the BIO grammar; BERT-CRF still adds a small, real gain on boundary-sensitive entities like dosages.
  • Constrained decoding — masking illegal transitions at inference time without learning a transition matrix — gets most of a CRF's benefit at a fraction of the training cost.

Lesson 3

Text classification and evaluation done right

A team builds a classifier to flag clinical notes that mention an adverse drug event, reports 96% accuracy on the held-out test set, and ships it. Three months later a pharmacovigilance auditor discovers the model missed a documented case of drug-induced lactic acidosis that led to a hospital readmission — the note was sitting in the "no adverse event" bucket the entire time. Nobody lied about the 96%; the number was just measuring the wrong thing. If 96% of clinical notes in the dataset mention no adverse event at all, a classifier that outputs "no adverse event" unconditionally, never looking at the text, scores 96% accuracy and catches zero real cases. This is the single most common way a text classification project fails in a clinical setting, and it is entirely avoidable once you stop trusting accuracy on imbalanced data.

Multi-class vs multi-label

Before evaluation, get the problem framing right. Multi-class classification assumes each note gets exactly one label from a set — useful for something like "which single specialty authored this note" when that's genuinely mutually exclusive. But a real clinical note routed to a triage queue is often legitimately about more than one thing: a note describing a diabetic patient with new cardiac symptoms is both endocrinology and cardiology relevant, and forcing a single label throws away information the downstream routing system needs. That's a multi-label problem — each note gets a subset of labels, scored with independent sigmoid outputs per label rather than one softmax over all labels, and evaluated per-label rather than assuming labels compete with each other.

The class-imbalance trap

Adverse-event detection, rare-disease flagging, and most safety-relevant clinical classification tasks share the same shape: the positive class is rare (often single-digit percentages) and the negative class dominates. Accuracy weights both classes' errors identically per example, which means it is dominated by whichever class has more examples — exactly backwards from what a safety-critical task needs, since a missed adverse event (a false negative) is far costlier than a false alarm (a false positive) that a human reviewer dismisses in ten seconds. Any time someone reports a bare accuracy number on a clinical classification task without stating the class balance, treat the number as meaningless until you can see the confusion matrix.

Precision, recall, F1, and the safety-critical trade

Precision is, of the notes the model flagged as containing an adverse event, what fraction actually did — it answers "when the model raises an alarm, can I trust it?" Recall is, of the notes that actually contained an adverse event, what fraction the model caught — it answers "of the real problems out there, how many did I miss?" F1 is their harmonic mean, a single number balancing both, useful for model comparison but not for setting the actual operating point of a deployed system.

precision = TP / (TP + FP) recall = TP / (TP + FN) F1 = 2 · precision · recall / (precision + recall)

For adverse-event detection specifically, the clinical stakes push hard toward prioritizing recall: missing a true adverse event (false negative) can mean a preventable readmission or worse, while a false positive just costs a reviewer a few minutes reading a flagged note that turns out benign. Most deployed clinical-safety classifiers are tuned to a target recall (say, 95%) and accept whatever precision comes with it, rather than optimizing F1 in the abstract — this is a deliberate business/clinical decision about which error type is tolerable, not a purely statistical one.

Micro, macro, and weighted averaging

When there are multiple classes or labels — say, seven clinical specialties for note routing — a single precision/recall/F1 number requires choosing how to average across classes, and the three standard choices tell different stories. Micro-averaging pools every true positive, false positive, and false negative across all classes before computing one precision/recall/F1 — this is dominated by frequent classes, so if 60% of notes are internal-medicine, micro-F1 mostly reflects performance on internal medicine. Macro-averaging computes precision/recall/F1 per class independently, then takes the unweighted mean across classes — every class counts equally regardless of frequency, so a rare specialty like rheumatology influences the score exactly as much as the dominant one, which is usually the honest choice when rare classes matter clinically (a rare-disease flag that the model handles poorly shouldn't be masked by strong performance on common categories). Weighted-averaging is macro's per-class scores weighted by each class's support (how many true examples it has) — a middle ground that reflects real-world class frequency without letting one dominant class fully define the number.

MicroPool TP/FP/FN across all classes, then one P/R/F1Overall system throughput matters more than per-class fairness

Entity-level vs token-level F1 for NER

Lesson 1 and 2's NER models need their own evaluation subtlety: token-level F1 scores every individual token's tag prediction, while entity-level F1 (the standard for NER leaderboards, following the CoNLL evaluation convention) counts a predicted entity as correct only if its full span boundaries and type exactly match the gold entity — get "500 mg twice daily" tagged as just "500 mg" and entity-level scoring counts that as a complete miss (a false negative for the true span plus a false positive for the wrong one) even though most of the tokens were tagged correctly. Some evaluation setups relax this to partial-match scoring, giving partial credit for overlapping-but-not-identical spans — useful diagnostically, but the strict exact-match entity-level F1 remains the standard reported number because a clinician reading "500 mg" when the actual dose was "500 mg twice daily" has been given a genuinely wrong, actionable fact, not a nearly-right one.

Threshold tuning

A classifier's sigmoid or softmax output is a probability, not a decision — the 0.5 default cutoff is an arbitrary convention, not a law. Sliding that decision threshold trades precision against recall along a single trained model's precision-recall curve: lowering the threshold for the adverse-event class catches more true cases (higher recall) at the cost of more false alarms (lower precision). Given a clinical recall target — "we must catch at least 95% of true adverse-event mentions" — the right workflow is to pick the lowest threshold that still hits that recall on a validation set, then report the precision that comes with it, rather than accepting whatever the default threshold happens to produce.

import numpy as np

# Confusion matrix for 3 note-routing classes on 500 imbalanced notes.
# Rows = true label, columns = predicted label.
classes = ["cardiology", "endocrinology", "rheumatology"]
cm = np.array([
    [300,  15,   5],   # true cardiology   (320 total)
    [ 10, 140,   5],   # true endocrinology (155 total)
    [  2,   3,  20],   # true rheumatology  (25 total, the rare class)
])

def per_class_prf(cm):
    n = len(cm)
    out = []
    for i in range(n):
        tp = cm[i, i]
        fp = cm[:, i].sum() - tp
        fn = cm[i, :].sum() - tp
        p = tp / (tp + fp) if (tp + fp) else 0.0
        r = tp / (tp + fn) if (tp + fn) else 0.0
        f1 = 2 * p * r / (p + r) if (p + r) else 0.0
        out.append((p, r, f1, cm[i, :].sum()))
    return out

scores = per_class_prf(cm)
for cls, (p, r, f1, support) in zip(classes, scores):
    print(f"{cls:14s} P={p:.2f} R={r:.2f} F1={f1:.2f} (n={support})")

macro_f1 = np.mean([s[2] for s in scores])
weighted_f1 = np.sum([s[2] * s[3] for s in scores]) / cm.sum()
print(f"macro-F1={macro_f1:.3f}  weighted-F1={weighted_f1:.3f}")
# rheumatology's weak F1 (~0.74) drags macro-F1 down noticeably;
# weighted-F1 barely moves because rheumatology is only 25 of 500 notes.

Macro-F1 exposes the rare rheumatology class's weaker performance; weighted-F1 nearly hides it because the class has so little support — exactly the gap that makes averaging choice a clinical, not just statistical, decision.

Key takeaways

  • High accuracy on an imbalanced clinical dataset is easy to achieve and easy to be meaningless — always check class balance before trusting an accuracy number.
  • A note can belong to multiple specialties at once, making note-routing a multi-label problem (independent per-label sigmoids), not a multi-class one forcing a single label.
  • Recall should usually be prioritized over precision for adverse-event detection, since a missed event is far costlier than a false alarm a reviewer dismisses quickly.
  • Macro-averaging is the honest choice when rare classes (rare diseases, rare adverse events) must not be masked by common ones; micro and weighted averaging both let frequent classes dominate the number to varying degrees.
  • Entity-level F1 for NER requires an exact span-and-type match — a partially correct dosage span counts as a full miss, matching the real clinical cost of a truncated fact.

Question 1 Multiple choice

A clinical note contains the phrase "stage II breast cancer," and your annotation guidelines want both "breast cancer" tagged as DISEASE and "stage II breast cancer" tagged as a separate STAGED_CONDITION span. Which statement about representing this with flat BIO tagging is correct?

  1. Flat BIO handles this fine as long as you use BIOES instead of plain BIO.
  2. Flat BIO cannot represent both spans simultaneously, because each token gets exactly one tag; you'd need to drop one span, merge them into a composite tag, or move to span-based/nested tagging.
  3. Flat BIO handles this by tagging "II" as O and the rest as normal DISEASE tokens.
  4. This is only a problem for BILOU, not for BIO/IOB2.
Show answer

Answer: B. BIO's data structure is a single label per token, so two overlapping spans covering the same tokens with different types cannot both be represented without losing information about one of them.

Option A is wrong because BIOES also assigns exactly one tag per token — adding E/S tags sharpens boundaries within a single flat sequence but does nothing to enable overlapping spans. This is a structural limit of flat tagging in general, not something a richer tag vocabulary fixes.

Question 2 Fill in the blank

In a linear-chain CRF, the algorithm used to find the highest-scoring legal tag sequence out of exponentially many candidates, using dynamic programming over emission and transition scores, is called   decoding.

Show answer

Answer: Viterbi. It builds up the best partial-path score for each tag at each position, reusing previous positions' best scores rather than re-scoring every possible full sequence from scratch.

A tempting but wrong answer here is "beam search" — beam search is an approximate, general-purpose search heuristic used in many sequence-generation settings, but Viterbi specifically is the exact dynamic-programming solution for chain-structured scoring like a CRF's, and it's guaranteed optimal, not approximate.

A model that classifies clinical notes for the presence of a rare adverse drug event reports 97% accuracy on a test set where only 3% of notes contain a true adverse event. This accuracy figure, by itself, is strong evidence the model is clinically useful.

  1. True
  2. False

Option A is wrong because closing the previous span on a new B- is correct BIO behavior, not the bug. Option C doesn't address the actual defect — a dict wouldn't stop the type-mismatch merge. Question 5 Multiple choice

Your team is choosing an averaging method for reporting F1 across seven note-routing specialty classes. Rheumatology makes up only 2% of notes, but a missed rheumatology-relevant note has serious clinical consequences and your team wants the reported metric to reflect the model's weakness on that class rather than let it hide behind strong performance on common specialties like internal medicine. Which averaging method should you report?

  1. Micro-averaged F1, because it reflects overall system throughput.
  2. Weighted-averaged F1, because it accounts for how rare rheumatology notes actually are.
  3. Macro-averaged F1, because it weights every class equally regardless of frequency, so rheumatology's score isn't diluted by common classes.
  4. Accuracy, since it's the simplest metric to communicate to stakeholders.
Show answer

Answer: C. Macro-averaging computes each class's F1 independently and takes an unweighted mean, so rheumatology's F1 counts exactly as much as internal medicine's, surfacing a weak rare class instead of letting it get outvoted.

Weighted averaging (B) would weight rheumatology's contribution by its 2% support, meaning a rheumatology failure barely moves the number — the opposite of what the team wants. Micro-averaging (A) pools raw counts across classes, so it's dominated by whichever class has the most notes. Accuracy (D) doesn't even decompose per class and inherits the same imbalance problems covered in Lesson 3.

Answer: B.
