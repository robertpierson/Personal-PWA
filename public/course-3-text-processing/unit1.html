Text preprocessing and tokenization fundamentals

Before any moderation model sees a post, someone decides what "the text" even is — and every one of those decisions throws away information. This unit re-grounds preprocessing and tokenization from scratch, using the messiest text on the internet: raw social posts.

Lesson 1

The preprocessing pipeline

Somewhere on a review queue sits this post: SOOO excited!!! 😍 #blessed @friend http://x.co. A moderation model — say, one flagging spam, harassment, or coordinated bot activity — never sees those sixty-odd characters directly. Something has to turn them into the fixed-shape numeric input a model expects, and whoever writes that "something" makes a dozen small decisions along the way, each of which quietly deletes information the model might have needed. Strip the emoji and you lose the sentiment amplifier that tells you this is excitement, not sarcasm. Lowercase the caps and you lose the shouting. Delete the URL and you lose the exact signal that this post is trying to route traffic somewhere, which is often the whole point of a spam classifier's job. Preprocessing is never neutral cleanup — it is a sequence of modelling decisions disguised as hygiene, and the goal of this lesson is to make each one visible so you can choose it on purpose instead of by default.

It helps to say plainly what preprocessing is for. Raw text arrives as a stream of Unicode codepoints with essentially unlimited variation: any user can type any sequence of any of the roughly 150,000 characters Unicode defines, in any order, with invisible formatting marks mixed in. A model's vocabulary and downstream layers assume far more regularity than that — the same word should usually map to the same representation, invisible characters should not silently change meaning, and semantically equivalent inputs should end up close together. Preprocessing is the bridge between those two worlds. It is lossy by construction — you cannot both compress a space of near-infinite variation into a small, reusable vocabulary and preserve every distinction the raw text carried. The question is never "how do I remove noise," it is "which distinctions can this particular model, for this particular task, afford to lose."

Unicode normalisation: NFC, NFKC, and why it's a security question

The first problem is that the "same" text can arrive as different byte sequences. The character é can be one codepoint (U+00E9, precomposed) or two (U+0065 "e" plus U+0301 a combining acute accent) — visually identical, but a naive string-equality check or a tokenizer's lookup table sees them as different tokens. Unicode normalisation fixes this by rewriting text into one of a handful of canonical forms. NFC (Normalization Form Canonical Composition) prefers the precomposed form wherever one exists, which is usually what you want for display and for a model vocabulary trained on typical text. NFKC (Compatibility Composition) goes further: it also collapses "compatibility" variants that are stylistically different but semantically identical — the fullwidth "Ａ" used in some East Asian typesetting becomes ordinary "A," a ligature like "ﬁ" becomes "f" + "i," superscript "²" becomes "2." For moderation pipelines specifically, this is not a cosmetic nicety — it is a security control. Adversarial posters use homoglyph substitution, swapping Latin letters for visually near-identical characters from other scripts (Cyrillic "а" for Latin "a," to dodge a keyword filter on a banned word), and they use invisible characters like zero-width spaces or joiners to break up a filtered string without changing how it renders. NFKC normalisation, combined with stripping known invisible-format codepoints, closes a real fraction of these evasions before the text ever reaches tokenization — though never all of them, since new homoglyph tricks appear as fast as filters catch old ones.

Case folding: not always a win

Lowercasing everything is the oldest reflex in text preprocessing, and for many tasks it is still correct: it collapses "Excited," "excited," and "EXCITED" to one vocabulary entry, shrinking vocabulary size and data sparsity. But notice what that collapse destroys for a moderation model specifically. ALL-CAPS is a real signal — of shouting, aggression, or excitement — and case is one of the cheapest features a classifier can use to detect it. Case also disambiguates named entities from common words ("US" the country versus "us" the pronoun, "March" the month versus "march" the verb), which matters for entity-aware moderation like doxxing detection. The honest rule is: fold case when your downstream representation is sparse (word counts, classic bag-of-words features) and case variants would otherwise fragment your vocabulary pointlessly; keep case when your model is expressive enough to use it as signal (subword or character-level neural models, which can learn "all-caps span" as a pattern) or when the task explicitly needs it. There is no universally correct answer — only the one your task and your downstream tokenizer can afford.

URLs, @mentions, and #hashtags: placeholder, keep, or drop

Social posts are dense with three structural elements that plain prose never has, and each admits three treatments with different costs. You can drop them entirely (delete the URL, mention, hashtag text), which shrinks the input but destroys any signal the element carried — dropping every URL makes it impossible to tell "post with a suspicious shortened link" from "post with no link at all." You can keep them verbatim, which preserves everything but explodes vocabulary size, since URLs and usernames are close to unique strings that will appear once each in the entire dataset and contribute nothing generalizable to a model. Or you can replace with a placeholder — a fixed token like <URL>, <MENTION>, <HASHTAG> — which is usually the right middle ground for moderation: the model learns "this post contains a link" as a reusable, generalizable feature without ever needing to memorize the specific link. Hashtags deserve special care, though, because unlike mentions and URLs they often carry real linguistic content: #blessed is not noise to discard, it is a sentiment-bearing word glued to a symbol. A common compromise is to strip the leading # but keep the word, and only fall back to a placeholder for hashtags that are pure keyword salad with no readable word boundary.

Emoji as signal, not noise

Early NLP pipelines stripped emoji as "non-text noise" reflexively, and for moderation that is close to a modelling error. Emoji frequently carry the sentiment or intent of a post more reliably than the words around them — 😍 and 😡 disambiguate tone in exactly the short, context-free posts where moderation models struggle most, and specific emoji sequences function as adversarial code themselves (certain fruit or plant emoji are used as drug-sale euphemisms, for instance). Modern subword tokenizers handle emoji natively as Unicode codepoints, so keeping them costs little; the decision that matters is whether to normalise emoji variants (skin-tone modifiers, gender variation selectors) to a base form, trading some nuance for a smaller, more learnable emoji vocabulary.

Whitespace and control characters

The least glamorous step is also the one that breaks pipelines silently: collapsing runs of whitespace, stripping non-printing control characters, and normalising line endings. Elongated words like "soooo goooood" are a whitespace-adjacent problem too — they are not extra spaces, but repeated-character runs that a naive tokenizer treats as unknown words distinct from "so good," even though they carry emphasis a moderation model benefits from detecting (repetition often correlates with strong emotion or bot-like scripted spam). A common move is to cap character repetition at two or three ("soooo" → "soo") rather than collapsing to one, preserving the "this was emphasized" signal while still controlling vocabulary blowup.

import unicodedata, re

def preprocess(text: str) -> str:
    print(f"0. raw:         {text!r}")

    # 1. Unicode normalisation — collapse homoglyphs/compatibility forms
    text = unicodedata.normalize("NFKC", text)
    print(f"1. NFKC:        {text!r}")

    # 2. Strip invisible/control characters (zero-width space, etc.)
    text = re.sub(r"[​‌‍﻿]", "", text)
    print(f"2. no ZW chars: {text!r}")

    # 3. Cap repeated characters at 2 (soooo -> soo), preserves emphasis signal
    text = re.sub(r"(.)\1{2,}", r"\1\1", text)
    print(f"3. de-elongate: {text!r}")

    # 4. Replace URLs and @mentions with placeholders; keep hashtag words
    text = re.sub(r"https?://\S+", "", text)
    text = re.sub(r"@\w+", "", text)
    text = re.sub(r"#(\w+)", r"\1", text)
    print(f"4. placeholders:{text!r}")

    # 5. Collapse whitespace runs
    text = re.sub(r"\s+", " ", text).strip()
    print(f"5. whitespace:  {text!r}")

    return text

preprocess("SOOO excited!!! 😍 #blessed @friend http://x.co")

Case is deliberately left untouched here — notice how each stage transforms the string incrementally, and how the final output still keeps "SOOO," the emoji, and "blessed" as meaningful tokens rather than deleting them.

Key takeaways

  • Preprocessing is always lossy — there is no "clean" version of a text that preserves all information; every step is a trade-off you should make deliberately for your task.
  • NFKC normalisation is a security control for moderation systems, not just cosmetic cleanup — it closes homoglyph and compatibility-character evasions of keyword filters.
  • Case folding helps sparse bag-of-words models but can delete shouting/emphasis signal that a moderation classifier specifically needs.
  • URLs, mentions, and hashtags each need their own treatment — placeholder tokens usually beat both keeping the raw string and dropping it outright.
  • Emoji and repeated-character elongation carry sentiment and intent — capping repetition instead of collapsing it preserves that signal while still controlling vocabulary size.

Lesson 2

Tokenization: word, subword, character

A token is the smallest unit of text a model treats as one item — the atomic thing that gets mapped to a vector, attended over, and predicted. That definition sounds simple until you try to pin down where one token ends and the next begins in real text, and social posts are exactly where the simple version breaks. This lesson works through three ways of drawing token boundaries — splitting on words, splitting on learned subword pieces, and splitting on raw characters or bytes — and shows concretely where each one wins or fails on noisy, adversarial user text.

Whitespace and rule-based tokenization — and where it breaks

The naive approach is to split on whitespace and call each resulting chunk a token. It is fast, requires no training data, and works fine on tidy formal prose. It breaks in at least four ways that matter for social text. Contractions like "don't" split inconsistently — is that one token or two ("do" + "n't")? Different downstream tasks want different answers, and whitespace alone can't decide. Hyphenated and multi-word units like "New York" or "co-worker" are semantically one thing but whitespace or punctuation splits them into two or three tokens, losing the fact that "New" and "York" together mean something a model can't recover from the parts alone. Punctuation glued to words — "excited!!!", "money?!" — needs its own splitting rule, or the tokenizer ends up with "excited!!!" as one opaque token distinct from every other use of "excited." And emoji and emoji sequences glued to text with no whitespace ("goooood😍😍") are invisible to a tokenizer that only looks for space characters. A slightly better approach layers regex rules on top of whitespace splitting: match words, punctuation, contractions, and emoji as separate regex classes and split at every boundary between classes. This is the regex tokenizer approach, and it is what the classic Penn Treebank (PTB) tokenization conventions formalize — a set of rules from the 1990s Penn Treebank corpus project that split contractions ("don't" → "do" + "n't"), separate most punctuation into its own token, and handle common abbreviations as exceptions. PTB-style tokenization is still the reference standard many English NLP tools default to, precisely because it made these judgment calls explicit and reusable instead of leaving every project to invent its own.

The out-of-vocabulary wall

Whitespace and regex tokenizers share a deeper problem once you fix a vocabulary size: every token that was not seen during training is out-of-vocabulary (OOV), typically mapped to a generic <UNK> token that throws away everything distinctive about that word. Formal prose has a fairly closed vocabulary — a large training corpus covers the vast majority of words a new document will contain. Social-media text does not. Users invent spellings constantly: elongation ("goooood"), leetspeak substitution ("fr€€ m0ney," "m0ney" for "money"), deliberate misspelling to dodge filters, and code-switching between languages mid-sentence. Every one of these generates a "new word" a word-level tokenizer has never indexed, and a word-level vocabulary trained on even millions of posts still hits a wall of UNK tokens on next week's slang and this week's obfuscation attempts. This is precisely the OOV problem that motivated the shift to subword tokenization industry-wide.

Subword tokenization: the modern default

Subword tokenization — BPE (byte-pair encoding) and WordPiece chief among them, covered in mechanical detail in Course 2 Unit 3 — solves the OOV problem by building a vocabulary of frequent character sequences rather than whole words, so any string can be represented as some sequence of known pieces even if the whole word was never seen. "m0ney" might not be in the vocabulary as one unit, but "m", "0", "ney" almost certainly are, since a subword vocabulary is built to guarantee full coverage of its training alphabet. That is exactly why subword schemes dominate the noisy-text case: a user who invents "goooood" or swaps letters for symbols to dodge a keyword filter cannot escape a subword tokenizer's coverage the way they escape a word-level vocabulary's fixed list, because subword pieces compose all the way down to individual characters when nothing larger matches. The cost is longer token sequences for unusual text — a heavily obfuscated post might tokenize into twice as many pieces as clean prose — but "more tokens, fully represented" beats "fewer tokens, mostly UNK" for almost every downstream task.

Character and byte tokenization: the fully robust fallback

Push subword splitting to its logical extreme and you get character-level tokenization — every individual Unicode codepoint (or, one level lower, every raw byte) is its own token. This has a vocabulary so small it is trivially complete: there is no such thing as an OOV character, because every possible input decomposes into the same closed alphabet, whether that's roughly 150,000 Unicode codepoints or, at the byte level, exactly 256 values. Character and byte tokenizers are the honest answer to "what handles truly arbitrary adversarial text," including homoglyph attacks, zero-width character insertion, and mixed-script obfuscation that would confuse a subword vocabulary trained mostly on clean English. The trade-off is sequence length: a ten-word post becomes ten tokens under a word tokenizer, roughly fifteen to twenty-five under subword tokenization, and fifty-plus characters under character tokenization — and since transformer compute scales with sequence length, character-level models pay substantially more compute per unit of text, which is exactly why production systems use subword tokenization as the default and reserve character/byte-level approaches for cases where robustness to arbitrary input matters more than efficiency.

Choosing a tokenizer in practice

None of these three approaches is "the" right tokenizer in the abstract — production moderation systems pick based on what they can afford to get wrong. A first-pass spam filter running on every post at ingestion time, where latency and throughput dominate, often still leans on cheap regex or word-level splitting for a coarse pre-filter, accepting some OOV loss because the downstream human reviewer is a safety net. A neural classifier making the actual moderation decision almost always uses subword tokenization matched to whatever pretrained transformer it's built on, because retraining a tokenizer from scratch throws away the pretraining the whole model depends on — the tokenizer is effectively part of the model's frozen interface, not a preprocessing detail you swap freely. Character or byte-level tokenization earns its cost specifically in adversarial-robustness settings: a system explicitly designed to catch obfuscated banned terms, homoglyph tricks, or mixed-script evasion may run a lightweight byte-level pass as a second check precisely because it cannot be fooled by any string a subword vocabulary hasn't seen. Notice that this mirrors Lesson 1's central lesson exactly — there is no universally correct tokenizer, only the one that fits the failure mode you're guarding against.

import re

posts = [
    "SOOO excited!!! 😍 #blessed",
    "don't trust this seller, it's a scam",
    "fr€€ m0ney click now!!",
    "New York is lit tonight 🔥🔥🔥",
    "lol going ¯\\_(ツ)_/¯ ok whatever",
]

def whitespace_tokenize(s):
    return s.split()

def regex_tokenize(s):
    # words, standalone punctuation runs, and emoji as their own tokens
    return re.findall(r"\w+|[^\w\s]", s, re.UNICODE)

def char_tokenize(s):
    return list(s.replace(" ", ""))  # drop plain spaces, keep every other char

for post in posts:
    ws, rx, ch = whitespace_tokenize(post), regex_tokenize(post), char_tokenize(post)
    print(f"{post!r}")
    print(f"  whitespace: {len(ws):>2} tokens  {ws}")
    print(f"  regex:      {len(rx):>2} tokens  {rx}")
    print(f"  char:       {len(ch):>2} tokens")
    print()

On "fr€€ m0ney click now!!" the whitespace tokenizer produces only three tokens ("fr€€", "m0ney", "click", "now!!" — none of which match a normal-English vocabulary and would mostly become UNK); the regex tokenizer splits the punctuation and symbols off, giving a real subword-style tokenizer more recognizable pieces to work with ("fr", "€", "€", "m0ney" still whole); only character tokenization guarantees zero unknown tokens, at the cost of roughly 4x the token count.

Key takeaways

  • A token is whatever unit a model treats as atomic — the boundary is a design choice, not a fact about language.
  • Whitespace tokenization fails on contractions, multi-word entities, glued punctuation, and glued emoji — all common in social posts.
  • Word-level vocabularies hit an OOV wall on invented spellings, elongation, and leetspeak obfuscation — exactly the adversarial patterns moderation systems must catch.
  • Subword tokenization (BPE/WordPiece) solves OOV by composing rare or unseen words from frequent sub-pieces, which is why it is the modern default for noisy user text.
  • Character/byte tokenization has a closed, complete vocabulary and cannot produce true OOV, but pays for that robustness in much longer sequences and higher compute per post.

Lesson 3

Stemming, lemmatization, stopwords, and the classical normalisation stack

In 1980, Martin Porter published a short, deliberately crude algorithm for stripping suffixes off English words — five phases of hand-written rules that turn "moderation," "moderating," and "moderated" all into "moder." The Porter stemmer was built for a world where computing a document's relevant words mattered more than computing them elegantly: the information retrieval (IR) systems of the 1970s and 80s indexed documents by exact word match, and a query for "moderate" needed to find documents containing "moderation" or the search was useless. Stemming solved a real, expensive problem cheaply, using no dictionaries, no part-of-speech tagging, and almost no computation — just suffix-stripping rules a machine from that era could run over a full document collection in reasonable time. That history matters because it explains both why this classical stack was once treated as mandatory in nearly every NLP pipeline, and why almost none of it is mandatory anymore. Understanding what changed is as important as understanding the techniques themselves.

Stemming: fast, crude, and rule-based

Stemming chops affixes off words using fixed rules, with no awareness of what part of speech the word is or whether the result is a real word at all. The Porter stemmer and its successor, the Snowball stemmer (a more systematic, multi-language rewrite Porter himself later produced), both do this: "running" becomes "run," but "universal," "university," and "universe" can all collapse toward "univers" — a string that is not an English word, because the algorithm has no model of meaning, only pattern-matching on letter endings. For a moderation pipeline, this crudeness cuts both ways. It usefully collapses spam variants ("clicking," "clicked," "click," "clicks" all becoming one indexed form) so a keyword-based filter catches all of them with one rule. But it also over-collapses in ways that lose distinctions moderation cares about — a stemmer can conflate "bullying" and "bully" (fine) with "bull" (a completely different word, if the stemmer is aggressive enough to strip "-y"), creating false-positive matches on an unrelated term.

Lemmatization: slower, correct, dictionary-driven

Lemmatization takes the more expensive, more correct path: it looks up a word's lemma — its canonical dictionary form — using an actual dictionary or morphological database, and typically needs to know the word's part of speech first, since the same surface form lemmatizes differently depending on grammatical role ("saw" the past tense of "see" versus "saw" the tool are different lemmas requiring different lookups). Because it consults real linguistic knowledge rather than pattern-matching suffixes, lemmatization never produces a non-word the way aggressive stemming can — "universities" correctly lemmatizes to "university," never to a fragment like "univers." The cost is speed and pipeline complexity: lemmatization typically requires a part-of-speech tagger to run first, plus a dictionary lookup, both of which are meaningfully slower than stemming's direct rule application, and both of which can themselves fail on the ungrammatical, fragment-heavy sentences typical of social posts, where a POS tagger trained on formal text may mistag words in a nonstandard slangy sentence and send lemmatization down the wrong path.

Why lemmatization struggles on the same text that breaks tokenizers

Lemmatization's dependence on part-of-speech tagging creates a second-order version of the exact noisy-text problem this unit keeps returning to. POS taggers, like most classical NLP tools, were trained overwhelmingly on grammatical, punctuated, formally-cased prose — news text, treebank sentences, edited writing. Feed one a real moderation-relevant fragment like "fr€€ m0ney click now!!" or "lol going ¯\\_(ツ)_/¯ ok whatever" and there is barely a well-formed clause for the tagger to anchor on: no reliable capitalization to mark proper nouns, no consistent punctuation to mark clause boundaries, slang verbs used as nouns and back again. A mistagged word sends the lemma lookup to the wrong dictionary entry entirely, which is a quieter failure than stemming's crude suffix errors but not a better one — it just moves the error upstream into a component that looks authoritative because it consulted a real dictionary. This is one more reason production moderation systems increasingly route around the whole classical stack for the neural half of the pipeline, reserving stemming and lemmatization for narrower, more controlled contexts — a curated banned-word list, say, where the input vocabulary is small and clean enough that a stemmer's crude rules or a lemmatizer's dictionary lookups both behave predictably, rather than the wide-open space of arbitrary user posts.

Stopword removal — and when it destroys meaning

Stopwords are the small set of extremely frequent function words — "the," "a," "is," "to," "not," "and" — that classical IR systems removed before indexing, on the reasoning that words appearing in nearly every document carry no discriminating power for retrieval. For plain document search this reasoning mostly holds. It fails hard the moment negation or exact phrasing carries the meaning. Shakespeare's "to be or not to be" is built entirely from words on a standard English stopword list — "to," "be," "or," "not" — and stripping them leaves nothing at all. More concretely for moderation: "not going to hurt you" and "going to hurt you" differ by exactly one stopword, "not," and mean opposite things; a pipeline that blindly strips stopwords before a threat-detection model sees the text has just erased the single word that decided the sentence's meaning. The safe version of stopword removal restricts itself to tasks where word-order and negation genuinely don't matter — coarse topic classification or document retrieval — and is switched off entirely for anything sentiment-, negation-, or intent-sensitive, which describes most moderation tasks.

N-grams and the honest modern verdict

One classical technique survives this whole shift essentially unscathed: n-grams, contiguous sequences of n tokens treated as a unit ("not going" as a bigram, rather than "not" and "going" scored separately). Preserving n-grams — or simply not stemming/stopword-filtering them apart — is one direct fix for the negation problem above, since "not_hurt" as one bigram feature survives stopword removal in a way that "not" and "hurt" scored independently do not. The larger honest verdict, though, is this: nearly the entire classical stack — stemming, lemmatization, and stopword removal — is skipped when preprocessing input for a modern subword transformer. A transformer's subword tokenizer already handles morphological variants implicitly, by breaking "running," "runs," and "run" into overlapping subword pieces the model learns to treat similarly through its own training, and its attention mechanism learns which words matter for a given prediction without needing frequency-based stopword heuristics decided in advance. What survives is narrower but real: classical stemming, lemmatization, and stopword removal still live inside sparse retrieval systems (keyword search indexes, some feature-engineered classical ML pipelines) where reducing vocabulary size and exact-match brittleness is still the whole game, and inside old-fashioned bag-of-words or TF-IDF feature pipelines for the same reason. Porter's 1980 rules did not become useless; they became a specialist tool for a narrower set of problems than they used to own.

# crude illustration of stemmer over-aggression vs. lemmatizer correctness
# (a real project would use nltk.PorterStemmer / spacy's lemmatizer;
#  this hand-rolled version shows the *mechanism*, not production code)

def toy_stem(word):
    for suffix in ("ational", "ing", "edly", "ed", "ies", "ly", "es", "s", "y"):
        if word.endswith(suffix) and len(word) - len(suffix) >= 3:
            return word[: -len(suffix)]
    return word

# lemma lookup pretending to be a dictionary + POS-aware system
LEMMA_TABLE = {
    "bullying": "bully", "bullied": "bully", "bullies": "bully",
    "universities": "university", "universal": "universal",
    "running": "run", "ran": "run", "saw": "see",
}

words = ["bullying", "bullied", "universities", "universal", "running", "ran", "saw"]

for w in words:
    stem = toy_stem(w)
    lemma = LEMMA_TABLE.get(w, w)
    flag = "  <- non-word fragment" if stem not in ("run", "see", "univers") and not stem.isalpha() is False and len(stem) < 4 else ""
    print(f"{w:14} stem={stem:12} lemma={lemma}")

Watch "universities" and "universal": the toy stemmer strips "-al" and "-ies" respectively and lands both near "univers," falsely suggesting they're related, while the lemma table correctly keeps "universal" as its own word and maps "universities" to "university" — the exact failure mode that makes stemming risky for moderation keyword matching.

Key takeaways

  • The Porter stemmer (1980) and the classical normalisation stack were built for 1970s–80s information retrieval, where cheap exact-match indexing mattered more than linguistic correctness.
  • Stemming is fast and rule-based but can produce non-words and false conflations (e.g., "bully" collapsing toward "bull") — lemmatization is slower and dictionary-driven but stays linguistically correct.
  • Stopword removal can delete the exact word that carries meaning — negation words like "not" are stopwords, and removing them can flip a sentence's meaning for a moderation model.
  • N-gram features are one direct fix for the negation problem, since a bigram like "not_hurt" survives as one unit where individually-scored words don't.
  • Modern subword transformers skip nearly this entire classical stack — but it still lives on inside sparse retrieval indexes and classical bag-of-words/TF-IDF feature pipelines.

Question 1 Multiple choice

A moderation pipeline needs to normalise a post that mixes precomposed accented characters, Cyrillic look-alike letters used to dodge a keyword filter, and a fullwidth Unicode variant of a Latin letter. Which normalisation form is the right choice, and why?

  1. NFD, because it decomposes everything into base characters plus combining marks
  2. NFC, because it always prefers the shortest possible representation
  3. NFKC, because it collapses both canonical and compatibility variants, including many homoglyph and fullwidth substitutions
  4. Lowercasing, because case folding also normalises alternate character forms
Show answer

Answer: C. NFKC (Compatibility Composition) collapses both canonically equivalent forms and compatibility variants — fullwidth letters, ligatures, and many stylistic substitutions — into one standard form, which is exactly the security-relevant behavior needed against homoglyph and compatibility-character evasion.

NFD decomposes rather than composes, which doesn't help match variants to a single form for lookup. NFC only handles canonical equivalence (like the two ways to write "é"), not compatibility variants like fullwidth letters. Lowercasing addresses case, not character-form substitution — a Cyrillic "а" lowercased is still a different codepoint from Latin "a".

Question 2 True / false

Lowercasing every post before feeding it to a moderation classifier is always the correct preprocessing choice, because case carries no useful signal.

  1. True
  2. False
Show answer

Answer: False. ALL-CAPS is a real signal of shouting, aggression, or heightened emotion, and case also disambiguates named entities from common words. Whether to lowercase depends on the downstream model and task — a sparse bag-of-words feature set benefits from folding, but an expressive subword/neural model can use case as a useful feature and loses information if it's folded away.

This is a common default reflex left over from classical NLP, where every fragmentation of the vocabulary was costly enough that folding almost always won. Modern moderation-relevant signals like shouting detection are exactly the case where that old default actively hurts.

Question 3 Fill in the blank

Replacing every URL in a post with a fixed <URL> token rather than keeping the raw link or deleting it entirely is preferred because it lets the model learn "this post contains a link" as a reusable,   feature, instead of either destroying that signal or bloating the vocabulary with near-unique strings.

Show answer

Answer: generalizable (reusable across posts). A placeholder token collapses every distinct URL into one shared symbol the model can learn to weight, whereas keeping raw URLs creates a huge number of effectively unique, one-off vocabulary entries, and dropping URLs erases the fact that a link was present at all.

A tempting wrong answer is "shorter," but token count reduction is a side effect, not the reason — the real payoff is that the model gets a signal it can generalize across every post containing any link, rather than memorizing individual URLs it will never see again.

Question 4 Multiple choice

Why does whitespace tokenization handle "fr€€ m0ney" (a leetspeak spam attempt) particularly badly compared to a subword tokenizer?

  1. Whitespace tokenization cannot process any string containing a currency symbol
  2. Whitespace tokenization treats "fr€€" and "m0ney" as whole, previously-unseen tokens likely to map to UNK, while a subword tokenizer can decompose them into known smaller pieces
  3. Whitespace tokenization always produces more tokens than subword tokenization on any input
  4. Whitespace tokenization requires a dictionary lookup that a subword tokenizer skips
Show answer

Answer: B. Since neither "fr€€" nor "m0ney" appeared during training as whole units, a word-level vocabulary has no entry for them and falls back to a generic UNK token, erasing the word entirely. A subword tokenizer can fall back to smaller known pieces — individual characters or short substrings — so the string is still represented, just with more tokens.

Whitespace tokenizers process currency symbols and digits fine as characters within a split token, that's not the failure mode. Whitespace tokenization typically produces fewer, not more, tokens than subword tokenization on any given input, since it splits less finely. And whitespace tokenization needs no dictionary at all, only spaces — the opposite of the stated distractor.

Question 5 Code completion

This function is meant to cap runs of a repeated character at two, so "soooo goooood" becomes "soo goood" without collapsing all the way to "so good" and losing the emphasis. Fill in the regex pattern and replacement.

import re

def de_elongate(text):
    return re.sub( ,  , text)
Show answer

Answer: re.sub(r"(.)\1{2,}", r"\1\1", text). The pattern (.)\1{2,} captures any character followed by at least two more repeats of itself (three or more total occurrences), and the replacement \1\1 writes back exactly two copies of that captured character, regardless of how many were originally there.

A tempting but wrong pattern is (.)\1+ replaced with \1, which collapses every repeated run down to a single character — that's the elongation-destroying version the lesson explicitly warns against, since it erases the "this was emphasized" signal instead of just normalising its length.

Question 6 True / false

Character-level (or byte-level) tokenization can never produce an out-of-vocabulary token, because its vocabulary already covers every possible input by construction.

  1. True
  2. False
Show answer

Answer: True. A character-level vocabulary is the full Unicode codepoint set (or, at the byte level, exactly the 256 possible byte values), so any input string, however novel or adversarial, decomposes into some sequence of already-known units — there is no such thing as an unrepresentable character or byte.

This is the one clean advantage character/byte tokenization has over both word-level and subword tokenization; the cost, covered in the lesson, is that this completeness comes with much longer token sequences and higher compute per unit of text, not that it fails to achieve full coverage.

Question 7 Bug fix

A junior engineer wrote this stopword-removal step for a threat-detection moderation model and can't figure out why it's producing false negatives on real threats. Identify the bug.

STOPWORDS = {"the", "a", "is", "to", "not", "and", "or", "be"}

def clean(text):
    tokens = text.lower().split()
    return [t for t in tokens if t not in STOPWORDS]

clean("i am not going to hurt you")
# -> ['i', 'am', 'going', 'hurt', 'you']
Show answer

Answer: The bug is including "not" in the stopword list at all for a negation-sensitive task. Removing it turns "not going to hurt you" into "going hurt you," which reads as an active threat rather than its explicit denial — the stopword filter has deleted the one word that decided the sentence's meaning.

The fix isn't a coding error in the loop or the split — the code runs exactly as written. The real fix is task-level: either skip stopword removal entirely for negation/intent-sensitive tasks like threat detection, or explicitly exclude negation words ("not," "no," "never") from any stopword list used upstream of such a model.

Question 8 Multiple choice

Why is the classical stemming/lemmatization/stopword-removal stack largely skipped when preparing input for a modern subword transformer, but still used in some retrieval and feature-engineering pipelines?

  1. Transformers cannot process any text that hasn't been stemmed first, so the stack is irrelevant, not skipped
  2. Subword tokenization already groups morphological variants into overlapping pieces the model learns to treat similarly, while classical exact-match retrieval and bag-of-words features still benefit from collapsing variants explicitly
  3. Stemming and lemmatization were both proven mathematically incorrect and are never used anywhere today
  4. Transformers are trained specifically to reverse stemming, making the step redundant
Show answer

Answer: B. A subword tokenizer implicitly captures morphological relationships by breaking related words into shared pieces, and attention lets the model weigh context without pre-decided stopword rules — so the classical stack's job is already partly done by the tokenizer and the model. Sparse retrieval indexes and classical feature pipelines (bag-of-words, TF-IDF) still rely on exact string matching, where explicit stemming/lemmatization/stopword removal genuinely reduces vocabulary fragmentation and improves match rates.

Transformers process raw (tokenized) text just fine without any classical normalisation — the claim that they require it is backwards. And the stack isn't "proven incorrect"; it is a working, still-used tool for a narrower category of systems than it used to serve, as the lesson's historical framing makes clear.
