Classical computer vision: edges, filters, features

Before a single neural net enters the picture, decades of hand-designed operators already solve most of what a production line needs: convolutional filters that clean up sensor noise, gradient operators that find edges, and descriptors that recognize a part despite rotation and lighting drift. Lesson 1

Convolution, filtering, and the frequency view

A linear filter is a small numerical grid, called a kernel, that slides across an image and replaces each pixel with a weighted sum of its neighborhood. Formally, for an image I and kernel K of size (2k+1)×(2k+1), the filtered output at position (x, y) is:

O(x,y) = Σ_i Σ_j I(x+i, y+j) · K(i,j), for i,j ∈ [-k, k]

That sum-of-products-over-a-sliding-window operation is what Course 1 Unit 6 called convolution when it was the operation a CNN learns the weights for. Here the weights are not learned — an engineer chooses them by hand, based on what frequency content or spatial pattern they want to keep or discard. Same arithmetic, opposite origin story: one side derives the kernel from gradient descent on millions of labeled images, the other derives it from signal-processing theory and a whiteboard.

Convolution versus correlation

Strictly, the formula above is cross-correlation — the kernel is applied in the same orientation it is written. True mathematical convolution flips the kernel 180° first: O(x,y) = Σ I(x-i, y-j)·K(i,j). For symmetric kernels like a Gaussian blur, flipping changes nothing, so the two terms get used interchangeably in vision code and papers — including "convolutional" neural networks, which technically perform correlation. The distinction matters only for asymmetric kernels (a directional edge detector, say), where flipping reverses the sign convention. On a production line this rarely bites you because most classical filters — box, Gaussian, Laplacian — are symmetric, but it is worth knowing which operation a library actually implements before you trust a sign.

Smoothing: box and Gaussian

The simplest smoothing kernel is the box filter: every entry in a k×k window is 1/k², so the output is a plain local average. It is cheap and it removes high-frequency noise, but it has an ugly property — a hard-edged window in the spatial domain corresponds to a ringing, oscillating response in the frequency domain (the sinc function), which can introduce faint false edges near real ones. On a solder-paste inspection image this shows up as a subtle halo around bright reflective blobs after blurring, which is exactly the kind of artifact that trips up a downstream edge detector.

The Gaussian filter fixes this. Its weights follow the bell curve G(x,y) = (1/2πσ²)·exp(-(x²+y²)/2σ²), so pixels near the center of the window count more than pixels near the edge, and the falloff is smooth with no hard cutoff — no ringing, no false edges. It is also the only 2-D filter that is simultaneously rotationally symmetric (no directional bias when a scratch could run at any angle) and separable (next section). There's a deeper reason Gaussian shows up everywhere in imaging: the point-spread function of a real lens, and the sum of many small independent noise sources on a sensor, both tend toward a Gaussian shape by the central limit theorem — so a Gaussian blur is a reasonable physical model of "what would this look like slightly out of focus or slightly noisier," not just a mathematically convenient choice.

Separability and the speed win

A 2-D Gaussian kernel can be written as the outer product of two 1-D Gaussians: G(x,y) = g(x)·g(y). That means you never actually need to build and slide a full k×k grid — you convolve each row with a 1-D kernel of length k, then convolve each column of the result with the same 1-D kernel. The cost per pixel drops from O(k²) multiply-adds to O(2k). For a 15×15 kernel that is 225 operations collapsing to 30 — a 7.5× speedup, and the gap only widens as kernel size grows. On an inspection line running at 40 frames per second with a hard latency budget before the reject gate, this is not an academic nicety; it's frequently the difference between a filter you can afford to run on every frame and one you can't. Box filters are separable too (and can be computed even faster with running sums via integral images), but Gaussian's separability combined with its lack of ringing is why it's the default smoothing choice in essentially every classical pipeline.

The frequency view: low-pass and high-pass

Every image can be decomposed into spatial frequencies — how fast intensity changes as you move across it. Flat, slowly-varying regions (the matte surface of a plastic housing) are low frequency; sharp transitions (a scratch edge, screen-printed text, component leads) are high frequency, and so is sensor noise. A smoothing kernel like the Gaussian is a low-pass filter: it attenuates high frequencies and passes low ones through, which is why it blurs edges along with removing noise — from a frequency standpoint, both edges and noise live in the same band, and a plain smoother cannot fully tell them apart. A high-pass filter does the reverse: it suppresses the slowly-varying background and emphasizes rapid transitions, which is exactly what makes edge detectors (Lesson 2) and sharpening kernels work. A common sharpening trick, unsharp masking, literally computes original + α·(original − blurred) — it isolates the high-frequency residual by subtracting a low-pass copy, then adds that residual back to boost edge contrast. You do not need to run a Fourier transform to use this framework day to day; it's enough to look at any kernel and ask whether its weights sum to preserve the average (low-pass, smoothing) or emphasize local differences (high-pass, sharpening or edge-finding).

Boundary handling

A kernel centered on a pixel near the image border needs neighbor values that don't exist. The common strategies are: zero-padding (treat missing pixels as black, which darkens filtered results near edges — a real problem when your part fills most of the frame and defects near its boundary are common, like edge chips on a PCB); replicate padding (repeat the border pixel outward, avoiding false darkening but potentially stretching a real edge feature); reflect padding (mirror the image across its border, which best preserves local statistics for smooth content); and crop (simply don't compute output where the full kernel doesn't fit, shrinking the output image by the kernel radius on each side). For fixed-camera QA rigs where the part is calibrated to sit in a known region of the frame with margin to spare, crop or replicate is usually safest — you don't want a boundary artifact masquerading as a defect signal three pixels in from an edge you don't even care about inspecting.

import numpy as np

def convolve2d(img, kernel):
    kh, kw = kernel.shape
    ph, pw = kh // 2, kw // 2
    padded = np.pad(img, ((ph, ph), (pw, pw)), mode="reflect")
    out = np.zeros_like(img, dtype=np.float64)
    for y in range(img.shape[0]):
        for x in range(img.shape[1]):
            region = padded[y:y+kh, x:x+kw]
            out[y, x] = np.sum(region * kernel)
    return out

# 5x5 Gaussian, sigma ~ 1.0 (rounded, unnormalized then normalized)
gaussian_5x5 = np.array([
    [1,  4,  7,  4, 1],
    [4, 16, 26, 16, 4],
    [7, 26, 41, 26, 7],
    [4, 16, 26, 16, 4],
    [1,  4,  7,  4, 1],
], dtype=np.float64)
gaussian_5x5 /= gaussian_5x5.sum()

# 3x3 sharpening kernel: identity plus a scaled Laplacian
sharpen_3x3 = np.array([
    [ 0, -1,  0],
    [-1,  5, -1],
    [ 0, -1,  0],
], dtype=np.float64)

solder_joint = np.load("solder_joint_gray.npy")   # (H, W) grayscale crop, 0-255
smoothed  = convolve2d(solder_joint, gaussian_5x5)
sharpened = convolve2d(solder_joint, sharpen_3x3)

The Gaussian kernel's weights peak at the center and taper outward, approximating the bell curve on a discrete grid; note it sums to 1 so average brightness is preserved. The sharpen kernel is the identity (a center weight of 5 keeps the original pixel dominant) minus its four-neighbor average, which is a discrete Laplacian — boosting the difference between a pixel and its surroundings is what makes solder joint edges pop before a human inspector reviews a borderline reject.

Key takeaways

  • Convolution and correlation differ only by a 180° kernel flip, and are identical for the symmetric kernels — box, Gaussian, Laplacian — that dominate classical filtering.
  • Gaussian beats box smoothing because it has no hard cutoff (no ringing/false edges), is rotationally symmetric, and — unlike most smoothing shapes — is separable.
  • Separability turns an O(k²) 2-D convolution into two O(k) 1-D passes, a real throughput win at line-rate frame rates.
  • Low-pass filters blur (remove high frequency = edges + noise together); high-pass filters sharpen or expose edges — a filter's behavior is readable straight off whether its weights average or difference neighboring pixels.
  • Boundary handling choice (zero, replicate, reflect, crop) changes pixel values near the frame edge and should be picked to avoid manufacturing false defect signals near a part's border.

Lesson 2

Edge and corner detection

In 1968, Irwin Sobel and Gary Feldman described a pair of small gradient kernels for estimating edge direction and magnitude — the Sobel operator is still the first thing most vision engineers reach for today, over half a century later. In 1986, John Canny formalized what an "optimal" edge detector should satisfy (good detection, good localization, one response per edge) and produced the multi-stage pipeline that bears his name and still ships, largely unchanged, in OpenCV. In 1988, Chris Harris and Mike Stephens extended the same gradient-based thinking from edges to corners, giving machines a way to find stable, trackable points rather than just boundaries. None of these three predates a single modern GPU. All three are still running, today, inside inspection cameras on real production lines — because for a fixed camera, fixed lighting, and a well-defined pass/fail criterion, they are fast, deterministic, and need zero labeled training data.

Image gradients: Sobel and Prewitt

An edge is, mathematically, a location where intensity changes rapidly — a large spatial derivative. The Sobel operator estimates the horizontal and vertical derivatives with two 3×3 kernels:

Gx = [[-1,0,1],[-2,0,2],[-1,0,1]] Gy = [[-1,-2,-1],[0,0,0],[1,2,1]]

Each kernel is a difference in one direction combined with a smoothing weight (the 1-2-1 pattern) in the perpendicular direction — Sobel is quietly doing a small Gaussian-like blur and a derivative in the same pass, which is why it's more noise-robust than the plain Prewitt operator, whose kernels use flat 1-1-1 weights instead of 1-2-1 and so give every row or column equal say with no central emphasis. Convolving an image with Gx and Gy gives two output images: how fast intensity changes left-to-right, and top-to-bottom, at every pixel.

Gradient magnitude and orientation

Combine the two directional derivatives into a single edge strength and a single edge direction per pixel:

magnitude = √(Gx² + Gy²) orientation = atan2(Gy, Gx)

Magnitude is what you'd threshold to get a binary edge map; orientation tells you which way the edge runs, perpendicular to the direction of steepest intensity change. On a scratch-detection task, orientation is often the more useful signal — a scratch running diagonally across a brushed-metal surface produces a gradient orientation histogram distinct from the surface's own grain texture, even where raw magnitude alone is ambiguous.

The full Canny pipeline

Sobel magnitude alone gives thick, noisy edge bands, not the crisp single-pixel-wide lines a QA system wants to count and measure. Canny's pipeline cleans this up in four stages. First, smooth with a Gaussian (Lesson 1) to suppress sensor noise before differentiating it — differentiation amplifies noise, so this step is not optional. Second, compute the gradient magnitude and orientation via Sobel. Third, apply non-maximum suppression: at each pixel, look along the gradient direction (rounded to one of 8 compass directions) at the two neighboring pixels, and keep the current pixel's magnitude only if it is a local maximum along that direction — otherwise zero it out. This thins a wide ridge of "edge-ish" pixels down to a single-pixel-wide line sitting exactly at the peak of the gradient. Fourth, hysteresis thresholding: pick a high threshold and a low threshold; any pixel above the high threshold is a confirmed edge, any pixel below the low threshold is discarded, and pixels in between are kept only if they connect (via 8-neighbor adjacency) to a confirmed edge. This lets a real edge with a locally weak segment — a shadow crossing a component lead, say — stay connected as one edge instead of fragmenting into disconnected dashes, while genuinely isolated weak responses (noise) get dropped.

Harris corners: what "corner" means mathematically

A corner is a location where the image gradient is strong in more than one direction at once — slide a small window over a flat region and nothing changes; slide it along a straight edge and nothing changes if you slide it along the edge's own direction; slide it over a corner and it changes no matter which way you slide it. The Harris detector formalizes this with the structure tensor, a 2×2 matrix built from local gradient products averaged over a window:

M = [[Σ Gx², Σ GxGy], [Σ GxGy, Σ Gy²]]

The two eigenvalues of M, λ1 and λ2, describe how much the windowed intensity changes along the two principal directions. Both small means a flat region; one large and one small means an edge (change in one direction only); both large means a corner. Computing eigenvalues directly is expensive, so Harris proposed a cheaper response score, R = det(M) − k·trace(M)², with k a small empirical constant (commonly 0.04–0.06) — R is large and positive at corners, negative at edges, and near zero on flat surfaces, without ever factoring the matrix. On a PCB inspection image, Harris corners cluster reliably at component leads, connector pin tips, and silkscreen text corners — exactly the stable, repeatable landmarks a later alignment step (Lesson 3) wants to lock onto.

Blob detection: the bridge to scale

Edges and corners are both essentially point- or line-local; they say nothing about the size of a feature. Blob detection asks a different question — where is there a roughly circular region of contrast at some particular scale — and answers it with the Laplacian of Gaussian (LoG): smooth with a Gaussian at scale σ, then apply the Laplacian (a second-derivative operator, ∂²/∂x² + ∂²/∂y²) to find locations where intensity curves sharply in a bowl or dome shape. Because computing LoG at every scale is costly, the practical approximation is the Difference of Gaussians (DoG): subtract two Gaussian-blurred copies of the image at nearby scales, σ and k·σ. A blob that matches the current scale gives the strongest DoG response, and searching across a stack of scales (a scale-space pyramid) locates both a blob's position and its characteristic size. This scale-space machinery — smoothing at increasing σ and differencing adjacent levels — is precisely what Lesson 3's SIFT keypoint detector reuses to find features that survive a change in how close the camera is to the part, or how much a defect has grown.

import numpy as np

def sobel_gradients(img):
    Gx = np.array([[-1,0,1],[-2,0,2],[-1,0,1]], dtype=np.float64)
    Gy = np.array([[-1,-2,-1],[0,0,0],[1,2,1]], dtype=np.float64)
    gx = convolve2d(img, Gx)   # convolve2d from Lesson 1
    gy = convolve2d(img, Gy)
    magnitude = np.sqrt(gx**2 + gy**2)
    orientation = np.arctan2(gy, gx)   # radians, range (-pi, pi]
    return magnitude, orientation

def non_max_suppression(mag, orient):
    H, W = mag.shape
    out = np.zeros_like(mag)
    angle = np.rad2deg(orient) % 180   # fold to 0-180 for 4 direction bins
    for y in range(1, H - 1):
        for x in range(1, W - 1):
            a = angle[y, x]
            # snap to nearest of 0, 45, 90, 135 degrees
            if a < 22.5 or a >= 157.5:
                n1, n2 = mag[y, x-1], mag[y, x+1]        # horizontal edge
            elif a < 67.5:
                n1, n2 = mag[y-1, x+1], mag[y+1, x-1]     # 45 degrees
            elif a < 112.5:
                n1, n2 = mag[y-1, x], mag[y+1, x]         # vertical edge
            else:
                n1, n2 = mag[y-1, x-1], mag[y+1, x+1]     # 135 degrees
            if mag[y, x] >= n1 and mag[y, x] >= n2:
                out[y, x] = mag[y, x]
    return out

lead_crop = np.load("component_lead_gray.npy")
mag, orient = sobel_gradients(lead_crop)
thinned = non_max_suppression(mag, orient)

Sobel gives one magnitude value and one orientation per pixel; non-max suppression then walks each pixel's orientation to the two neighbors that straddle it perpendicular to the edge (rounded to the nearest 45°) and zeroes the pixel unless it is the local peak — the thick gradient band around a component lead collapses to a single-pixel-wide contour ready for hysteresis thresholding.

Key takeaways

  • Sobel estimates directional derivatives with a built-in smoothing weight (1-2-1), making it more noise-robust than the flat-weighted Prewitt operator, at the same 3×3 cost.
  • Canny is a four-stage pipeline — smooth, gradient, non-max suppression, hysteresis threshold — not a single formula; skipping any stage degrades to raw thresholded Sobel with thick, noisy edges.
  • Hysteresis thresholding keeps weak-but-connected edge pixels and drops weak-and-isolated ones, preserving edges that dip below a single hard threshold in only part of their length.
  • A Harris corner is where both eigenvalues of the local structure tensor are large — gradient strong in every direction, not just one — approximated cheaply via R = det(M) − k·trace(M)².
  • Difference of Gaussians across a scale pyramid finds a blob's position and size together, the same scale-space idea SIFT reuses for scale-invariant keypoints in Lesson 3.

Lesson 3

Feature descriptors and the classical recognition pipeline

A reference PCB layout sits in your inspection database as a clean, correctly-populated image captured under ideal lighting. A camera on the line just captured a board that may be rotated two or three degrees in its fixture, lit slightly warmer than the reference shot, and possibly missing a capacitor. The obvious approach — subtract the two images pixel by pixel and flag large differences — fails immediately: a two-degree rotation shifts every edge in the image by several pixels, so the subtraction lights up almost the entire board as "different," burying the one real defect in thousands of false ones caused purely by misalignment. What you actually need is a way to find the same physical points on both images — a specific connector corner, a specific resistor's edge — regardless of the small rotation and lighting shift, align the two images using those matched points, and only then compare them. That's the job of a feature descriptor: a numerical fingerprint computed around a keypoint that stays recognizably similar even when the image around it has been rotated, rescaled, or relit.

What makes a good keypoint descriptor

Three properties matter most for QA-style matching. Illumination invariance: a descriptor should not care whether the LED ring light ran a touch brighter today, so it should be built from gradient directions and relative contrast rather than raw pixel brightness. Rotation invariance: the descriptor should be computed relative to a keypoint's own dominant orientation, not relative to the image's fixed x-axis, so a board rotated in its fixture still matches. Scale invariance: the descriptor should be computed at the scale at which the keypoint was actually detected (recall Lesson 2's DoG scale-space), so the same physical feature matches whether the camera is 20 cm or 22 cm from the board. A descriptor that nails all three is expensive to compute; one that only needs some of them (fixed-mount inspection rigs rarely need real scale invariance) can trade accuracy for speed — which is exactly the trade-off separating SIFT, HOG, and ORB below.

SIFT: the thorough, expensive baseline

SIFT (Scale-Invariant Feature Transform, Lowe 1999/2004) builds a scale-space pyramid of Difference-of-Gaussian images (Lesson 2), finds keypoints as local extrema across both space and scale, then discards low-contrast points and points sitting on an edge rather than a true corner-like blob (using a Harris-style eigenvalue-ratio test) to keep only stable, well-localized keypoints. Each surviving keypoint is assigned a dominant orientation from the local gradient histogram, so everything that follows is measured relative to that orientation, not the image frame — this is where rotation invariance comes from. The descriptor itself samples a 16×16 pixel neighborhood around the keypoint, splits it into a 4×4 grid of cells, and in each cell builds an 8-bin histogram of gradient orientations, weighted by magnitude — 16 cells × 8 bins gives the famous 128-dimensional descriptor vector. SIFT is thorough and historically the accuracy benchmark other descriptors were judged against, but it is comparatively slow, and its original patent (expired 2020) kept it out of default OpenCV builds for years — pushing the field toward faster or unencumbered alternatives even before deep learning arrived.

HOG: histograms as a shape template

HOG (Histogram of Oriented Gradients, Dalal & Triggs 2005) skips keypoint detection entirely and instead computes a dense grid of local gradient-orientation histograms across an entire fixed-size window — originally built for pedestrian detection, but its real strength is capturing overall shape and silhouette rather than fine texture, which makes it a natural fit for defect and component-shape templates in QA: a HOG descriptor computed over a "correctly seated connector" template can be compared against a candidate region to catch a connector that's tilted, only partially inserted, or the wrong part entirely, without needing pixel-level alignment. Because HOG describes a whole patch rather than a single interest point, it pairs naturally with a sliding-window or a classifier (historically an SVM) rather than a point-matching-and-RANSAC pipeline.

ORB: fast and free

ORB (Oriented FAST and Rotated BRIEF, Rublee et al. 2011) was built explicitly as an unencumbered, low-compute alternative to SIFT. It detects keypoints with FAST (checks a ring of 16 pixels around a candidate for a contiguous arc that's uniformly brighter or darker — cheap enough to run at video rate), assigns each keypoint an orientation from the intensity centroid of its patch, then builds a BRIEF descriptor: a string of a few hundred bits, each bit the result of comparing the brightness of one fixed pair of pixels near the keypoint against another. Because the descriptor is binary, matching two ORB descriptors is a Hamming distance (XOR and popcount) rather than a Euclidean distance over floats — orders of magnitude faster on commodity hardware, and exactly why ORB is the default choice when a QA rig needs feature matching at full frame rate on a CPU rather than a GPU.

Matching and RANSAC: geometric verification

Detecting descriptors on both images is only half the job — you then need to pair them up and reject bad pairs. The standard approach compares each reference descriptor against all candidates, keeps the closest match, and applies Lowe's ratio test: accept a match only if the best match is meaningfully closer than the second-best (typically below a 0.7–0.8 ratio), filtering out ambiguous matches in repetitive regions — a real risk on a PCB with dozens of visually identical resistors. Even after the ratio test, some matches will be wrong. RANSAC (Random Sample Consensus) cleans this up: repeatedly sample a minimal set of matches (3–4 point pairs), fit a geometric transform (an affine or perspective homography) from just that sample, then count how many remaining matches agree with it within a small pixel tolerance. The transform with the most agreeing "inliers" across many trials is kept, and only its inliers are trusted — outliers from repeated components or noise are discarded without individual inspection. The recovered transform is also the alignment: warp the captured board onto the reference frame, and a pixel-difference check finally becomes meaningful.

The honest bridge to learned features

SIFT, HOG, and ORB were the state of the art for a decade and a half of general-purpose vision, and modern learned features (Unit 3) have since overtaken them on open-world tasks — recognizing arbitrary objects under uncontrolled lighting and viewpoint is where hand-designed gradient histograms run out of expressive power. But controlled industrial QA is a different problem: fixed camera, fixed lighting, one part family, hard latency budget. There, classical descriptors keep three real advantages — single-digit-millisecond CPU runtime with no GPU required, zero labeled training data or retraining when a part revision ships, and a failed match that's fully explainable (you can point to exactly which keypoint pairs disagreed) in a way a CNN's activations are not. That's why ORB-based alignment checks still run alongside CNN-based defect classifiers on the same line — classical vision for the deterministic geometric part, learned features for the part too visually rich to hand-design a descriptor for.

import numpy as np

def hog_cell_histogram(magnitude, orientation, n_bins=9):
    """One cell's HOG histogram: gradient orientation binned 0-180deg,
    each vote weighted by that pixel's gradient magnitude."""
    bin_width = 180.0 / n_bins
    angle = np.rad2deg(orientation) % 180
    hist = np.zeros(n_bins)
    bins = (angle // bin_width).astype(int) % n_bins
    for b in range(n_bins):
        hist[b] = magnitude[bins == b].sum()
    return hist

def hog_descriptor(mag, orient, cell_size=8, n_bins=9):
    H, W = mag.shape
    cells_y, cells_x = H // cell_size, W // cell_size
    cell_hists = np.zeros((cells_y, cells_x, n_bins))
    for cy in range(cells_y):
        for cx in range(cells_x):
            ys, xs = cy*cell_size, cx*cell_size
            m = mag[ys:ys+cell_size, xs:xs+cell_size]
            o = orient[ys:ys+cell_size, xs:xs+cell_size]
            cell_hists[cy, cx] = hog_cell_histogram(m, o, n_bins)
    return cell_hists.flatten()   # length = cells_y * cells_x * n_bins

connector_patch = np.load("connector_template_gray.npy")   # e.g. 32x32
mag, orient = sobel_gradients(connector_patch)
descriptor = hog_descriptor(mag, orient, cell_size=8, n_bins=9)

A 32×32 patch with 8-pixel cells gives a 4×4 grid of cells, each contributing a 9-bin orientation histogram — a 144-length vector describing the connector's shape as "how much edge energy points in each direction, per region," which is far more tolerant of a few degrees of tilt or a slightly different exposure than comparing raw pixels would be.

Key takeaways

  • Pixel-difference comparison fails under any rotation or lighting shift because it has no notion of "same physical point" — feature matching solves exactly that correspondence problem.
  • A useful descriptor is illumination-, rotation-, and (often) scale-invariant by construction — built from local gradient orientation and relative contrast, anchored to the keypoint's own dominant direction and detection scale.
  • SIFT's 128-d descriptor (4×4 cells × 8 orientation bins) is thorough but costly; ORB trades float descriptors and Euclidean distance for binary strings and Hamming distance to run at video rate on a CPU.
  • HOG describes a whole patch's shape as a grid of gradient histograms, making it suited to template-style matching of component or defect silhouettes rather than sparse point matching.
  • RANSAC finds the geometric transform with the most agreeing matches, simultaneously rejecting bad correspondences and recovering the alignment needed before a meaningful pixel-level defect comparison.
  • Classical descriptors still win in controlled QA settings — no GPU, no training data, fully explainable failures — even though learned features have overtaken them for open-world recognition.

Question 1 Multiple choice

Why is a Gaussian kernel generally preferred over a box kernel for smoothing an inspection image before edge detection?

  1. The Gaussian kernel is not separable, so it captures diagonal structure the box filter misses.
  2. The Gaussian's smooth falloff avoids the ringing/false-edge artifacts a box filter's hard cutoff can introduce, while still being separable for speed.
  3. The box filter cannot be implemented as a convolution, only the Gaussian can.
  4. The Gaussian kernel always uses fewer multiply-adds than a box kernel of the same size.
Show answer

Answer: B. The Gaussian's bell-curve weighting has no hard spatial cutoff, so its frequency response lacks the ringing (false edges) a box filter's rectangular window produces, and it remains separable into two 1-D passes for the same speed benefit as a box filter.

A is wrong on two counts: the Gaussian is separable, and separability is precisely why it's fast, not a downside. D is wrong — a box and Gaussian kernel of the same size cost the same number of raw operations before separability is applied; the win is architectural (both are separable), not intrinsic to one shape over the other.

Question 2 True / false

Separating a 2-D Gaussian convolution into two 1-D passes (rows then columns) produces a mathematically different result than applying the full 2-D kernel directly.

  1. True
  2. False
Show answer

Answer: False. Because a 2-D Gaussian factors exactly as the outer product of two identical 1-D Gaussians, convolving with the two 1-D kernels in sequence produces the identical numerical result as the full 2-D convolution — separability is a computational shortcut, not an approximation.

It's tempting to assume any decomposition trades accuracy for speed, but separability here is an algebraic identity: G(x,y) = g(x)·g(y) means the sums simply factor, with no information lost.

Question 3 Fill in the blank

In the Canny edge detector, the stage that thins wide bands of high gradient magnitude down to single-pixel-wide edge lines by keeping only local maxima along the gradient direction is called  .

Show answer

Answer: non-maximum suppression. At each pixel, the algorithm checks the two neighbors that lie along the gradient orientation (rounded to the nearest 45°) and zeroes the pixel unless its magnitude is greater than or equal to both.

A common confusion is with hysteresis thresholding, which is a separate, later stage — hysteresis decides which already-thinned edge pixels to keep or discard based on high/low thresholds and connectivity, not which pixels to thin in the first place.

Question 4 Multiple choice

Hysteresis thresholding in Canny uses two thresholds instead of one primarily to:

  1. Speed up the computation by skipping the gradient magnitude step for weak pixels.
  2. Detect corners in addition to edges within the same pass.
  3. Keep weak-but-connected edge segments that dip below a single hard cutoff in places, while still discarding isolated weak responses like noise.
  4. Replace the need for a prior Gaussian smoothing step.
Show answer

Answer: C. A real edge can have a locally weak segment — a shadow crossing it, for instance — that would be lost under one hard threshold. By keeping any weak pixel that connects to a confirmed strong edge, hysteresis preserves that continuity while still dropping isolated weak-only responses, which are more likely noise.

D is a common misread: hysteresis thresholding is entirely separate from, and comes after, the Gaussian smoothing stage — smoothing still has to happen first to keep differentiation from amplifying sensor noise.

Question 5 Multiple choice

The Harris corner response R = det(M) − k·trace(M)² is large and positive at a corner because:

  1. Both eigenvalues of the local structure tensor M are large, meaning the windowed intensity changes significantly in every direction, not just one.
  2. The gradient magnitude is zero in a flat region, which maximizes det(M).
  3. Corners always have a higher raw pixel brightness than edges or flat regions.
  4. The trace of M is always negative at corners, driving R positive.
Show answer

Answer: A. M's two eigenvalues describe how much windowed intensity varies along the two principal directions; a corner is exactly the case where both are large, giving a large determinant relative to the squared trace, so R is large and positive.

Question 6 Bug fix

A teammate's non-max suppression only ever checks the pixel's left and right neighbors, regardless of gradient orientation:

def non_max_suppression_buggy(mag, orient):
    H, W = mag.shape
    out = np.zeros_like(mag)
    for y in range(1, H - 1):
        for x in range(1, W - 1):
            n1, n2 = mag[y, x-1], mag[y, x+1]
            if mag[y, x] >= n1 and mag[y, x] >= n2:
                out[y, x] = mag[y, x]
    return out

What is the concrete effect of this bug on inspection images?

  1. It has no visible effect — Canny edges look the same regardless of orientation handling.
  2. It only thins horizontal edges correctly; vertical and diagonal edges (e.g. vertical scratches, angled solder bridges) stay several pixels wide because the wrong neighbor pixels are compared.
  3. It causes every pixel in the image to be suppressed to zero.
  4. It makes the algorithm run non-deterministically.
Show answer

Answer: B. Non-max suppression must compare each pixel against the two neighbors that straddle it perpendicular to the local gradient orientation. Always checking left/right only correctly thins edges whose gradient runs horizontally; a vertical scratch's gradient points left-right, so its edge, running top-to-bottom, needs the up/down neighbors compared instead — the buggy version leaves those edges wide and unthinned.

A is the tempting-but-wrong answer for anyone who's only tested the code on a single horizontal test edge; the failure only becomes visible once you inspect vertical or diagonal defects, which is most of what a real production line produces.

Question 7 Multiple choice

Why is ORB typically chosen over SIFT for a fixed-mount PCB inspection rig that needs to align a captured board to a reference layout at full camera frame rate on a CPU?

  1. ORB is more scale-invariant than SIFT, which matters more than speed for a fixed camera distance.
  2. ORB's binary descriptor and Hamming-distance matching are far cheaper to compute and compare than SIFT's float descriptors and Euclidean distance, at a small cost in raw matching accuracy.
  3. SIFT cannot be computed on grayscale images, only ORB can.
  4. ORB does not require any keypoint detection step, unlike SIFT.
Show answer

Answer: B. ORB's BRIEF-based binary descriptors compare against each other with XOR and a bit count instead of a full floating-point Euclidean distance, and are built from cheap FAST keypoints — a large speed advantage that matters when a rig has no GPU and a hard frame-rate budget, at some cost in the descriptor's discriminative power versus SIFT's richer 128-d histogram.

A is backwards for the scenario described — a fixed camera distance means scale invariance is one of the least important properties here, which is exactly why trading it away for speed is a reasonable choice. D is also wrong: ORB still detects keypoints, using FAST, it just replaces SIFT's DoG-based detector and float descriptor with cheaper alternatives.

Question 8 True / false

RANSAC works by computing a geometric transform from every possible pair of matched keypoints and averaging the results together.

  1. True
  2. False
Show answer

Answer: False. RANSAC repeatedly samples a small minimal subset of matches, fits a candidate transform from just that subset, and counts how many of the other matches agree with it within a tolerance ("inliers"); the transform with the most inliers across many random trials is kept, and only its inlier matches are trusted going forward.

Averaging over every match would be defeated by exactly the bad correspondences RANSAC exists to reject — a handful of wrong matches (common on a PCB full of visually identical resistors) would skew any transform computed from all matches at once, which is the whole reason the random-subsample-and-vote approach exists instead.
