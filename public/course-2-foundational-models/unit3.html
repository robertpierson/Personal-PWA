Tokenization and embeddings

Before any attention block in Unit 2 can run, raw text has to become a sequence of integers, and those integers have to become vectors. This unit covers how that split happens and what it costs — using a multilingual product catalogue as the proving ground.

Lesson 1

Why splitting text is harder than it looks

Feed a whitespace tokenizer the German catalogue string "Waschmaschinenzubehör" and you get exactly one token — a single opaque string meaning "washing machine accessories" that the model has almost certainly never seen before, since German compounds are formed on the fly and a corpus can contain thousands of distinct ones. Feed the same tokenizer the Japanese title "洗濯機用ステンレスホース" (stainless steel hose for washing machines) and you get one token too, because Japanese doesn't put spaces between words at all. Feed it "US 10.5 / EU 44" next to "US 10,5 / EU 44" — the German decimal-comma convention for the same shoe size — and whitespace splitting treats 10.5 and 10,5 as two unrelated strings with nothing in common. Three catalogue entries, three different failures, and none of them are edge cases — they're Tuesday for a global storefront.

A tokenizer's job is to turn a string into a sequence of discrete units a model can assign an embedding to. Where you draw the boundaries between units is a design decision with real consequences at every point along a spectrum, and understanding that spectrum is the foundation for everything else in this unit.

The character / word / subword spectrum

At one extreme, a character-level tokenizer splits every string into individual characters or bytes. "Waschmaschinenzubehör" becomes 22 tokens, one per character. The vocabulary is tiny — a few hundred entries can cover every character in Latin, Cyrillic, Arabic, and the common Japanese kana, though full coverage of Han characters used in Japanese and Chinese pushes that into the tens of thousands. The catch is sequence length: every string you tokenize gets long, and since a transformer's self-attention cost scales quadratically with sequence length (Unit 2), doubling token count roughly quadruples compute for a given batch. A model that reads character-by-character also has to learn word structure itself from scratch, spending capacity re-deriving "this cluster of six characters is a unit" over and over.

At the other extreme, a word-level tokenizer splits on whitespace and punctuation and gives every distinct word its own token. This keeps sequences short — one token per word — but the vocabulary explodes. English alone has hundreds of thousands of word forms once you count plurals, verb conjugations, and proper nouns; add German's productive compounding (any noun can combine with any other noun to form a new one, so the vocabulary is in principle unbounded) and Turkish's agglutination (a single verb stem can take a long chain of suffixes — "gönderemeyeceğim", "I will not be able to send [it]", is one word built from a stem plus five suffixes) and no fixed-size vocabulary can ever contain every word a catalogue will produce. Whatever isn't in the vocabulary becomes an out-of-vocabulary (OOV) token, collapsed to a single <UNK> placeholder. Every product title with an unseen compound, an unseen SKU, or an unseen brand name loses information the moment it hits <UNK> — a model can't distinguish "Waschmaschinenzubehör" from "Geschirrspülerzubehör" (dishwasher accessories) if both became the same unknown token. Rare tokens the model has seen only a handful of times during training are barely better: their embeddings are undertrained and carry almost no useful signal.

Subword tokenization sits between the two and is what every modern production model actually uses. Common whole words like "machine" or "Waschmaschine" stay as single tokens; rare or novel strings get broken into smaller frequent pieces — "Zubehör" might split into "Zube" + "hör", or a SKU like "WM-4471X" into "WM" + "-" + "44" + "71" + "X". This bounds the vocabulary to a fixed size (typically 30k–130k entries) while keeping the OOV rate near zero, because any string can always be decomposed down to individual characters or bytes as a fallback. The price is that sequences get longer than word-level tokenization and a single semantic unit sometimes gets fragmented in linguistically arbitrary ways — but that price is far smaller than either extreme's failure mode.

Normalization and why lowercasing is dangerous here

Before splitting happens, most tokenizer pipelines run a normalization step — a canonicalization pass so that visually or semantically identical strings map to the same sequence of characters. NFKC (Normalization Form Compatibility Composition) is the standard Unicode normalization: it collapses characters that can be encoded multiple ways — a "ü" that arrived as one composed codepoint versus a "u" followed by a combining diaeresis mark — into one canonical form, and maps compatibility variants like full-width Japanese digits (１２３) to their ordinary ASCII equivalents (123). Skip this step and a tokenizer trained mostly on one encoding of "ü" silently treats the other encoding as a different, rare token, quietly inflating the OOV rate on a catalogue pulled from multiple upstream systems.

Casefolding — lowercasing everything — looks harmless and is a common step for general text, but it is actively destructive on a product catalogue. "US" (United States sizing) and "us" mean different things only by capitalization convention in some contexts; more concretely, SKU codes and model numbers routinely encode meaning in case: "WM-4471X" versus "wm-4471x" may be treated by an inventory system as the same part, but a tokenizer that's already lowercased everything has thrown away the one signal — the deliberate capitalization convention manufacturers use — that let it recognize "this is a SKU-shaped string" as a pattern distinct from ordinary prose. Turkish makes this worse in a way that's a well-known trap: the Turkish alphabet has a dotless ı and a dotted i as two distinct letters, and naive lowercasing logic built for English (which assumes I → i) turns Turkish İ into the wrong character entirely. Production tokenizers for multilingual text generally skip casefolding altogether and let the subword vocabulary learn case-sensitive pieces instead.

Pre-tokenization: splitting before you split

Pre-tokenization is a coarse first cut — usually on whitespace and punctuation — that runs before the learned subword algorithm (Lesson 2) ever sees the text, establishing boundaries the subword merges are not allowed to cross. For English and German this is cheap and mostly reliable: split on spaces, keep punctuation as separate tokens. For Japanese and Chinese, which have no whitespace between words, naive pre-tokenization on spaces does almost nothing useful — the entire sentence "洗濯機用ステンレスホース" arrives as one pre-token, and the subword algorithm has to do all the work of finding word-like boundaries statistically. This is precisely the gap that SentencePiece was built to close, which Lesson 2 covers directly. Arabic adds a different wrinkle: it's written right-to-left, but digits and Latin-script SKU fragments embedded in an Arabic title (a common pattern — "مقاس 44 EU") run left-to-right, so the raw byte stream a tokenizer sees interleaves two writing directions, and pre-tokenization has to avoid corrupting either.

The following comparison tokenizes five real catalogue titles three ways — whitespace, naive character-level, and a plausible subword result — and reports token counts and OOV behavior for each.

titles = [
    "Waschmaschinenzubehör Edelstahlschlauch 1.5m",      # German compound
    "洗濯機用ステンレスホース",                              # Japanese, no spaces
    "Kadın spor ayakkabısı gönderemeyeceğim",             # Turkish agglutination
    "حذاء رياضي مقاس 44 EU / US 10.5",                     # Arabic + embedded Latin/digits
    "SKU WM-4471X size US 10.5 / EU 44",                  # SKU + locale-variant sizing
]

def whitespace_tokenize(s):
    return s.split(" ")

def char_tokenize(s):
    return list(s)

# vocab a real subword tokenizer would already know from training
# on a large multilingual catalogue corpus
KNOWN_SUBWORDS = {
    "Wasch", "maschinen", "zubehör", "Edelstahl", "schlauch", "1.5", "m",
    "洗濯機", "用", "ステンレス", "ホース",
    "Kadın", "spor", "ayakkabı", "sı", "gönder", "emeyeceğim",
    "حذاء", "رياضي", "مقاس", "44", "EU", "/", "US", "10.5",
    "SKU", "WM", "-", "4471", "X", "size",
}

def toy_subword_tokenize(s):
    # greedy longest-match-first against the known vocab; falls back
    # to per-character pieces (never UNK) for anything unseen
    tokens, i = [], 0
    while i < len(s):
        if s[i] == " ":
            i += 1
            continue
        match = None
        for end in range(len(s), i, -1):
            if s[i:end] in KNOWN_SUBWORDS:
                match = s[i:end]
                break
        tokens.append(match if match else s[i])
        i += len(match) if match else 1
    return tokens

for title in titles:
    ws  = whitespace_tokenize(title)
    ch  = char_tokenize(title)
    sub = toy_subword_tokenize(title)
    ws_oov = sum(1 for t in ws if t not in KNOWN_SUBWORDS)
    print(f"{title[:28]:<30} whitespace={len(ws):>3} (oov={ws_oov})  "
          f"char={len(ch):>3} (oov=0)  subword={len(sub):>3} (oov=0)")

Whitespace tokenization produces the fewest tokens but the highest OOV count — the Japanese and Turkish titles are nearly all-OOV under whitespace splitting because whitespace rarely coincides with a real word boundary in those languages. Character tokenization has zero OOV by construction but the longest sequences. The toy subword pass lands in between and, by falling back to individual characters instead of a single UNK, never loses information — only length efficiency.

Key takeaways

  • The character/word/subword spectrum is a three-way trade between vocabulary size, sequence length, and out-of-vocabulary rate — you cannot minimize all three at once.
  • Word-level tokenizers fail on morphologically rich languages — German compounding and Turkish agglutination generate unbounded vocabularies, and whitespace assumptions fail outright on Japanese and Chinese.
  • The UNK token is a silent information sink — collapsing distinct rare strings to one placeholder makes them indistinguishable to the model, which is why production tokenizers are built to avoid ever emitting it.
  • Normalization (NFKC) fixes encoding inconsistency but casefolding actively destroys meaning in SKUs, model numbers, and Turkish's dotted/dotless i distinction — multilingual catalogue tokenizers generally skip lowercasing.
  • Pre-tokenization draws boundaries the subword algorithm can't cross — and whitespace-based pre-tokenization is close to useless for Japanese, which is why SentencePiece (Lesson 2) works on the raw stream instead.

Lesson 2

BPE, WordPiece, Unigram, SentencePiece

Byte-Pair Encoding did not start as a machine learning algorithm at all. Philip Gage published it in 1994 as a general-purpose data compression trick: scan a byte stream, find the most frequent adjacent pair of bytes, replace every occurrence with a new byte that stands for that pair, and repeat until you run out of spare byte values or the data stops compressing. It sat there as a compression footnote for two decades. Then in 2016, Rico Sennrich, Barry Haddow, and Alexandra Birch repurposed it for an entirely different problem: neural machine translation models at the time had fixed output vocabularies and, like the word-level tokenizer in Lesson 1, fell over on rare and compound words — exactly the German-compound and morphology problems this unit's catalogue keeps surfacing. Their insight was that Gage's byte-merging loop, run over characters instead of bytes, naturally produces a vocabulary of frequent subword chunks: common full words survive as single units, rare words decompose into smaller frequent pieces, and nothing is ever fully unrepresentable. That 22-year-old compression algorithm is the direct ancestor of the tokenizer in GPT-2, GPT-3, and every open model that lists "BPE" in its tokenizer config today.

The BPE training loop, with real counts

BPE training works over a corpus of words, each represented as a sequence of characters with a special end-of-word marker (here, _) so the algorithm never merges across a word boundary. Take a tiny corpus of six catalogue title fragments with their frequencies:

WordCountStarting split
schuh_ (shoe)5s c h u h _
schuhe_ (shoes)3s c h u h e _
schrank_ (cabinet)4s c h r a n k _
schrauben_ (screws)2s c h r a u b e n _

Step one: count every adjacent character pair across the whole corpus, weighted by word frequency. The pair (s, c) appears in every word (5+3+4+2 = 14 occurrences), and so does (c, h) — also 14. Both are tied for most frequent; a real implementation breaks ties by encounter order, so say (s, c) merges first, becoming a new symbol sc. Recount: now (sc, h) appears 14 times — the most frequent pair remaining — so it merges to sch. Recount again: (sch, u) appears in schuh_ and schuhe_, 5+3 = 8 times, while (sch, r) appears in schrank_ and schrauben_, 4+2 = 6 times. (sch, u) wins, merging to schu. Three merges in, the vocabulary already contains a piece that means "shoe-ish" and the algorithm hasn't needed a single hand-written rule — it discovered that sch is common German-catalogue structure purely from frequency counts, and it will keep merging (next likely: schu+h → schuh) until it hits the target vocabulary size, at which point training stops and the merge list is frozen for use at inference time.

WordPiece: likelihood instead of raw frequency

Google's WordPiece, used in BERT, runs the same merge-and-repeat loop as BPE but changes the criterion for which pair to merge. Instead of picking the most frequent adjacent pair, WordPiece picks the pair whose merge most increases the likelihood of the training corpus under a unigram language model — roughly, it scores each candidate merge by count(pair) / (count(first) × count(second)), favoring pairs that co-occur far more often than their individual frequencies would predict by chance. This tends to prefer merging pieces that are more "glued together" linguistically over pieces that are merely both common. The practical difference from raw-frequency BPE is usually small, but it's the reason BERT-family tokenizers use a ## prefix convention (##ing, ##zubehör) to mark that a piece continues a word rather than starting one — WordPiece's scoring, unlike BPE's, was designed around that continuation distinction from the start.

Unigram: pruning down instead of merging up

The Unigram algorithm (Kudo, 2018) inverts the direction entirely. Instead of starting from characters and merging upward, it starts from a large candidate vocabulary — every substring that appears often enough in the corpus, potentially hundreds of thousands of candidates — and repeatedly removes the pieces whose loss would hurt the corpus's overall likelihood the least, under a unigram model where each token's probability is estimated independently. It stops once it's pruned down to the target vocabulary size. The payoff is that Unigram naturally supports probabilistic segmentation: because it models a token's probability independent of context, the same string can be segmented multiple valid ways ("Zubehör" as one piece, or as "Zube" + "hör") with a computable likelihood for each, and a Unigram tokenizer can sample among plausible segmentations rather than always returning one deterministic split — a form of built-in data augmentation that BPE and WordPiece, which produce one fixed segmentation per string, don't offer.

SentencePiece: no pre-tokenization required

SentencePiece (Kudo & Richardson, 2018) is not a fourth merge algorithm — it's an implementation of BPE or Unigram (your choice) that changes what the algorithm is allowed to see. Recall from Lesson 1 that ordinary BPE/WordPiece pipelines run pre-tokenization first (split on whitespace), then apply subword merges within each whitespace-delimited piece. SentencePiece skips that step: it treats the input as a raw stream of Unicode characters (or, in byte-level mode, raw bytes) including the whitespace itself, which it encodes as a literal token, conventionally shown as ▁. This is exactly the fix Lesson 1 flagged as missing for Japanese — since SentencePiece never assumes whitespace marks a word boundary, it works identically on English, German, and whitespace-free Japanese without any language-specific pre-processing rule, and it can reconstruct the original string byte-for-byte from its tokens (including exact spacing), which plain BPE cannot always guarantee once punctuation-splitting rules have discarded information. This is why SentencePiece, not classic BPE, is the tokenizer underneath multilingual models like mT5 and the XLM family.

Byte-level BPE: the version that structurally cannot emit UNK

Byte-level BPE, used by GPT-2 and its successors, runs the BPE merge loop over raw UTF-8 bytes (256 possible values) instead of Unicode characters. Since every possible string — in any language, any emoji, any malformed encoding — decomposes into some sequence of the 256 byte values, and all 256 byte values are guaranteed to be in the base vocabulary before any merges happen, there is no string byte-level BPE cannot represent. It literally cannot need an <UNK> token, because unlike character-level BPE (which can still be handed a Unicode codepoint it has never indexed), the byte alphabet is closed and complete by construction. A catalogue emoji like 👟 (running shoe) or 🇹🇷 (Turkey flag, itself a multi-codepoint sequence) simply decomposes into its UTF-8 bytes and gets merged like anything else — worst case, it costs 3–4 tokens instead of one, but it is never lost.

from collections import Counter

# tiny catalogue corpus: word -> frequency, pre-split into characters + end marker
corpus = {
    tuple("schuh_"):    5,
    tuple("schuhe_"):   3,
    tuple("schrank_"):  4,
    tuple("schrauben_"):2,
}

def get_pair_counts(corpus):
    counts = Counter()
    for word, freq in corpus.items():
        for a, b in zip(word, word[1:]):
            counts[(a, b)] += freq
    return counts

def merge_pair(pair, corpus):
    a, b = pair
    merged = a + b
    new_corpus = {}
    for word, freq in corpus.items():
        new_word, i = [], 0
        while i < len(word):
            if i < len(word) - 1 and word[i] == a and word[i+1] == b:
                new_word.append(merged)
                i += 2
            else:
                new_word.append(word[i])
                i += 1
        new_corpus[tuple(new_word)] = freq
    return new_corpus

learned_merges = []
for step in range(3):
    pair_counts = get_pair_counts(corpus)
    best_pair = max(pair_counts, key=pair_counts.get)
    learned_merges.append((best_pair, pair_counts[best_pair]))
    corpus = merge_pair(best_pair, corpus)

for i, (pair, count) in enumerate(learned_merges, 1):
    print(f"merge {i}: {pair[0]!r} + {pair[1]!r}  (seen together {count}x)")

This reproduces the three merges walked through above: running it prints ('s','c'), then ('sc','h'), then ('sch','u') — each with its actual weighted count — confirming that BPE training is nothing more than "count pairs, merge the winner, repeat" applied mechanically.

Key takeaways

  • BPE is a 1994 compression algorithm, not an ML invention — Sennrich et al. (2016) repurposed byte-pair merging to fix rare/compound-word handling in neural machine translation.
  • BPE and WordPiece both merge upward from characters but score candidate merges differently — raw co-occurrence frequency for BPE, a likelihood-improvement ratio for WordPiece.
  • Unigram prunes downward from a huge candidate vocabulary instead of merging upward, and uniquely supports probabilistic multi-segmentation of the same string.
  • SentencePiece removes the whitespace pre-tokenization assumption entirely, treating text as a raw character/byte stream — the direct fix for Japanese's lack of whitespace.
  • Byte-level BPE has a closed, complete base vocabulary (256 byte values), so it is structurally incapable of producing an UNK token, unlike character-level schemes.

Lesson 3

Embeddings: from token id to vector

An embedding matrix is a single learned tensor of shape (vocab_size, d_model) — one row per token id, each row a dense vector of length d_model. Tokenization (Lessons 1–2) ends with an integer sequence, like [4471, 892, 15] for a tokenized SKU fragment; the embedding matrix is the lookup table that turns each integer into the vector the rest of the model — the attention blocks from Unit 2 — actually operates on. Everything downstream of tokenization is vectors, and this matrix is where integers become vectors.

Lookup as one-hot matmul — but never actually done that way

Mathematically, looking up row i of an embedding matrix E is identical to constructing a one-hot vector v (all zeros except a 1 at position i) and computing v @ E — the matmul zeroes out every row except row i and returns it. This equivalence is worth knowing because it's how embeddings are sometimes explained on a whiteboard and because it clarifies that an embedding table is mathematically just a special case of a linear layer with a one-hot input. But no real implementation does it this way: multiplying a sparse one-hot vector against a matrix with (say) 128,000 rows to extract one 4096-length row wastes essentially all of the multiply-adds computing zeros. Every real framework instead implements embedding lookup as direct indexed memory access — E[token_id] — which is why the operation is called nn.Embedding in PyTorch rather than expressed as a matmul: same math, but a gather instead of a multiply, because gathers over a 128k-row table are vastly cheaper than a 128k-wide matmul per token.

The parameter cost hides in plain sight

The embedding matrix's size is vocab_size × d_model, and for models with large multilingual vocabularies this is often the single biggest tensor anywhere in the network — bigger than most individual attention or feed-forward weight matrices. A 128,000-token vocabulary (roughly what a well-built multilingual SentencePiece vocabulary needs to give reasonable per-language coverage across English, German, Japanese, Arabic, and Turkish without excessive fragmentation) at d_model = 4096 is:

128,000 × 4,096 = 524,288,000 ≈ 512M parameters

Half a billion parameters spent purely on the input lookup table, before a single attention or feed-forward layer runs. For a smaller model — say a 1–2B parameter total budget — that's a substantial fraction of the entire parameter count going toward vocabulary rather than reasoning capacity, which is exactly why vocabulary size is a real design trade-off and not a free lunch: bigger vocabulary buys shorter sequences and lower OOV (Lesson 1) at the direct cost of parameters that scale linearly with every token you add.

Weight tying: reusing the same matrix twice

A language model needs an embedding matrix to turn input token ids into vectors, and it separately needs an output projection to turn the final hidden state back into a probability distribution over the vocabulary — a matrix of shape (d_model, vocab_size), the transpose shape of the input embedding. Weight tying is the decision to make these the literal same matrix (transposed for the output side) rather than two independently learned ones. This halves the parameter cost of the two largest tensors in a small model — instead of 512M for input embeddings plus another 512M for the output head, tying gives you 512M total, reused in both directions. It also has a regularizing effect: because the same row of E is pulled toward being a good input representation for a token and a good output-scoring direction for predicting that same token, the two objectives constrain each other, generally helping smaller models. The cost is a loss of flexibility — the input and output roles are subtly different (input embeddings benefit from clustering similar tokens close together; output projections benefit from being able to sharply separate a token from its near-neighbors to make confident predictions), and very large models sometimes untie the two matrices once the parameter cost is proportionally small enough not to matter.

The multilingual token tax

Tokenizers are trained on a mix of languages, and even a well-balanced training mix tends to under-represent morphologically complex or non-Latin-script languages relative to English, because BPE/Unigram merge frequency is driven by what's common in the training corpus — and English word fragments dominate most web-scale corpora. The practical result, well documented across production multilingual tokenizers: the same sentence, translated, costs measurably more tokens in some languages than others. A German sentence with the same meaning as an English one might run 1.4x the token count, a byproduct of compounding meaning fewer whole words survive as single tokens. Japanese, with no whitespace and a large, sparser-per-corpus character set, is worse — commonly cited multilingual tokenizer measurements put it around 2.5x the English token count for equivalent meaning. This "token tax" is not cosmetic: every downstream cost that scales with sequence length scales with it too. A model with a fixed context window of, say, 8,000 tokens gives an English-language user roughly 8,000 tokens of catalogue history and conversation, but gives a Japanese-language user the equivalent of only about 3,200 English-token-equivalents of content before running out of room. Per-request API pricing that bills per token charges the Japanese-language user 2.5x more for saying the same thing. Nothing about the model's competence differs — the tax is entirely a property of the tokenizer's vocabulary allocation, which is why tokenizer design is itself a fairness and cost consideration for a genuinely multilingual product, not just an engineering detail.

import numpy as np

def build_tied_embedding(vocab_size, d_model, seed=0):
    rng = np.random.default_rng(seed)
    # single matrix, shared by input lookup and output projection
    E = rng.normal(scale=0.02, size=(vocab_size, d_model))
    return E

def embed_lookup(E, token_ids):
    # real implementation: indexed gather, not a one-hot matmul
    return E[token_ids]

def output_logits(E, hidden_state):
    # tied output head: reuse E transposed instead of a second matrix
    return hidden_state @ E.T   # (d_model,) @ (d_model, vocab) -> (vocab,)

for vocab_size, d_model in [(32_000, 1024), (128_000, 4096)]:
    E = build_tied_embedding(vocab_size, d_model)
    tied_params = E.size
    untied_params = 2 * E.size
    print(f"vocab={vocab_size:>7} d_model={d_model:>5} "
          f"tied={tied_params:,} untied_equivalent={untied_params:,}")

# trace one lookup + logits pass on a toy tokenized SKU fragment
E = build_tied_embedding(vocab_size=32_000, d_model=1024)
token_ids = np.array([4471, 892, 15])          # e.g. "WM", "-", "44"
vectors = embed_lookup(E, token_ids)
print("embedded shape:", vectors.shape)         # (3, 1024)
logits = output_logits(E, vectors[-1])
print("logits shape:", logits.shape)            # (32000,) — one score per vocab entry

The printed parameter counts show the small-vocab config (32k × 1024) costing about 33M tied parameters versus 66M untied, and the large multilingual config (128k × 4096) costing 512M tied versus 1.05B untied — the tying decision alone is worth hundreds of millions of parameters at this scale. The lookup returns shape (3, 1024) for three token ids, and the tied output projection turns one 1024-length hidden vector into 32000 logits using the exact same matrix the lookup used, just transposed.

Key takeaways

  • The embedding matrix has shape (vocab_size, d_model) and is looked up by index — mathematically equivalent to a one-hot matmul, but always implemented as a direct gather for efficiency.
  • Embedding parameter cost scales linearly with vocabulary size — a 128k-vocab, d_model-4096 matrix is roughly 512M parameters, often the largest single tensor in a smaller model.
  • Weight tying reuses the input embedding matrix as the output projection, halving the cost of the two largest tensors and adding a mild regularizing effect, at some loss of representational flexibility.
  • The same content costs different numbers of tokens in different languages — roughly 1.4x for German and 2.5x for Japanese relative to English in common multilingual tokenizers — a direct consequence of training-corpus imbalance in the merge/prune process from Lesson 2.
  • The multilingual token tax has real downstream costs: less usable context window and higher per-request price for languages the tokenizer represents less efficiently.

Question 1 Multiple choice

A whitespace tokenizer is run over the Japanese title "洗濯機用ステンレスホース" (no spaces in the original). What is the most likely outcome?

  1. The title splits cleanly into individual words.
  2. The entire title becomes a single token, since there is no whitespace to split on.

  4. The title is automatically transliterated to Latin script before splitting.
Show answer

Answer: B. Whitespace tokenization can only split where whitespace exists. Japanese doesn't use whitespace between words, so the whole string is treated as one indivisible token — exactly the failure mode Lesson 1 opens with.

Option A is the tempting distractor because it assumes the tokenizer has some awareness of word boundaries — it doesn't; whitespace splitting has no model of language, only of the space character. Options C and D describe behaviors no whitespace tokenizer implements.

Answer: False. Lowercasing destroys information in this domain: SKU and model-number case conventions carry meaning, and Turkish has a dotted i / dotless ı distinction that English-centric lowercasing logic (which assumes I → i) gets wrong.

Unigram tokenization (Kudo, 2018) builds its final vocabulary by:

Answer: B. Unigram inverts the direction of BPE/WordPiece: it starts large (a huge set of candidate substrings) and repeatedly removes the pieces whose loss hurts corpus likelihood the least, stopping at the target vocabulary size.

Option A describes BPE/WordPiece's merge-upward approach, the opposite direction from Unigram. Option C inverts which words get treated specially, and no widely used subword scheme covered in this unit trains a neural network to choose split points directly.

Question 6 True / false

Byte-level BPE (as used in GPT-2) can still produce an UNK token when it encounters a character it has never seen during training, such as an emoji.

  1. True
  2. False
Show answer

Answer: False. Byte-level BPE's base vocabulary is all 256 possible byte values, and every string in any encoding decomposes into some sequence of those bytes. An unseen emoji just costs more tokens (its UTF-8 bytes, individually or partially merged) — it is never unrepresentable.

This is easy to confuse with character-level BPE, which operates over Unicode codepoints rather than bytes and can in principle be handed a codepoint outside its trained vocabulary — byte-level BPE closes exactly that gap by working one level lower, on the closed 256-value byte alphabet.

Question 8 Bug fix

This embedding lookup is meant to return the vector for a batch of token ids by directly indexing the embedding matrix, the way nn.Embedding does it — a gather, not a matmul. Find the bug.

def embed_lookup(E, token_ids, vocab_size):
    one_hot = np.zeros((len(token_ids), vocab_size))
    for i, tok in enumerate(token_ids):
        one_hot[i, tok] = 1
    return one_hot @ E   # returns embedded vectors, shape (len(token_ids), d_model)
  1. There is no bug — the one-hot matmul returns the mathematically correct vectors.
  2. The bug is that one_hot should have shape (vocab_size, len(token_ids)) instead.
  3. The bug is that E should be transposed before the matmul.
  4. The bug is that this uses @ instead of element-wise multiplication.
Show answer

Answer: A, there is no bug. This is a direct, correct implementation of Lesson 3's one-hot-matmul equivalence — one_hot @ E does zero out every row of E except the ones at each token's index and correctly returns shape (len(token_ids), d_model). It's mathematically right; the actual problem with this code is that it's wasteful, not wrong — a real implementation would use direct indexing (E[token_ids]) instead of building and multiplying a mostly-zero one-hot matrix, exactly the distinction Lesson 3 draws between "mathematically equivalent" and "how it's actually implemented."

The shape and transpose distractors (B and C) both describe changes that would actually break the matmul dimensions or produce the wrong result — the code as written is dimensionally and numerically correct, just computationally wasteful compared to a gather.
