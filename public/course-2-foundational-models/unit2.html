The transformer architecture, in detail

Attention replaced recurrence by turning "read the sequence in order" into "look up whatever's relevant, regardless of distance." This unit derives that mechanism from scratch — query/key/value, multi-head, position, masking — using 40–400 clause commercial contracts as the running example, because contract cross-referencing is exactly the long-range dependency problem attention was built to solve.

Lesson 1

Self-attention from first principles

Suppose you hand an LSTM a 400-clause commercial supply agreement, token by token, and ask it a question that only clause 287 can answer: does this indemnification obligation actually apply to the loss in front of us? Clause 287 says the indemnifying party "shall reimburse the Indemnified Party for all Losses arising from a breach of Section 4.2." Whether that sentence means anything depends entirely on clause 3, roughly 284 clauses earlier, which defines Losses as "any and all damages, costs, and expenses, including reasonable attorneys' fees, but excluding consequential or punitive damages." Course 1 Unit 7 walked through why an LSTM struggles here: every token between clause 3 and clause 287 forces another sequential update of a single fixed-size hidden state, and each update is another opportunity for the gates to overwrite, dilute, or simply fail to preserve that one definition. The vanishing-gradient argument from Unit 7 was about training signal decaying backward through time; the same recurrence has a forward-pass problem too — a fixed-width hidden vector has finite capacity, and 284 steps of forget-gate arithmetic is a lot of chances to lose one sentence. By the time the network reaches clause 287, clause 3 is not necessarily "remembered" in any useful sense. It has been compressed, and compression is lossy.

Self-attention exists to remove the sequential bottleneck entirely. Instead of forcing information to survive a relay race through hundreds of hidden-state updates, every token gets direct, one-step access to every other token in the sequence. Clause 287's representation of "Losses" doesn't wait for clause 3's meaning to survive 284 hops of compression — it reaches back and reads clause 3 directly, in parallel with every other clause, weighted by how relevant each one is right now. This is why attention is best understood not as a sequence-processing trick but as content-based lookup: given what clause 287 is currently trying to resolve, which of the other 399 clauses actually matter, and how much?

Query, key, value — and why three projections, not one

A content-based lookup needs three distinct roles, and a transformer layer learns a separate linear projection for each. The query is what a token is asking for right now — for the token "Losses" sitting inside clause 287's obligation, the query encodes something like "I am an undefined capitalized term inside an indemnity clause; where was I defined?" The key is what a token advertises about itself to anyone searching — the token "Losses" sitting inside clause 3's definition carries a key that encodes "I am a defined term, matchable by later uses of the same capitalized word, in a definitions section." The value is the actual content that gets copied forward once a match is found — not a matching signal at all, but the substance of the definition itself: the exclusion of consequential damages, the inclusion of attorneys' fees.

Those three roles are in real tension. The same word, "Losses," must simultaneously behave as an asker (in clause 287), an advertiser (in clause 3), and a payload (the definition's content). Cramming all three jobs into a single vector would mean the geometry that makes two tokens "match well" (query and key pointing in compatible directions) is the same geometry that determines what content gets passed along — there is no reason those two things should share a representation. Query, key, and value are each a separate learned linear map, W_Q, W_K, W_V, applied to the same input embedding, so the network can rotate the same token into three different subspaces: one for asking, one for being found, one for what to hand over.

The scaled dot product — and why the scaling isn't cosmetic

Once every token has a query, key, and value vector, relevance between token i and token j is scored as the dot product q_i · k_j — large when the query and key point in similar directions, near zero when they're unrelated, negative when they conflict. But raw dot products have a scaling problem that is easy to miss and important to derive properly. Assume, as is roughly true after a well-initialized projection, that each component of q and k is independent, mean zero, and unit variance. The dot product is a sum of d_k such term-products, q · k = Σ q_i k_i. Each individual term q_i k_i has mean zero (independence) and variance E[q_i² k_i²] = E[q_i²]·E[k_i²] = 1. Summing d_k independent terms, each with variance 1, gives a total variance of d_k — so the standard deviation of the raw dot product grows as √d_k. For a 64-dimensional head, that's a standard deviation around 8, not 1. Left unscaled, that means as head dimension grows, the softmax inputs get proportionally more extreme, the softmax saturates toward a near one-hot distribution regardless of whether that's the right answer, and its gradient — which is largest when the distribution is close to uniform — collapses toward zero almost everywhere. Dividing every score by √d_k before the softmax cancels exactly that growth, holding the pre-softmax variance at roughly 1 no matter how large d_k is. This is the full derivation behind the otherwise-mysterious √d_k in the attention formula from Vaswani et al.'s 2017 "Attention Is All You Need."

Attention(Q, K, V) = softmax( QKᵀ / √d_k ) · V

The softmax is taken across the key dimension — for a fixed query (clause 287's "Losses"), it produces one probability per key (every other token in the contract), summing to 1. Those probabilities become the weights in a weighted sum over the value vectors: the output for clause 287's token is a blend of every value in the contract, but overwhelmingly dominated by clause 3's value once training has pushed the query/key geometry to align "Losses"-the-user with "Losses"-the-definition.

The cost, in real numbers

This lookup is not free. Scoring every query against every key is an n × n comparison, so the cost of one attention layer scales as O(n²) in sequence length. For a 400-clause contract represented at clause granularity — one vector per clause, n = 400 — that's 400² = 160,000 pairwise scores per attention head, computed and softmaxed every layer, every head. At d_model = 768, the QKᵀ multiply and the subsequent weighted sum over values together cost roughly 2 · n² · d_model ≈ 2 · 160,000 · 768 ≈ 246 million multiply-adds per layer. That's entirely tractable on modern hardware. But real transformers don't usually operate at clause granularity — they operate on subword tokens, and a 400-clause commercial agreement with roughly 120 tokens per clause is around 48,000 tokens long. At that granularity, n² = 48,000² ≈ 2.3 billion pairwise scores, and the same FLOP estimate balloons to the trillions per layer. The quadratic cost is the reason full self-attention over an entire long contract, at raw token resolution, is not something you do casually — a constraint this course returns to directly when Unit 5 covers scaling.

import numpy as np

np.random.seed(0)

# Toy fragment: 6 token embeddings spanning the antecedent
# definition in clause 3 and its later use in clause 287,
# projected into a shared d_model = 8 space for illustration.
seq_len, d_model, d_k = 6, 8, 8

X = np.random.randn(seq_len, d_model)
print("X (token embeddings):", X.shape)              # (6, 8)

W_q = np.random.randn(d_model, d_k) * 0.1
W_k = np.random.randn(d_model, d_k) * 0.1
W_v = np.random.randn(d_model, d_k) * 0.1

Q, K, V = X @ W_q, X @ W_k, X @ W_v
print("Q:", Q.shape, "K:", K.shape, "V:", V.shape)    # each (6, 8)

scores = Q @ K.T
print("raw scores (QKᵀ):", scores.shape)              # (6, 6)

scaled_scores = scores / np.sqrt(d_k)
print("scaled scores:", scaled_scores.shape)           # (6, 6)

def softmax(x, axis=-1):
    x = x - x.max(axis=axis, keepdims=True)
    e = np.exp(x)
    return e / e.sum(axis=axis, keepdims=True)

weights = softmax(scaled_scores, axis=-1)
print("attention weights:", weights.shape,
      "row sums:", weights.sum(axis=-1))                # (6, 6), rows sum to 1.0

output = weights @ V
print("output:", output.shape)                          # (6, 8) — same shape as X

# Empirical check of the variance derivation above, at a
# realistic head width (d_k = 64):
d_k2, trials = 64, 20000
q = np.random.randn(trials, d_k2)
k = np.random.randn(trials, d_k2)
dots = (q * k).sum(axis=1)
print("std(raw dot), d_k=64:", dots.std())              # ~8.0
print("sqrt(d_k):", np.sqrt(d_k2))                      # 8.0

Every intermediate shape is printed on purpose: notice Q, K, V all keep the input's (seq_len, d_k) shape, scores and weights are square (seq_len, seq_len), and the final output returns to the input's shape — attention re-mixes content, it doesn't resize it. The last two lines confirm the √d_k derivation numerically: raw dot products at d_k=64 really do have a standard deviation of about 8, matching the closed-form prediction, not a rounder or smaller number.

Key takeaways

  • Attention is direct lookup, not sequential compression. A token attends to every other token in one step, replacing the LSTM's relay race of 284 hidden-state updates between clause 3 and clause 287 with a single weighted read.
  • Query, key, and value are separate projections because they play conflicting roles — asking, advertising, and delivering content — for the same underlying token, and one vector can't optimize for all three geometries at once.
  • The √d_k scaling is a variance correction, not a convention. A dot product of d_k independent unit-variance terms has variance d_k; dividing by √d_k restores unit variance so the softmax doesn't saturate as head width grows.
  • Attention cost is O(n²) in sequence length — 160,000 pairwise scores per head for a 400-clause contract at clause granularity, but billions at full 48,000-token resolution, which is why long-document transformers need more than brute-force full attention.

Lesson 2

Multi-head attention and the full transformer block

Multi-head attention runs several independent attention computations — heads — in parallel on the same input, each with its own learned Q/K/V projections, and concatenates their outputs before a final linear mix. That's the definition; the reason it exists is that a single attention head is a single weighted average, and a single weighted average cannot represent several genuinely different relevance judgments at once. Take the word "Losses" in clause 287 again. Resolving that one obligation actually depends on several unrelated things: clause 3's definition (what counts as a Loss at all), clause 15's carve-outs (categories of Loss specifically excluded), and clause 199's aggregate liability cap (a ceiling that changes whether the obligation is even worth invoking). A single head has to produce one softmax distribution over all 400 clauses and one resulting blend of values. If it leans toward attending strongly to clause 3, it necessarily attends less to clause 199; the geometry of one query vector can point in one dominant direction at a time. Multiple heads sidestep this by giving each relationship its own private query/key/value subspace and its own softmax — one head's attention pattern can specialize in defined-term resolution while a different head specializes in liability-cap tracking, and neither has to compromise with the other before the outputs are recombined.

Splitting d_model into heads: the reshape/transpose dance

Mechanically, multi-head attention does not project into a larger space per head — it splits the model's existing width across heads. If d_model = 768 and you use h = 12 heads, each head operates in d_head = d_model / h = 64 dimensions. The projections W_Q, W_K, W_V are each still a single 768 × 768 matrix — computed once, in full width, for efficiency — and the result is then reshaped so that the 768 output dimensions are viewed as 12 separate 64-dimensional chunks. Concretely, a tensor of shape (batch, seq_len, 768) is reshaped to (batch, seq_len, 12, 64) and then transposed to (batch, 12, seq_len, 64), moving the head dimension next to the batch dimension so that the h heads behave, for the purposes of the QKᵀ matmul, like h independent batches. Attention runs identically to Lesson 1 within each of those 12 slices, entirely in parallel, producing (batch, 12, seq_len, 64). The heads are then transposed back to (batch, seq_len, 12, 64) and reshaped (concatenated) into (batch, seq_len, 768) — the same width the block started with — before a final output projection W_O (another 768 × 768 matrix) mixes information across heads, since concatenation alone only places head outputs side by side without letting them interact.

What heads specialize in, empirically

You don't have to take specialization on faith. Studies probing trained transformers — notably Clark et al.'s 2019 analysis of BERT's attention and Voita et al.'s 2019 head-pruning work — found consistent, interpretable patterns: some heads are overwhelmingly positional, attending almost entirely to the immediately preceding or following token regardless of content (useful for local syntax — matching a modifier to the noun right after it, like "Indemnifying Party" to "shall"); some are syntactic, tracking a specific grammatical relationship such as a verb attending to its direct object no matter how far apart they sit in the sentence; and some specialize in exactly the long-range, sparse pattern this unit cares about — a head that, across many contracts, learns to attend from a capitalized defined term back to its definition, or from an obligation clause to the liability cap that bounds it. Voita et al. also showed that many heads in a trained model are redundant and can be pruned with little quality loss — evidence that "more heads always help" is not automatic; what matters is that the heads that survive cover genuinely different relationships.

Around the attention: residuals, layer norm, and the FFN

Multi-head attention is one sub-layer of a transformer block, not the whole block. Every sub-layer — attention and the feed-forward network — is wrapped in a residual connection, output = sublayer(x) + x, so the network always has an unobstructed identity path and only has to learn a correction, not a full re-derivation of the input at every layer. Each sub-layer is also paired with layer normalization, which rescales activations across the feature dimension to stabilize the distribution of values flowing into the next sub-layer. Where the normalization sits relative to the residual matters more than it looks: the original 2017 design applied it after the residual sum (post-LN): LayerNorm(x + sublayer(x)). Later work — Xiong et al., 2020, "On Layer Normalization in the Transformer Architecture" — showed that post-LN puts the normalization directly in the residual path, so the identity shortcut isn't truly clean; gradients flowing back through many stacked blocks pass through a normalization at every step, and empirically this made very deep post-LN stacks unstable to train without a careful learning-rate warmup. Pre-LN instead normalizes the sub-layer's input before it runs — x + sublayer(LayerNorm(x)) — leaving the residual stream itself untouched by any normalization. That keeps the identity path truly linear all the way through the stack, which is why pre-LN became the default for deep transformers (12, 24, 96+ blocks) even though post-LN, at shallow depth, can reach marginally better final quality.

The other sub-layer is the position-wise feed-forward network (FFN): two linear layers with a nonlinearity between them, applied identically and independently to every token's vector. Its defining trait is a 4x expansion — d_model = 768 in, up to 3072, a nonlinearity (originally ReLU, now usually GELU), back down to 768. Attention mixes information across tokens; the FFN then processes each token's (now context-aware) vector in isolation, and the wider hidden layer is where a large share of the block's raw parameter count and representational capacity actually lives.

Counting the parameters in one block

At d_model = 768, h = 12: the four attention projections (W_Q, W_K, W_V, W_O) are each 768 × 768 + 768 (weights plus bias) ≈ 590,592 parameters, for 4 × 590,592 ≈ 2,362,368 total. The FFN's two linear layers are 768 × 3072 + 3072 and 3072 × 768 + 768, roughly 2,359,296 + 3,072 each way, summing to ≈ 4,722,432. Two layer norms add a negligible ≈ 3,072 more (a scale and a bias vector per normalization). Total: ≈ 7.09 million parameters for one block — closely matching the common back-of-envelope estimate of 12 × d_model² (12 × 768² ≈ 7,077,888), since attention contributes 4d² and the FFN contributes 8d² of that count. A 12-layer encoder at this width is therefore roughly 85 million parameters before counting embeddings — a useful sanity check whenever you're told a model's depth and width and need to estimate its size.

import numpy as np

def softmax(x, axis=-1):
    x = x - x.max(axis=axis, keepdims=True)
    e = np.exp(x)
    return e / e.sum(axis=axis, keepdims=True)

batch, seq_len, d_model, h = 2, 400, 768, 12
d_head = d_model // h
print("d_head:", d_head)                              # 64

X = np.random.randn(batch, seq_len, d_model) * 0.02

W_q = np.random.randn(d_model, d_model) * 0.02
W_k = np.random.randn(d_model, d_model) * 0.02
W_v = np.random.randn(d_model, d_model) * 0.02
W_o = np.random.randn(d_model, d_model) * 0.02

Q, K, V = X @ W_q, X @ W_k, X @ W_v
print("Q before split:", Q.shape)                     # (2, 400, 768)

def split_heads(t):
    b, n, dm = t.shape
    t = t.reshape(b, n, h, d_head)                     # (batch, seq, heads, d_head)
    return t.transpose(0, 2, 1, 3)                      # (batch, heads, seq, d_head)

Qh, Kh, Vh = split_heads(Q), split_heads(K), split_heads(V)
print("Q after split_heads:", Qh.shape)                # (2, 12, 400, 64)

scores = Qh @ Kh.transpose(0, 1, 3, 2) / np.sqrt(d_head)
print("scores:", scores.shape)                          # (2, 12, 400, 400)

weights = softmax(scores, axis=-1)
head_out = weights @ Vh
print("head_out:", head_out.shape)                      # (2, 12, 400, 64)

concat = head_out.transpose(0, 2, 1, 3).reshape(batch, seq_len, d_model)
print("concat:", concat.shape)                          # (2, 400, 768)

output = concat @ W_o
print("output:", output.shape)                          # (2, 400, 768) — same shape as X

The two reshape/transpose calls are the whole trick: splitting (batch, seq, d_model) into (batch, heads, seq, d_head) makes the 12 heads behave like 12 independent batches during the matmul, and the mirror-image reshape at the end merges them back before the output projection W_o lets the heads interact.

Key takeaways

  • One head is one weighted average — it can't simultaneously attend strongly to a definition, a carve-out, and a liability cap without compromising between them. Multiple heads give each relationship its own subspace and softmax.
  • Heads split d_model, they don't multiply it: 768 / 12 = 64-dimensional heads, via a reshape-transpose-attend-transpose-reshape pipeline, followed by an output projection W_O that lets heads' information interact.
  • Trained heads specialize measurably — positional, syntactic, and rare/defined-term tracking are documented patterns (Clark et al. 2019, Voita et al. 2019), and not every head is necessary (some are prunable).
  • Pre-LN beats post-LN at depth because it keeps the residual stream free of any normalization, giving deep stacks a clean identity path and removing the need for careful warmup schedules.
  • One transformer block at d_model=768 is ≈7.09M parameters — roughly 12·d_model², split about a third attention and two-thirds FFN.

Lesson 3

Position, masking, and the three architecture families

Picture the 400 clauses of a supply agreement photocopied onto loose index cards, then dropped and shuffled into a random pile. Self-attention, as built so far, effectively operates on that shuffled pile: each token's output is a weighted sum over every other token's value, and that weighting depends only on how well queries and keys match by content — nothing in the mechanism cares whether a given key came from the token immediately before or three hundred tokens away. Permute the input order and the entire set of outputs permutes identically along with it; attention is permutation-equivariant. That's a serious problem for legal text specifically, because clause order is not decoration — "the foregoing," "Section 4.2 above," and the clause numbering itself are all load-bearing references to sequence, and "clause 287 governs an obligation defined back in clause 3" is a fact about where things sit, not just what they say. A bag of shuffled index cards has no way to represent "clause 3 comes before clause 287." Position has to be injected into the model explicitly, as an additional signal added to (or combined with) the content embedding, before attention ever runs.

Sinusoidal, learned, and rotary position schemes

The original transformer's solution was a fixed sinusoidal position encoding: for position pos and embedding dimension index i, each pair of dimensions is assigned a sine and a cosine at a different frequency, PE(pos, 2i) = sin(pos / 10000^(2i/d_model)) and PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model)). The frequency intuition is a clock with many hands: low dimension indices oscillate fast, like a seconds hand, distinguishing clause 3 from clause 4; high dimension indices oscillate slowly, like an hour hand, distinguishing clause 3 from clause 303. Together, the full set of frequencies gives every position a unique, fixed fingerprint, and because sine and cosine of a shifted angle can be written as a linear combination of the unshifted sine and cosine, a fixed offset between two positions (clause 287 minus clause 3 = 284) corresponds to a fixed linear transformation of the encoding — in principle letting the network learn to attend by relative offset, not just absolute position. A simpler alternative, learned absolute position embeddings, just assigns each position index its own trainable vector, exactly like a token embedding table indexed by position instead of by word. It's simpler and often works as well in practice, but it hard-codes a maximum sequence length — a model trained on contracts up to 2,048 tokens has no embedding, learned or otherwise, for token 2,049, and can't extrapolate to a longer document at all.

RoPE (Rotary Position Embedding) takes a different approach: rather than adding a position signal to the content embedding, it rotates each query and key vector by an angle proportional to its position, treating pairs of dimensions as 2D coordinates and applying a rotation matrix. The key property is that the dot product between a rotated query at position m and a rotated key at position n depends mathematically only on the relative offset m − n, not on m and n individually — relative position falls out of the geometry rather than being learned or hand-designed. Because the rotation is a smooth, well-defined function of position for any integer, RoPE tends to extrapolate to sequence lengths beyond what was seen in training noticeably better than sinusoidal or learned absolute encodings, which is a meaningful advantage when a model trained mostly on shorter agreements later has to process an unusually long one. ALiBi (Attention with Linear Biases) skips modifying Q and K altogether and instead adds a fixed, non-learned penalty directly to the attention scores: a bias of −m · (i − j) subtracted from the score between positions i and j, growing linearly with distance. It has no parameters to learn at all, is simple to compute, and in the original paper extrapolated to much longer sequences than the model was trained on, at the cost of imposing a fairly rigid "closer is generally more relevant" prior that isn't always true of legal cross-references, where the most relevant clause can be the furthest one away.

Masking: causal, padding, and composing the two

Position solves "where is this token," but some architectures also need to restrict "which tokens is this one even allowed to see." A causal mask enforces that token i can attend only to tokens at position ≤ i, never to anything later — essential for any model that generates text left to right, since at generation time the tokens after the current position don't exist yet. It's implemented by adding −∞ to every score in the strictly-upper-triangular part of the score matrix (row i, column j, for j > i) before the softmax, so those positions receive exactly zero weight after exponentiating. A padding mask solves a different, purely practical problem: contracts in a batch are different lengths, so shorter ones are padded to match the longest, and those pad tokens carry no real content — they must never receive or contribute attention weight, or the model's output would depend on an implementation detail (batch composition) rather than the document itself. The fix is the same mechanism, a large negative value added at every padded position's column. The two masks compose by simple addition: sum the causal mask and the padding mask (each 0 where allowed, −∞ where forbidden) and add the result to the raw scores once, before a single softmax — a position that's blocked by either mask ends up at −∞ either way, and a position has to clear both to receive nonzero weight.

Encoder-only, decoder-only, encoder-decoder

Whether a model uses causal masking at all, and how many "stacks" of transformer blocks it has, defines three architecture families with different jobs. Encoder-only models (BERT is the canonical example) use no causal mask — every token attends bidirectionally to every other token in the same pass — because the goal is a rich contextual representation of text that already exists, not generating new text. That makes encoder-only models the right fit for clause retrieval (represent all 400 clauses, then rank them against a query clause by embedding similarity) and indemnity-clause classification (does this clause create an indemnification obligation, yes or no), where the whole document is available up front and the output is a label or a ranking, not new prose. Decoder-only models (the GPT family) use a causal mask throughout and are trained to predict the next token, which is what makes them naturally suited to generation — drafting a proposed indemnification clause, or continuing a contract from a partial draft — at the cost of never getting to look ahead at tokens the model itself hasn't produced yet, even during analysis of a document that already fully exists. Encoder-decoder models (T5, and the original 2017 Transformer) run an unmasked encoder over the full input and a causal decoder that additionally cross-attends to the encoder's output — well suited to tasks with a clearly separate input and output, such as summarizing a 400-clause contract into a two-paragraph risk memo, where the model needs to fully digest the source (encoder, bidirectional) before producing new text (decoder, causal) grounded in it.

import numpy as np

def sinusoidal_position_table(max_len, d_model):
    pos = np.arange(max_len)[:, None]                       # (max_len, 1)
    i = np.arange(d_model)[None, :]                          # (1, d_model)
    angle_rates = 1 / np.power(10000, (2 * (i // 2)) / d_model)
    angles = pos * angle_rates                               # (max_len, d_model)
    pe = np.zeros((max_len, d_model))
    pe[:, 0::2] = np.sin(angles[:, 0::2])
    pe[:, 1::2] = np.cos(angles[:, 1::2])
    return pe

pe = sinusoidal_position_table(max_len=8, d_model=16)
print("position table:", pe.shape)                         # (8, 16)

def causal_mask(seq_len):
    # -inf strictly above the diagonal: clause i cannot see clause j > i
    return np.triu(np.full((seq_len, seq_len), -np.inf), k=1)

seq_len = 6   # six clauses in reading order
mask = causal_mask(seq_len)

np.random.seed(1)
raw_scores = np.random.randn(seq_len, seq_len)
masked_scores = raw_scores + mask
print("masked scores:\n", np.round(masked_scores, 2))

def softmax(x, axis=-1):
    x = x - x.max(axis=axis, keepdims=True)
    e = np.exp(x)
    return e / e.sum(axis=axis, keepdims=True)

weights = softmax(masked_scores)
print("attention weights:\n", np.round(weights, 2))

The masked-scores printout shows real values on and below the diagonal and -inf strictly above it; after softmax, every one of those -inf entries becomes exactly 0.0, so clause 2 (row index 1) shows nonzero weight only in columns 0–1, never columns 2–5 — the causal mask made visible as a matrix.

Key takeaways

  • Attention alone is permutation-equivariant — shuffle the input tokens and the outputs shuffle identically, which is exactly wrong for text where order (clause numbering, "the foregoing") carries meaning.
  • Sinusoidal encodings use multiple frequencies to give every position a unique fingerprint, and their sine/cosine structure lets relative offsets be expressed as linear transforms; learned absolute embeddings are simpler but can't extrapolate past the trained max length.
  • RoPE rotates Q/K by position so their dot product depends only on relative offset, which is why it extrapolates to longer sequences better than sinusoidal or learned absolute schemes; ALiBi achieves similar extrapolation with a simple learned-free linear distance penalty on the scores instead.
  • Causal and padding masks both work by adding -∞ to forbidden score positions before the softmax, and compose by simple addition — a position needs to clear both to receive any attention weight.
  • Pick the family by the shape of the task: encoder-only for classification/retrieval over existing text, decoder-only for generation, encoder-decoder when input and output are distinct documents (contract in, risk memo out).

Question 1 Multiple choice

Why does a transformer layer learn three separate projections (query, key, value) instead of reusing one projected vector for all three roles?

  1. Three projections are required so the layer has more trainable parameters, which always improves quality.
  2. Query, key, and value play conflicting roles for the same token — asking, being matched against, and supplying content — and one shared geometry can't optimize for all three at once.
  3. It's purely a historical convention from the original paper with no functional benefit; a single shared projection produces identical results.
  4. Separate projections are needed only to allow multi-head attention; single-head attention needs just one projection.
Show answer

Answer: B. A query encodes what a token is looking for, a key encodes how a token advertises itself for matching, and a value is the content actually copied forward once a match is found. These are different jobs with different natural geometries, so a single vector reused for all three would force a compromise between them.

A is wrong because parameter count isn't the reason — the three projections exist for a functional, not a scaling, reason. C is wrong because the roles genuinely differ; collapsing them changes what the layer can represent, it isn't a no-op. D is wrong because single-head attention already needs all three roles — Lesson 1's derivation used one head and still required Q, K, and V separately.

Question 2 True / false

Dividing the raw dot-product scores by √d_k before the softmax is necessary because, for query and key vectors with independent, mean-zero, unit-variance components, the variance of q·k grows linearly with d_k — so without scaling, the softmax would saturate more as head dimension increases.

Show answer

Answer: True. The dot product is a sum of d_k independent terms, each with variance 1, so the total variance is d_k and the standard deviation is √d_k. Dividing by √d_k restores unit variance regardless of d_k, keeping the softmax's input scale — and therefore its gradient behavior — stable as head width changes.

A tempting misconception is that the scaling is just "normalizing to keep numbers small" for numerical stability in general — that's not the specific mechanism. The derivation is about controlling variance growth with d_k specifically, not about avoiding overflow in general arithmetic.

Question 3 Fill in the blank

A transformer block with d_model = 768 and h = 12 attention heads gives each individual head a dimension d_head =  .

Show answer

Answer: 64. d_head = d_model / h = 768 / 12 = 64. Multi-head attention splits the model's existing width across heads rather than projecting into extra dimensions per head — the four projection matrices (W_Q, W_K, W_V, W_O) each stay a single 768×768 matrix; only the reshape after projection carves the 768 output dimensions into 12 chunks of 64.

Complete the missing line so that split_heads turns a (batch, seq, d_model) tensor into (batch, heads, seq, d_head), ready for the per-head QKᵀ matmul.

Question 5 Multiple choice

Why did pre-LN (normalizing a sub-layer's input before it runs, x + sublayer(LayerNorm(x))) become the default over post-LN (LayerNorm(x + sublayer(x))) for very deep transformer stacks?

  1. Pre-LN keeps the residual stream itself free of normalization, so the identity path through many stacked blocks stays linear and gradients don't have to pass through a normalization at every block.
  2. Pre-LN uses fewer parameters than post-LN, since it only normalizes half as often.
  3. Post-LN was discovered to be mathematically identical to pre-LN, so the switch was purely about code readability.
  4. Pre-LN removes the need for residual connections entirely, simplifying deep stacks.
Show answer

Answer: A. In post-LN, the normalization sits directly in the residual path, so the "clean" identity shortcut isn't actually clean — every block adds a normalization the gradient must pass through. Pre-LN normalizes only the sub-layer's input, leaving x + sublayer(...) as a true, unobstructed identity path, which is what made very deep (dozens of blocks) stacks trainable without careful warmup.

B is wrong — both apply the same number of LayerNorm operations per block (one per sub-layer), just at a different point relative to the residual add. C is wrong — the two are not equivalent; Xiong et al. (2020) showed they behave differently, especially at depth. D is wrong — pre-LN still uses residual connections; it changes where normalization sits relative to them, not whether they exist.

Question 6 Bug fix

This function is meant to build a causal mask so that clause 287 can attend to clauses 1 through 287 but never to clauses 288 onward. Run against a 6-clause example, every clause loses the ability to attend to itself. What's the bug, and what's the fix?

def causal_mask(seq_len):
    mask = np.triu(np.full((seq_len, seq_len), -np.inf), k=0)
    return mask
Show answer

Bug: k=0 includes the diagonal in the masked region. np.triu(..., k=0) zeroes out (here, sets to -∞) everything on and above the main diagonal, which means position i is masked against itself, not just against future positions j > i. Fix: use k=1, which starts the masked region strictly above the diagonal, leaving each position free to attend to itself and everything before it.

It's a one-character bug but a meaningful one — with k=0, every row's softmax would have to assign all its weight to strictly earlier positions and none to the current token, silently discarding a clause's own content, which is never the intended behavior of a causal mask.

Question 7 Multiple choice

A legal-tech team needs a model that reads all 400 clauses of a contract at once and outputs a binary label per clause — "creates an indemnification obligation" or not — with the full document available up front. Which architecture family fits best, and why?

  1. Decoder-only, because causal masking is required for any classification task.
  2. Encoder-decoder, because the task has a separate input and output.
  3. Encoder-only, because the whole document is available at once and the task needs rich bidirectional context per clause, not text generation.
  4. None of the three families apply; classification requires a separate non-transformer architecture.
Show answer

Answer: C. Clause classification over an already-complete document benefits from every clause attending bidirectionally to every other clause — there's no "future" to hide, since nothing is being generated left to right. That's exactly the encoder-only setup (BERT-style), which produces per-token or per-clause representations for a downstream classifier head.

A is backwards — causal masking is for generation, where future tokens genuinely don't exist yet at inference time; it would only throw away useful bidirectional context here. B is wrong because the output is a per-clause label, not a separate generated document, so there's no need for a decoder stage at all. D is wrong — encoder-only transformers are a standard, effective choice for exactly this kind of classification task.

Question 8 True / false

Because self-attention is permutation-equivariant, a transformer with no position information at all would still correctly distinguish "clause 3 defines Losses, and clause 287 uses the term" from a shuffled version of the same contract with clauses 3 and 287 swapped in position.

Show answer

Answer: False. Permutation-equivariance means the exact opposite: if you permute the input tokens, the attention outputs permute identically along with them, purely as a function of content-based matching. Without any injected position signal, the model has no way to represent "comes before" or "comes after" at all — every clause-ordering fact, including which clause is the definition and which is the later use, would have to come from content alone, which is exactly why sinusoidal, learned, RoPE, or ALiBi position information has to be added before attention runs.

The tempting error is assuming that because attention can already match "Losses" in clause 287 to "Losses" in clause 3 by content similarity, order doesn't matter for this example — but that content match tells you the two mentions are related, not which one is the definition and which is the use, a distinction that depends on their relative position in the document.
