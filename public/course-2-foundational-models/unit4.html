Pretraining objectives

Lesson 1

Causal language modelling

Causal language modelling (CLM) is the objective that turns a decoder-only transformer into a next-token predictor: given every token up to position i, produce a distribution over the vocabulary for the token at position i+1, and do this at every position in every training sequence at once. It is the objective behind GPT-family models, and it is the one you will spend the most wall-clock time on, because it is what a code-completion model — the kind that finishes a line as you type it in an editor — is actually trained to do.

Feed the model a batch of tokenized functions from your pretraining corpus — say, four snippets that survived the license filter and near-duplicate removal from your crawl, tokenized with the byte-pair tokenizer you built in Unit 3 across Python, Java, Go, and JavaScript source. The causal mask from Unit 2 guarantees that position i's output can only attend to positions 0..i; it can never peek at position i+1. That masking is what makes the training procedure below sound — without it, "predicting" the next token would just be copying it.

Teacher forcing and the shift

During training you do not let the model generate its own prefix and then predict from that (self-generated) prefix — you feed it the ground-truth prefix at every step, regardless of what the model would have predicted on its own. This is teacher forcing: at position i the model conditions on the real tokens 0..i from the training corpus, not on its own possibly-wrong guesses. It is what makes CLM training parallelizable — every position's loss term can be computed independently in a single forward pass, because you already know what every position's input should be.

The mechanical consequence of teacher forcing is the single most common implementation bug in this objective: the input/label shift. Your raw tensor is one sequence of token ids. The model's input at position i is the token at position i; the model's label at position i is the token at position i+1 — the one it's supposed to predict. Concretely, if your tokenized tensor is tokens with shape (batch, seq_len), the input is tokens[:, :-1] and the label is tokens[:, 1:] — both of shape (batch, seq_len-1). Get this wrong — feed the model the same slice as both input and label, or shift in the wrong direction — and the loss will look deceptively reasonable (the model can partially solve "predict the token you were just given" by learning an identity-like shortcut through the earlier layers) while validation perplexity and generation quality quietly collapse. This is the bug that Question 5 below asks you to catch.

Cross-entropy and perplexity

At each position, the model outputs a logit vector of size vocab_size; softmax turns it into a probability distribution; the loss at that position is the negative log-probability the model assigned to the true next token — cross-entropy. Averaged over every valid position in the batch:

L = -(1/N) Σᵢ log P(yᵢ | y<ᵢ)

where yᵢ is the true token at position i and N is the total number of predicted positions across the batch. Perplexity is just this loss re-expressed on a scale that is easier to reason about: PPL = exp(L). A perplexity of 1 means the model assigns probability 1 to every correct next token — perfect prediction. A perplexity of V (your vocabulary size) means the model is doing no better than picking uniformly at random among all tokens. For a code model with a ~50k-token BPE vocabulary, a well-trained model typically lands in the range of 2–4 perplexity on held-out repository code — meaning that, on average, the model behaves as if it were choosing uniformly among only 2 to 4 plausible next tokens at each position, not 50,000. That's a useful mental model: perplexity is "effective branching factor," not an abstract log-loss number.

The training/generation asymmetry

Because teacher forcing supplies the true prefix at every position, CLM training computes the loss at all seq_len - 1 positions in one forward pass — it is embarrassingly parallel across positions, which is exactly why transformers pretrain efficiently on accelerators built for parallel matrix multiplication. Generation is the opposite: at inference time there is no ground-truth prefix to feed, so the model must produce token i+1, append it, and feed the whole extended sequence back in to produce token i+2. Generation is inherently serial — one forward pass per output token (modulo tricks like speculative decoding, which are out of scope here). This asymmetry — parallel training, serial inference — is why a code-completion model is cheap to pretrain on a cluster and comparatively expensive to serve one keystroke at a time in an IDE.

import torch
import torch.nn.functional as F

# tokens: BPE ids from the Unit 3 tokenizer, one row per function opening
# "def add(a, b):" / "def sub(a, b):" / "def mul(a, b):" / "def div(a, b):"
tokens = torch.tensor([
    [101, 45, 902, 17, 6, 88, 231, 5],
    [101, 45, 77,  17, 6, 12, 231, 5],
    [101, 45, 300, 17, 6, 88, 231, 5],
    [101, 45, 512, 17, 6, 12, 231, 5],
])                                    # shape: (4, 8)

# THE OFF-BY-ONE, done correctly:
inputs = tokens[:, :-1]               # shape: (4, 7)  -- what the model sees
labels = tokens[:, 1:]                # shape: (4, 7)  -- what it must predict

print("inputs[0]:", inputs[0].tolist())   # [101, 45, 902, 17, 6, 88, 231]
print("labels[0]:", labels[0].tolist())   # [45, 902, 17, 6, 88, 5*]
# position i of `inputs` and position i of `labels` are DIFFERENT tokens:
# labels[i] is whatever came immediately after inputs[i] in the source file.

vocab_size = 4096
logits = model(inputs)                            # (4, 7, vocab_size)
loss = F.cross_entropy(
    logits.reshape(-1, vocab_size),
    labels.reshape(-1),
    reduction="mean",
)
perplexity = torch.exp(loss)
print(f"mean NLL = {loss.item():.3f}   perplexity = {perplexity.item():.2f}")

The input and label tensors are the same underlying data at different offsets, not two different tensors — notice they share every value, just shifted by one column. Perplexity is computed once, over the whole batch, from the mean cross-entropy.

Key takeaways

  • CLM predicts token i+1 from tokens 0..i, using the causal mask from Unit 2 to prevent any position from seeing its own answer.
  • Teacher forcing feeds the true prefix, not the model's own generations, which is what makes every position's loss independently computable in one parallel forward pass.
  • The input/label shift is tokens[:, :-1] vs. tokens[:, 1:] — the single most common CLM bug is getting this shift wrong or omitting it.
  • Perplexity is exp(mean cross-entropy) and reads as an effective branching factor: a well-trained code model sits around 2–4, not near the vocabulary size.
  • Training is parallel across positions; generation is serial across tokens — the same objective that pretrains cheaply is expensive to sample from one token at a time.

Lesson 2

Masked language modelling and denoising

Say you want to build code search: given a natural-language query like "parse a JSON config file and validate required keys," retrieve the function in your indexed repositories that does that. To rank candidates you need a single vector representing each whole function — but a causal model, by construction, has never let its representation of line 1 see line 40. Its encoding of the function's opening is permanently ignorant of the function's return statement, because the causal mask from Unit 2 forbids it. For generation that's the whole point; for building a representation of a complete, already-written function, it's a real limitation. This is the problem masked language modelling and its descendants solve: how do you get a transformer to build representations that are informed by both directions of context, when the objective that made training parallel and cheap (CLM) explicitly denies it that context?

BERT-style masking: 15%, 80/10/10

BERT's answer is masked language modelling (MLM): remove the causal mask entirely (full bidirectional attention), corrupt roughly 15% of the input tokens, and train the model to reconstruct the corrupted positions from the uncorrupted rest of the sequence in both directions. Naively you'd corrupt all 15% by replacing them with a special [MASK] token. BERT instead splits that 15% three ways: 80% of selected positions are replaced with [MASK], 10% are replaced with a random token from the vocabulary, and 10% are left completely unchanged — yet all three groups still count toward the loss (the model must still predict the original identifier at all three types of positions).

The reason for the split is subtler than "add noise." If 100% of masked positions became [MASK], the model would learn to produce good representations for [MASK] tokens specifically — but at finetuning or downstream-inference time, on a real function you feed it, [MASK] never appears at all. That's the pretrain-finetune mismatch: a representation-quality gap caused by the model over-specializing to an artifact of the pretraining recipe. The 10% random-token branch forces the model to keep verifying every position against its context — including positions that look perfectly plausible on the surface — rather than learning "only bother checking positions that are literally [MASK]." The 10% unchanged branch goes one step further: since the model cannot tell, from the token alone, whether a given real-looking token (say, an identifier like config) is one of the 15% it must reconstruct or one of the 85% it can trust outright, it is forced to build a genuinely contextual representation of every token, not just the visibly corrupted ones. Both branches exist to close the same gap: making the representation the model learns during pretraining match the representation it will need to produce at inference, when no [MASK] tokens exist.

T5-style span corruption

T5 generalizes single-token masking to span corruption: instead of masking scattered individual tokens, mask contiguous spans (a run of 2–5 tokens, say a full argument list or a whole conditional expression) and replace each entire span with a single unique sentinel token (<extra_id_0>, <extra_id_1>, ...). The encoder sees the corrupted sequence with sentinels standing in for the missing spans; a separate decoder is trained, causally, to generate the sequence: sentinel, the missing span, next sentinel, its missing span, and so on. For code, this is a much better match to real corruption than independent token masking, because code's syntax is spanned — an entire slice expression or an entire decorator is the unit that's missing, not a scattered token here and there. It also reframes MLM as a sequence-to-sequence generation task, which is why T5-family models can be finetuned generatively in a way vanilla BERT cannot.

Fill-in-the-middle: denoising for a causal model

You don't have to give up the causal architecture (and its cheap, parallel training) to get infilling. Fill-in-the-middle (FIM), used to pretrain code models like Codex/CodeGen-style infilling variants, uses a document rotation trick: take a training document (a function body), split it at two random points into prefix, middle, and suffix, then reorder the pieces as prefix, suffix, middle (the "PSM" layout), wrapping each piece with a sentinel (<PRE>, <SUF>, <MID>). The model is still trained with plain next-token CLM and the plain causal mask — nothing about the architecture changes. But because the suffix now appears in the sequence before the middle, the model conditions on tokens that came both before and after the missing region in the original source, without ever needing bidirectional attention. At inference time, an IDE plugin gives the model the code before the cursor as the prefix and the code after the cursor as the suffix, and asks it to generate the middle — exactly the "complete what I'm typing given everything already in the file" use case that a pure left-to-right model can't do on its own.

import torch

def bert_style_mask(tokens, vocab_size, mask_id, special_ids, p_mask=0.15):
    """tokens: 1D LongTensor of code token ids for one function body."""
    labels = tokens.clone()
    candidate = torch.rand(tokens.shape) < p_mask
    for sid in special_ids:                 # never corrupt CLS/SEP/pad, etc.
        candidate &= tokens != sid
    labels[~candidate] = -100               # -100: ignored by cross_entropy

    choice = torch.rand(tokens.shape)
    to_mask   = candidate & (choice < 0.80)                    # 80%: -> [MASK]
    to_random = candidate & (choice >= 0.80) & (choice < 0.90)  # 10%: -> random id
    # remaining 10% of `candidate`: left unchanged, but still scored in `labels`

    tokens = tokens.clone()
    tokens[to_mask] = mask_id
    n_rand = int(to_random.sum())
    tokens[to_random] = torch.randint(0, vocab_size, (n_rand,))
    return tokens, labels                   # (corrupted_input, reconstruction_target)


def fim_reorder(func_tokens, split_a, split_b, pre_id, suf_id, mid_id):
    """Document-rotation FIM: causal model, PSM layout."""
    prefix = func_tokens[:split_a]
    middle = func_tokens[split_a:split_b]
    suffix = func_tokens[split_b:]
    return [pre_id] + prefix + [suf_id] + suffix + [mid_id] + middle
    # trained with plain CLM + plain causal mask on THIS reordered sequence --
    # the model still only ever predicts left-to-right

In bert_style_mask, all three branches (mask / random / unchanged) still show up in labels as positions to reconstruct — only the input token differs between them. In fim_reorder, note that no attention mask changes at all; the "bidirectionality" comes entirely from reordering the tokens the causal model reads.

Key takeaways

  • MLM exists because a causal model's early-token representations can't see later tokens — a real limitation for whole-function representations needed for search, not a limitation for generation.
  • The 80/10/10 split closes the pretrain-finetune mismatch: pure [MASK] would teach the model to build good representations only for a token that never appears outside pretraining.
  • The 10% random branch forces contextual verification everywhere, and the 10% unchanged branch forces it precisely because the model can't tell which real tokens are "trustworthy" from the token alone.
  • T5 span corruption masks contiguous spans with sentinel tokens and reframes reconstruction as sequence generation — a better match to code's syntactic, spanned structure than scattered single-token masking.
  • FIM gets bidirectional-ish infilling out of a purely causal model via document rotation (prefix, suffix, middle) — no architecture change, just a reordering of the training document.

Lesson 3

Contrastive objectives

Think about how you'd teach someone to recognize a composer's style without ever describing the notes: you'd play one of their pieces next to a piece by someone else and ask "which two sound like the same hand?" You never reconstruct a missing note; you never generate anything. You just learn a notion of similar versus different by comparison. That's the whole idea behind contrastive pretraining objectives — instead of reconstructing corrupted or shifted input (as in Lessons 1 and 2), the model learns by being shown pairs and asked to pull true pairs together and push everything else apart in embedding space. For code search specifically — matching a docstring like "recursively deletes empty directories under a path" to the function that implements it — this comparison-based framing turns out to be a more direct fit than either CLM or MLM, because search is fundamentally a comparison problem, not a generation problem.

Positive and negative pairs, and InfoNCE

A contrastive dataset for code search consists of positive pairs — a docstring and the function it documents, mined straight out of your license-filtered, de-duplicated repository corpus — and, implicitly, every other docstring/function combination in the batch serves as a negative pair. Given a batch of batch_size such pairs, encode all the docstrings with one encoder and all the functions with another (or a shared one), producing two matrices of embeddings, then compute the similarity between every docstring and every function. The loss most commonly used is InfoNCE (used in SimCLR, CLIP, and code-search systems like CodeSearchNet-derived setups):

L = -(1/N) Σᵢ log [ exp(sim(dᵢ, cᵢ)/τ) / Σⱼ exp(sim(dᵢ, cⱼ)/τ) ]

Read the inner fraction as a softmax over "which code snippet, among all N in the batch, is the one this docstring goes with" — the numerator is the score for the true match cᵢ, the denominator sums over every candidate cⱼ in the batch including the true one. It is literally a cross-entropy loss where the "classes" are the other items in the batch and the correct class index for row i is i itself.

Temperature, and in-batch negatives as the difficulty knob

The τ (temperature) parameter rescales the similarity scores before the softmax. As τ → 0, the softmax sharpens toward a one-hot distribution concentrated on whichever candidate has the single highest similarity — the loss becomes dominated by hard negatives (near-miss functions that are almost, but not quite, the right match) and gradients become large and noisy. As τ → ∞, every similarity score gets crushed toward the same value, the softmax flattens toward uniform, and the loss gradient vanishes — the model gets almost no signal about which negatives it's failing to distinguish. In practice τ is a tuned hyperparameter (commonly 0.05–0.1 for normalized embeddings), not something you set at either extreme.

The negatives in the denominator above come from other pairs already in the same training batch — in-batch negatives — which is why effective batch size is the difficulty knob for this objective in a way it simply isn't for CLM or MLM. A batch of 64 pairs gives the model 63 negatives per anchor to distinguish against; a batch of 8,192 pairs (achievable with large-batch contrastive training, as CLIP and later code-search systems do) gives it 8,191 — a much harder discrimination problem, which drives the encoders toward finer-grained, more separable representations. This is also why contrastive pretraining is unusually batch-size-sensitive compared to the other two objectives in this unit: shrinking the batch doesn't just slow training, it changes what the model is being asked to learn.

Dual encoders, collapse, and when to prefer this objective

The architecture that pairs naturally with InfoNCE is the dual encoder (CLIP-style, but for docstrings and functions instead of captions and images): one encoder maps docstrings to vectors, a separate encoder maps code to vectors, both projected into a shared embedding space, trained jointly so that matching pairs land near each other under cosine similarity and everything else lands far apart. At serving time this is what makes large-scale code search fast — you embed every function in your corpus once, offline, and a query only needs one forward pass through the docstring encoder plus a nearest-neighbor lookup, rather than a forward pass of a joint model against every candidate function.

The failure mode specific to this objective is representation collapse: without care, an encoder can satisfy "positives close, negatives far" by mapping every input to a nearly identical vector (which trivially makes all pairwise similarities equal and, degenerately, minimizes badly-normalized versions of the loss) or by collapsing onto a narrow subspace that loses the information a retrieval system actually needs. The standard countermeasures are L2-normalizing every embedding before the dot product (turning similarity into pure cosine similarity, so blowing up vector norms can't cheat the loss), using a large and diverse negative pool (a bigger, more varied batch makes the "map everything to one point" solution score badly, since it would confuse every negative for the positive too), and a properly tuned, non-degenerate temperature.

Prefer a contrastive objective over CLM or MLM specifically when the downstream task is retrieval or ranking across a large candidate set that must be searched by comparison, not generated. A generative model (CLM decoder, or an MLM-derived encoder finetuned with a classification head) can be adapted for search, but it wasn't optimized for "produce a vector such that true matches are near and everything else is far" — that alignment has to be bolted on after the fact, usually with a worse embedding geometry than an encoder trained contrastively from the start. Conversely, don't reach for a contrastive objective when you need the model to actually produce new text — completion, infilling, and summarization stay squarely CLM/FIM/span-corruption territory.

import torch
import torch.nn.functional as F

# doc_emb, code_emb: (batch, d) -- already L2-normalized outputs of the two
# encoders. doc_emb[i] is the docstring paired with code_emb[i]: the ONLY
# positive for row i is column i -- every other column is a negative.
temperature = 0.07
sim = doc_emb @ code_emb.T / temperature      # (batch, batch) similarity matrix
print("similarity matrix shape:", sim.shape)  # e.g. (256, 256)
print("diagonal (positive) scores:", sim.diag()[:5])

targets = torch.arange(sim.size(0))           # [0, 1, 2, ..., batch-1]
# row i's correct "class" is class i -- the diagonal IS the label vector
loss_doc_to_code = F.cross_entropy(sim, targets)      # doc -> best-matching code
loss_code_to_doc = F.cross_entropy(sim.T, targets)    # code -> best-matching doc
loss = (loss_doc_to_code + loss_code_to_doc) / 2       # symmetric InfoNCE

The similarity matrix's diagonal holds every true (docstring, function) match; everything off the diagonal is a negative supplied for free by the rest of the batch. Doubling batch doubles the negatives every row must be distinguished from — that's the difficulty knob in code.

Key takeaways

  • Contrastive objectives learn by comparison, pulling positive (docstring, function) pairs together and pushing every other in-batch pairing apart, rather than reconstructing corrupted input.
  • InfoNCE is a softmax classification loss where the "class" for row i is index i itself — the similarity matrix's diagonal is the entire label structure.
  • Temperature trades off signal against noise: τ → 0 overweights the hardest negative and produces noisy gradients; τ → ∞ flattens the softmax and the gradient vanishes.
  • Effective batch size is the difficulty knob for contrastive training specifically, because in-batch negatives scale directly with batch size — a property CLM and MLM training don't share.
  • Dual encoders trained contrastively beat generative models for large-scale retrieval because they directly optimize the embedding geometry search depends on; guard against collapse with normalization, big/diverse negative pools, and a tuned temperature.

Question 1 Multiple choice

You're building a search index over 40 million functions scraped from public repositories: given a natural-language query, return the best-matching function. You need embeddings such that true (query, function) matches are close and everything else is far, checked against millions of candidates at query time. Which pretraining objective should the two encoders use?

  1. Causal language modelling, then take the last hidden state as the embedding.
  2. BERT-style masked language modelling on each side independently, with no shared training signal between the two encoders.
  3. A contrastive objective (InfoNCE) trained jointly over (docstring, function) pairs.
  4. Span corruption with sentinel tokens, generating the function from the docstring.
Show answer

Answer: C. Retrieval at this scale needs an embedding space explicitly optimized so that true matches are close and everything else is far under a fast similarity operation (cosine/dot product) — that is exactly what InfoNCE trains for, and a dual encoder lets you embed all 40 million functions once, offline.

Option A produces representations optimized for predicting the next token, not for being compared to unrelated text; nothing in CLM training ever asks two different sequences' representations to be near or far from each other. Option B trains each encoder to reconstruct its own masked tokens in isolation — there's no joint signal at all linking docstring space to function space, so their embeddings aren't even guaranteed to be comparable. Option D would work eventually but is generation, not retrieval — you'd need to run a full generative pass per candidate rather than one embedding lookup, which doesn't scale to 40 million candidates.

Question 2 True / false

Raising the InfoNCE temperature τ toward infinity makes the loss focus more sharply on the hardest negative in the batch (the highest-similarity wrong function for a given docstring).

  1. True
  2. False
Show answer

Answer: B — False. It's the opposite. As τ → 0, dividing similarities by a tiny number amplifies differences between them, so the softmax sharpens toward whichever candidate has the single highest score — that's when the hardest negative dominates the gradient. As τ → ∞, dividing by a huge number crushes all similarities toward the same value, the softmax flattens toward uniform, and the gradient signal vanishes.

It's tempting to assume "higher temperature = more intense training signal" by analogy with everyday language, but in the softmax-temperature sense it's the reverse: low temperature sharpens, high temperature flattens.

Question 3 Fill in the blank

In causal language modelling with teacher forcing, if your tokenized batch has shape (batch, seq_len), the label tensor is constructed by slicing the same tensor and shifting it by   position(s), so that labels[:, i] holds the token that came immediately   inputs[:, i] in the original source file.

Show answer

Answer: 1 (one), and "after." inputs = tokens[:, :-1], labels = tokens[:, 1:] — the label tensor is the input tensor's data shifted left by exactly one column, so that at every position the target is literally the next real token from the source file, not a copy of the input at that position.

A common wrong instinct is "shifted by one token id" (as if you add 1 to the id) rather than shifted by one position in the sequence — the shift is positional (an index offset into the same sequence), not arithmetic on the token id values themselves.

Question 4 Bug fix

A teammate's CLM training loop trains without crashing, but held-out perplexity never drops below the vocabulary size (i.e., it's no better than random), and sampled completions are nonsense that ignores the given prompt. Find and fix the bug.

tokens = batch["input_ids"]          # (batch, seq_len), causal-masked model

inputs = tokens[:, :-1]
labels = tokens[:, :-1]              # <-- bug is somewhere in this block

logits = model(inputs)               # (batch, seq_len-1, vocab_size)
loss = F.cross_entropy(
    logits.reshape(-1, vocab_size),
    labels.reshape(-1),
)
Show answer

Answer: labels is sliced identically to inputs (tokens[:, :-1] for both), so the model is being trained to predict the token it was just given, at the same position — not the next one. The fix is labels = tokens[:, 1:], so inputs and labels are offset by one position as in Lesson 1's worked example. With the bug in place, the "easiest" solution to the (wrong) objective is closer to an identity/copy function, which is why perplexity plateaus near the vocabulary size and generated text ignores the actual prompt — the model never learned to predict forward at all.

It's tempting to suspect the reshape(-1, vocab_size) call, since flattening errors are a classic shape bug elsewhere — but the shapes here are consistent (both tensors are (batch, seq_len-1) before flattening); the defect is semantic (wrong slice), not dimensional, so no shape-related exception is ever raised.

Question 5 Multiple choice

You want a single causal, left-to-right decoder — not a separate bidirectional encoder — to autocomplete a function body in an IDE, using both the code already typed before the cursor and the code that already exists after the cursor (e.g., a later return statement). Which technique achieves this without adding bidirectional attention?

  1. BERT-style 80/10/10 masking applied to the causal model's input.
  2. Fill-in-the-middle with document rotation: reorder training documents as prefix, suffix, middle, and train with ordinary next-token prediction.
  3. Lower the softmax temperature during generation so the model attends further ahead.
  4. Increase the effective batch size so in-batch negatives expose more context.
Show answer

Answer: B. FIM reorders the document so the suffix appears in the token sequence before the middle the model must generate — the causal mask and the CLM objective never change, but the model now conditions on tokens from both sides of the gap by the time it has to fill it in.

Option A requires bidirectional attention to work at all (that's the premise of MLM), which contradicts "without adding bidirectional attention." Option C confuses a sampling hyperparameter with an architectural attention constraint — temperature reshapes the output distribution's sharpness, it doesn't change what positions the model can attend to. Option D is an in-batch-negatives idea from contrastive training (Lesson 3) that has no bearing on what a single sequence's causal mask allows a decoder to see.

Question 7 Multiple choice

During contrastive pretraining of your docstring/function dual encoder, both embedding towers converge to output nearly identical vectors regardless of input, and training loss stays suspiciously low. Which combination of causes and fixes is most consistent with this failure?

  1. This is expected behavior once InfoNCE converges — it's a sign of a well-trained encoder, not a bug.
  2. Representation collapse — likely from a too-small or too-uniform negative pool and/or missing embedding normalization; fix with L2-normalized embeddings, a larger and more diverse batch of in-batch negatives, and a properly tuned (non-extreme) temperature.
  3. The causal mask from Unit 2 is misconfigured, letting the decoder see future tokens; fix by re-checking the attention mask.
  4. The tokenizer from Unit 3 has an unresolved out-of-vocabulary bug; fix by increasing the BPE merge count.
Show answer

Answer: B. Mapping every input to nearly the same vector is the textbook signature of representation collapse: if negatives are scarce, easy to tell apart trivially, or embeddings aren't normalized (letting norm inflation game the loss), the model can drive the loss down without learning a meaningful embedding geometry at all. The standard fixes are exactly normalization, negative pool size/diversity, and temperature tuning — the three tools named in Lesson 3.

Option A mistakes a low loss number for genuine convergence; a collapsed encoder can produce low loss on a weak setup (small/easy negative pool) while providing zero retrieval usefulness, which is precisely why this is a "gotcha" failure mode rather than success. Options C and D misattribute the problem to unrelated components from Units 2 and 3 that this training setup doesn't even use in the same way — dual encoders for contrastive training are typically not causally masked in the way a CLM decoder is, and a tokenizer OOV bug would show up as garbled/unknown tokens, not as embedding collapse.
